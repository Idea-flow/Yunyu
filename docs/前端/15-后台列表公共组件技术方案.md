# 后台列表公共组件技术方案

## 1. 目标

后台多页面都存在“筛选 + 表格 + 分页”的重复结构，导致：

- 页面模板重复度高，样式调整需要多处同步；
- 列表交互（空态、骨架、分页）不一致；
- 新增后台列表页开发效率偏低。

本方案目标：

1. 抽象统一的后台列表工作台组件；
2. 保留业务差异化能力（列定义、行操作、筛选项）；
3. 先在附件列表页落地迁移，作为样板页面。

## 2. 组件分层设计

### 2.1 基础层（已存在）

- `AdminListFilterBar`：筛选布局容器。
- `AdminDataTable`：表格容器（骨架、空态、横向滚动）。
- `AdminPaginationBar`：统一分页交互。

### 2.2 工作台层（本次新增）

- `AdminListWorkbench`：
  - 组合基础层组件；
  - 对外暴露统一 props 和 slot；
  - 让业务页仅关注“筛选字段、列头、行内容、数据请求”。

## 3. AdminListWorkbench 接口约定

### 3.1 Props

- `isLoading`：是否加载中。
- `hasData`：是否有数据。
- `minWidth`：表格最小宽度。
- `headerClass`：表头 grid 布局类。
- `emptyTitle` / `emptyIcon` / `skeletonCount`：空态和骨架配置。
- `page` / `pageSize` / `total` / `totalPages`：分页状态。
- `showFilterBar` / `showPagination`：可选开关。

### 3.2 Emits

- `update:page`
- `update:pageSize`

### 3.3 Slots

- `search`：搜索区。
- `filters`：筛选区。
- `actions`：筛选条按钮区。
- `header`：表头列。
- `default`：表格行内容。

## 4. 附件列表页迁移实践

迁移对象：`AdminAttachmentLibraryPanel`

迁移结果：

1. 结构由“页面内手写三段”迁移为 `AdminListWorkbench` 统一承载；
2. 业务逻辑仍在 `AdminAttachmentLibraryPanel`（查询、删除、选择）；
3. 列结构按需求调整：
   - 最左列为 `ID`（纯数字，不带 `#`）；
   - 第二列为 `上传人ID`（纯数字，不带 `#`）；
   - `链接信息`列加宽，并提供“查看完整链接”toast。

## 5. 兼容与扩展策略

1. 兼容性：
   - 不改后端接口；
   - 不改业务数据模型；
   - 仅前端结构抽象。
2. 扩展性：
   - 后续文章、评论、用户、分类等列表可逐步迁移；
   - 支持按页面保留自定义操作列与渲染逻辑。

## 6. 验收标准

1. `AdminListWorkbench` 可独立复用；
2. 附件列表页面功能无回归（查询、分页、选择、删除）；
3. 附件列表列顺序满足：
   - 第一列 `ID`，第二列 `上传人ID`，均不带 `#`；
4. 链接信息列可展示更长内容，并可通过 toast 查看完整链接；
5. 前端构建通过。
