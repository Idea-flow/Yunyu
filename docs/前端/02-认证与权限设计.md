# 认证与权限设计

## 文档目标

本文档用于明确 `Yunyu` 当前阶段前后端认证联调方案、登录态保存方式、后台权限校验规则与跨域策略，确保后续前端页面开发与后端接口扩展保持一致。

## 一、当前阶段结论

当前项目已完成第一版前后端认证联调，采用以下方案：

- 后端认证基础设施：`Spring Security + JWT`
- 前端认证框架：`Nuxt 4`
- 登录入口：前台与后台共用 `/login`
- 当前用户接口：`GET /api/auth/me`
- 登录接口：`POST /api/auth/login`
- 后台访问角色：`SUPER_ADMIN`
- 前端令牌保存：前端 Cookie
- 前后端联调方式：前端直接请求 Spring Boot 接口

## 二、认证链路

### 2.1 登录流程

当前登录流程如下：

1. 用户访问共享登录页 `/login`
2. 前端输入账号和密码
3. 前端通过统一接口层调用 `POST /api/auth/login`
4. 后端校验账号密码后返回 `accessToken`
5. 前端将 `accessToken` 保存到 Cookie
6. 前端继续请求 `GET /api/auth/me`
7. 后端返回当前用户基础信息
8. 前端根据 `redirect` 参数或当前角色决定后续跳转

### 2.2 当前用户恢复流程

为了在页面刷新后仍然维持登录态，当前采用以下恢复方式：

1. 前端从 Cookie 中读取访问令牌
2. 若令牌存在，则调用 `GET /api/auth/me`
3. 若返回成功，恢复当前用户信息
4. 若返回失败，则清理本地令牌并回到未登录状态

## 三、共享登录页设计

当前共享登录页承担两类角色：

- 前台用户登录入口
- 后台管理员登录入口

统一登录页的原因：

- 避免前后台重复维护两套登录页面
- 保持认证交互一致
- 便于后续扩展多角色和统一用户中心

### 3.1 回跳规则

共享登录页通过 `redirect` 参数控制登录后的目标页面。

当前典型路径：

- `/login?redirect=/admin`

规则如下：

- 若存在合法 `redirect` 参数，则优先回跳
- 若无 `redirect` 参数，则默认进入 `/admin`
- 若目标是后台，但当前用户不是站长，则不允许进入后台

### 3.2 用户可见文案规则

共享登录页及后续认证相关页面必须遵循以下规则：

- 不向用户展示联调说明
- 不向用户展示角色字段、接口路径、技术状态等实现细节
- 页面只保留完成登录所必需的操作文案与失败反馈
- 技术说明统一保留在代码注释与文档中，不进入用户界面

## 四、后台权限设计

### 4.1 当前规则

当前后台仅允许站长角色访问。

后端角色判断依据：

- `SUPER_ADMIN`

### 4.2 前端校验规则

前端后台页面进入前，必须先执行以下检查：

1. 是否存在访问令牌
2. 是否能够成功获取当前用户
3. 当前用户角色是否为 `SUPER_ADMIN`

校验结果：

- 未登录：跳转到 `/login?redirect=/admin&reason=login-required`
- 非站长：跳转到 `/login?redirect=/admin&reason=not-owner`
- 已登录且为站长：允许进入后台页面

### 4.3 后端校验规则

后端对后台接口继续做服务端权限控制：

- `/api/admin/**` 仅允许 `SUPER_ADMIN`

因此后台访问控制采用：

- 前端路由守卫做体验层拦截
- 后端接口做最终权限校验

## 五、前端接口访问策略

### 5.1 当前方案

前端不再通过 Nuxt 服务端代理层访问后端，而是直接请求 Spring Boot 接口。

当前统一约束：

- 所有前端业务请求优先通过 `useApiClient`
- 不在页面中直接散落写后端地址
- 后端地址统一由 `NUXT_PUBLIC_API_BASE` 控制

### 5.2 环境变量

当前开发环境示例：

```bash
NUXT_PUBLIC_API_BASE=http://127.0.0.1:20000
```

线上环境只需要替换该变量，不需要修改页面业务代码。

## 六、跨域策略

### 6.1 当前结论

由于前端直接请求 Spring Boot，因此后端必须开放跨域能力。

当前阶段已在 Spring Boot 中加入全局跨域配置，允许前后端分离开发阶段直接联调。

### 6.2 当前策略

当前采用：

- 允许所有来源
- 允许所有方法
- 允许所有请求头

这个策略适用于当前开发联调阶段。

后续进入生产环境后，建议收敛为：

- 仅允许明确的前端域名
- 仅开放必要方法
- 仅开放必要请求头

## 七、统一返回结构约定

后端当前统一响应结构为：

```json
{
  "code": 0,
  "message": "操作成功",
  "data": {}
}
```

当前前端约定：

- `code = 0` 视为成功
- 非 `0` 视为业务失败

这一点必须在前端统一接口层中集中处理，不能在页面里各自判断。

## 八、当前落地实现

当前已落地的核心能力包括：

- 共享登录页
- 登录后获取当前用户
- 当前用户 Cookie 持久化
- 后台路由守卫
- 非站长禁止进入后台
- 后端全局跨域
- 前端统一接口访问入口

## 九、后续演进建议

后续建议继续补齐：

- 退出登录接口与统一登出流程优化
- 登录失败错误码映射与更细粒度提示
- 前台普通用户角色与后台管理员角色的跳转分流
- 基于权限码的后台菜单控制
- 统一刷新令牌机制

后续如认证机制、角色模型或跨域策略发生变化，必须先更新本文档，再进入实现阶段。
