# 文章内容展示与主题化渲染实现方案

## 文档目标

本文档用于定义 `Yunyu` 前端文章内容展示层的正式实现方案，覆盖：

- `Markdown / HTML` 内容在前端如何渲染为最终文章视图
- 如何支持不同内容主题与代码主题
- 如何统一处理标题、粗体、引用、链接、列表、表格、代码块等常见内容元素
- 后台编辑页预览组件与前台详情页正文组件如何复用一套展示能力
- 后续如何在不改动内容源的前提下扩展更多主题和展示能力

本文档关注“前端怎么实现内容展示层”，不是数据库设计文档，也不是后端派生链路文档。

## 一、当前问题与设计目标

当前正文预览虽然已经具备基础渲染能力，但还不满足正式内容平台的展示要求，主要问题包括：

- 正文只有基础 `prose` 渲染，缺少稳定的主题表达
- 链接、粗体、引用、分割线、表格等元素没有形成统一视觉语言
- 代码块虽然有基础高亮链路，但没有形成完整的代码展示体验
- 不支持通过组件显式传入“内容主题”和“代码主题”
- 后台编辑页预览与未来前台文章详情页之间还没有明确的组件分层规范

因此本阶段前端展示层要实现以下目标：

- 同一份 `contentHtml` 在后台预览、弹窗最终预览、前台详情页中可复用
- 正文展示支持主题化，而不是页面里散写样式
- 代码块支持不同语言高亮、文件头信息、复制按钮、折叠能力预留
- 组件允许按页面场景传入 `contentTheme` 与 `codeTheme`
- 展示规则文档化，后续新增内容页和预览页都按同一套能力实现

## 二、核心结论

前端正式采用以下实现方向：

- 内容渲染输入仍以服务端返回的 `contentHtml` 为主
- 内容展示层新增统一组件：`ArticleContentRenderer`
- 展示主题通过 `contentTheme` 控制
- 代码主题通过 `codeTheme` 控制
- 代码块结构通过 `Markdown / HTML + 额外数据属性` 驱动
- 后台编辑页右侧预览与弹窗“最终预览”复用同一套内容展示组件
- 前台文章详情页后续直接复用这套组件，不再重复造一套正文样式

推荐的组件接口形态如下：

```ts
type ArticleContentTheme =
  | 'editorial'
  | 'documentation'
  | 'minimal'

type ArticleCodeTheme =
  | 'github-light'
  | 'github-dark'
  | 'vitesse-light'
  | 'vitesse-dark'

interface ArticleContentRendererProps {
  html: string
  toc?: ArticleTocItem[]
  contentTheme?: ArticleContentTheme
  codeTheme?: ArticleCodeTheme
  mode?: 'editor-preview' | 'modal-preview' | 'article-detail'
}
```

## 三、展示层分层设计

### 3.1 统一分层

前端内容展示层统一拆成四层：

1. 渲染层：`useMarkdownRenderer`
2. 内容组件层：`ArticleContentRenderer`
3. 代码块增强层：`ArticleCodeBlock`
4. 主题样式层：`css/md/` 目录下的基础层与主题层文件

### 3.2 各层职责

`useMarkdownRenderer` 负责：

- 将 `Markdown` 转换为 `HTML`
- 提取 `TOC`
- 生成纯文本与阅读时长
- 给代码块增加语言标识、文件名占位、复制能力预留字段

`ArticleContentRenderer` 负责：

- 接收最终 `HTML`
- 以统一容器渲染正文
- 根据主题切换正文视觉
- 处理标题锚点、图片、链接、表格、引用、代码块等元素的统一样式

`ArticleCodeBlock` 负责：

- 提供统一代码块头部
- 展示语言名
- 提供复制按钮
- 提供折叠 / 展开能力
- 控制滚动区域与移动端展示体验

主题样式层负责：

- 输出不同主题的 CSS Variables
- 控制正文色板、链接、标题、引用、代码块、分割线、表格、选中文本等视觉

## 四、推荐前端组件结构

建议新增以下组件与文件：

- `app/components/content/ArticleContentRenderer.vue`
- `app/components/content/ArticleCodeBlock.vue`
- `app/components/content/ArticleTocTree.vue`
- `app/composables/useArticleContentTheme.ts`
- `app/assets/css/md/index.css`
- `app/assets/css/md/content-base.css`
- `app/assets/css/md/code-base.css`
- `app/assets/css/md/content-theme-editorial.css`
- `app/assets/css/md/content-theme-documentation.css`
- `app/assets/css/md/content-theme-minimal.css`
- `app/assets/css/md/code-theme-github-light.css`
- `app/assets/css/md/code-theme-github-dark.css`
- `app/assets/css/md/code-theme-vitesse-light.css`
- `app/assets/css/md/code-theme-vitesse-dark.css`

其中：

- `MarkdownPreview.vue` 后续不再承担长期正文标准能力
- `MarkdownPreview.vue` 可以内部逐步收口为 `ArticleContentRenderer` 的轻封装
- 后台文章编辑页、预览弹窗、前台文章详情页最终都应收敛到 `ArticleContentRenderer`

### 4.1 为什么不建议只用一个 `content-theme.css`

如果把正文基础规则、代码块结构规则、所有内容主题和所有代码主题都写进一个 CSS 文件，短期虽然看起来快，但后续会有明显问题：

- 文件会快速膨胀
- 主题之间容易互相污染
- 很难看出哪些是基础规则，哪些是主题差异
- 新增主题时容易复制粘贴一整份正文样式

因此当前项目正式约定：

- 可以有一个入口文件 `index.css`
- 不能把所有主题和所有规则堆进一个单体 CSS 文件

### 4.2 正式目录规范

当前文章内容展示层 CSS 正式采用以下目录结构：

```text
app/assets/css/md/
  index.css
  content-base.css
  code-base.css
  content-theme-editorial.css
  content-theme-documentation.css
  content-theme-minimal.css
  code-theme-github-light.css
  code-theme-github-dark.css
  code-theme-vitesse-light.css
  code-theme-vitesse-dark.css
```

职责约定如下：

- `index.css`
  只负责统一聚合导入，不承载业务规则
- `content-base.css`
  只放正文公共结构和基础排版规则
- `code-base.css`
  只放代码块公共结构规则
- `content-theme-*.css`
  只放内容主题变量和少量主题差异规则
- `code-theme-*.css`
  只放代码主题变量和代码外层容器差异规则

### 4.4 SSR 与 Hydration 约束

`ArticleContentRenderer` 同时服务于：

- 前台文章详情页
- 后台编辑页实时预览
- 后台弹窗最终预览

这意味着它必须兼容 `Nuxt SSR + 客户端 Hydration` 场景，因此当前项目新增以下硬性约束：

- 不要在 `setup` 阶段对 `v-html` 注入内容再依赖 `resolveComponent()` 动态解析图标组件
- 不要在服务端渲染阶段通过 `render(createVNode(...))` 往 `v-html` 内部节点挂载运行时组件
- 代码块工具栏图标优先使用静态 `SVG` 或纯 HTML 占位结构
- 复制、折叠、滚动测量等增强逻辑只在客户端执行
- 所有基于 `scrollHeight`、`navigator`、`window`、`document` 的增强逻辑都要放到 `onMounted` 或显式客户端分支中

这条约束的原因很明确：

- `v-html` 生成的内容本身已经绕过 Vue 模板编译
- 如果再在 `setup` 或 hydration 早期对内部节点挂运行时组件，服务端首屏结构和客户端首帧结构很容易不一致
- 一旦不一致，就会出现后台编辑页常见的 `Hydration node mismatch`，严重时还会连带触发 `setup function` 报错

本项目已经踩过一次这个坑，具体表现为：

- 后台文章编辑页 `/admin/posts/:id/edit`
- `ArticleContentRenderer` 在代码块按钮增强时动态解析图标组件
- 服务端输出与客户端 hydration 结构不一致
- 控制台出现 `Hydration node mismatch` 与图标组件相关报错

因此当前正式结论是：

- `ArticleContentRenderer` 内部的代码块工具栏图标统一使用静态 `SVG`
- `v-html` 区域只做客户端增强，不做 SSR 阶段的运行时组件挂载

```html
<article
  class="yy-content"
  data-content-theme="editorial"
  data-code-theme="github-light"
>
  ...
</article>
```

这样做的好处包括：

- 主题入口单一
- 运行时切换简单
- 便于组件封装
- 便于后续扩展更多主题

### 4.4 基础层与主题层的边界

必须严格区分“基础层”和“主题层”。

基础层负责：

- `h1-h6`
- `p`
- `ul / ol / li`
- `blockquote`
- `table`
- `img`
- `pre / code`
- 响应式换行、滚动、间距、结构布局

主题层负责：

- 颜色
- 背景
- 边框强弱
- 标题权重差异
- 引用块氛围
- 链接强调方式
- 行内代码与代码块语义色

正式约束：

- 新增主题时，不允许复制一整套正文结构规则
- 新增主题时，优先只新增变量和极少量差异覆盖

## 五、内容主题设计

### 5.1 主题目标

内容主题不是简单换背景色，而是完整控制以下维度：

- 正文字体层级
- 标题节奏
- 段落间距
- 链接样式
- 引用块样式
- 分割线样式
- 图片圆角与边框
- 表格容器与表头风格
- 列表缩进与标记
- 行内代码与代码块风格

### 5.2 首期主题方案

首期前端内容主题建议正式支持三套：

#### `editorial`

适合正式文章详情页与高质量最终预览。

特点：

- 标题层级鲜明
- 段落留白充足
- 引用块有更明显的编辑感
- 链接强调但不过度跳出
- 图片与表格更偏杂志化呈现

#### `documentation`

适合技术文档、方案文档、后台检查预览。

特点：

- 信息密度更高
- 标题与段落间距更克制
- 表格与代码块权重更高
- 适合长篇技术内容与参数类说明

#### `minimal`

适合后台窄空间预览、小卡片内嵌展示、摘要式阅读。

特点：

- 更轻
- 更紧凑
- 减少装饰感
- 适合弹窗、小屏、嵌入场景

### 5.3 主题切换规则

- 后台编辑页右侧实时预览默认使用 `minimal`
- 后台弹窗“最终预览”默认使用 `editorial`
- 前台文章详情页默认使用 `editorial`
- 文档类专题页后续可切换为 `documentation`

## 六、代码主题设计

### 6.1 代码主题目标

代码主题必须独立于内容主题管理，原因包括：

- 同一篇正文可能需要不同代码视觉
- 明亮内容主题不代表必须配明亮代码主题
- 后续技术内容页可能希望固定使用一套代码风格

因此正式支持：

- 内容主题 `contentTheme`
- 代码主题 `codeTheme`

两者分开传入，独立控制。

### 6.2 首期代码主题建议

首期建议支持以下 `Shiki` 主题映射：

- `github-light`
- `github-dark`
- `vitesse-light`
- `vitesse-dark`

主题使用规则：

- 明亮页面默认：`github-light`
- 暗黑页面默认：`github-dark`
- 技术感更强的页面可切：`vitesse-light / vitesse-dark`

### 6.3 代码主题生效方式

代码高亮分两部分：

- `Shiki` 负责生成 token 级别颜色
- 前端组件负责控制代码块容器、头部、边框、滚动、复制、折叠等 UI

也就是说：

- 代码“文字颜色”主要由 `Shiki` 决定
- 代码“容器样式”主要由前端组件主题决定

### 6.4 代码主题与内容主题的关系

内容主题与代码主题必须解耦，不能绑定死。

例如：

- `editorial` 可以配 `github-light`
- `editorial` 也可以配 `vitesse-dark`
- `documentation` 可以根据页面模式切换不同代码主题

正式约束如下：

- `contentTheme` 负责正文阅读气质
- `codeTheme` 负责代码高亮与代码容器语义
- 页面不得假设“明亮内容主题一定对应明亮代码主题”

## 七、代码块展示能力设计

### 7.1 首期必须支持

- 不同语言高亮
- 语言标识显示
- 长代码横向滚动
- 复制按钮
- 行内代码样式
- 代码块与正文之间的明确层级区分

### 7.2 第二阶段支持

- 折叠 / 展开代码块
- 显示文件名
- 显示高亮行
- 显示行号
- 支持差异对比块

### 7.3 折叠能力设计

代码块折叠不是 Markdown 基础能力，而是展示组件能力。

建议设计如下：

- 短代码块默认展开
- 超过设定高度的代码块显示“展开 / 收起”
- 折叠阈值按高度判断，不按行数硬编码
- 组件 props 支持：

```ts
interface ArticleCodeBlockProps {
  code: string
  language?: string
  fileName?: string
  collapsible?: boolean
  defaultExpanded?: boolean
  maxCollapsedHeight?: number
}
```

当前阶段先把接口和文档定好，代码实现可以在下一阶段补齐。

## 八、正文元素展示规范

### 8.1 标题

- `h1-h4` 必须有稳定层级差
- 标题颜色不能与正文同权重
- 标题锚点 hover 时再显式出现，不长期占视觉焦点

### 8.2 段落

- 正文行高必须稳定
- 段落之间通过节奏控制，不靠密集边框分割
- 连续短段落不能过度挤压

### 8.3 粗体与强调

- `strong` 必须显著强于正文
- `em` 采用轻斜体或轻强调色，不可与链接混淆
- 不能只依赖浏览器默认粗体效果

### 8.4 超链接

- 链接需要有明确的颜色、下划线或 hover 反馈
- 外链建议支持新窗口标识的视觉预留
- 长链接要支持断行，不能撑破容器

### 8.5 引用块

- 引用块不能只是默认左边框
- 应具备独立背景、边界和文字层级
- 技术引用与观点引用可以先用同一套样式

### 8.6 列表

- 有序列表与无序列表层级要清楚
- 列表嵌套需要控制缩进
- 列表项之间不能过度紧凑

### 8.7 表格

- 表格外层必须支持横向滚动
- 表头与正文需区分
- 长文本单元格允许换行
- 移动端不能直接撑爆布局

### 8.8 图片

- 图片默认圆角
- 图片必须限制最大宽度
- 独立图片与图文混排都要保持统一边距

### 8.9 分割线

- 分割线用于章节切换，不应过重
- 应更接近内容节奏线，而不是硬性分块线

## 九、主题 Token 设计

正文主题建议基于 CSS Variables 统一输出，不在页面里散写类名。

建议基础 Token 包括：

- `--yy-content-bg`
- `--yy-content-text`
- `--yy-content-muted`
- `--yy-content-heading`
- `--yy-content-link`
- `--yy-content-link-hover`
- `--yy-content-strong`
- `--yy-content-quote-bg`
- `--yy-content-quote-border`
- `--yy-content-code-inline-bg`
- `--yy-content-code-inline-text`
- `--yy-content-code-block-bg`
- `--yy-content-code-block-border`
- `--yy-content-table-border`
- `--yy-content-hr`

代码块补充 Token：

- `--yy-code-header-bg`
- `--yy-code-header-text`
- `--yy-code-copy-bg`
- `--yy-code-copy-text`
- `--yy-code-collapse-mask`

### 9.1 Token 实现方式

正式要求使用原生 CSS Variables 实现主题切换。

推荐形式如下：

```css
.yy-content {
  --yy-content-text: #334155;
  --yy-content-heading: #0f172a;
  --yy-content-link: #0ea5e9;
}

.yy-content[data-content-theme='editorial'] {
  --yy-content-text: #334155;
  --yy-content-heading: #0f172a;
  --yy-content-link: #0ea5e9;
}

.yy-content[data-code-theme='github-light'] {
  --yy-code-block-bg: #f6f8fa;
  --yy-code-block-border: #d0d7de;
}
```

### 9.2 为什么当前不使用 Sass 作为主题系统

当前阶段不建议为了内容主题系统引入 `Sass`，原因如下：

- 当前主题切换是运行时行为，不是编译时行为
- 运行时主题切换更适合 `CSS Variables`
- 当前项目已经使用 `Tailwind CSS + Nuxt UI`，再引入 `Sass` 会增加维护成本
- `Sass` 更适合做编译辅助，不适合作为主题切换本身

因此当前正式结论为：

- 主题系统主实现使用 `CSS Variables`
- 当前阶段不强制引入 `Sass`
- 后续如主题数量显著增加，可评估用 `Sass` 作为“生成辅助”，但不改变运行时主题机制

### 9.3 未来什么情况下再考虑引入 Sass

只有在以下情况同时变多时，再考虑引入 `Sass`：

- 内容主题数量明显增长
- 代码主题数量明显增长
- 出现大量重复 token 生成逻辑
- 出现大量重复的主题映射模板

即使未来引入 `Sass`，也仍然要遵循：

- `CSS Variables` 负责运行时主题切换
- `Sass` 仅作为构建辅助层
- 业务页面不直接依赖 Sass 变量

### 9.4 后续新增主题的正式流程

后续无论新增内容主题还是代码主题，都必须按以下流程执行：

1. 新增对应主题 CSS 文件
2. 只增加变量和少量差异规则，不复制基础层
3. 在组件类型定义中注册主题枚举
4. 在主题映射配置中补齐默认值
5. 补充对应文档说明
6. 至少检查标题、段落、链接、粗体、引用、表格、图片、代码块 8 类内容的实际表现

内容主题新增示例：

- 新增 `content-theme-magazine.css`
- 在 `ArticleContentTheme` 中增加 `magazine`
- 在 `useArticleContentTheme.ts` 中注册默认映射

代码主题新增示例：

- 新增 `code-theme-nord-dark.css`
- 在 `ArticleCodeTheme` 中增加 `nord-dark`

## 十一、与当前项目的衔接规则

- `Markdown` 仍是主内容源
- `contentHtml` 仍是展示层输入
- 主题切换只影响展示，不影响内容真相
- 后续新增文章预览、专题详情、文档详情时，统一复用 `ArticleContentRenderer`
- 后续页面不得在业务页散写正文元素样式，必须走公共内容组件

## 十二、当前定稿结论

本项目文章内容展示层正式定为：

- 使用统一正文组件承载最终展示
- 正文支持 `contentTheme`
- 代码块支持 `codeTheme`
- 代码高亮由 `Shiki` 提供
- 正文元素样式由统一主题 token 控制
- 主题样式层采用“入口文件 + 基础层 + 多主题文件”的目录方式
- 当前阶段主题系统使用 `CSS Variables`，不强制引入 `Sass`
- 后台预览与前台详情页逐步收敛为同一套展示体系
