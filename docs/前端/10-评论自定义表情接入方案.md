# 评论自定义表情接入方案

## 1. 目标

为前台评论区提供一套可复用的自定义表情能力，满足以下要求：

- 评论输入框可选择自定义表情
- 评论正文展示时可将表情编码渲染为图片
- 根评论与回复评论复用同一套能力
- 后台评论管理页也能看到与前台一致的表情效果
- 后续可以继续扩展更多表情资源，而不需要修改后端表结构

## 2. 当前落地方案

本项目当前采用“`文本编码 + 前端渲染`”方案。

核心思路：

- 评论提交到后端时，正文中保存表情编码，例如：`[微笑]`
- 数据库存储的仍然是普通字符串，不额外增加图片字段
- 评论展示时，前端把 `[微笑]` 之类的编码解析成对应的 SVG 图片

这样做的优点：

- 不需要修改评论表结构
- 不依赖富文本编辑器
- 可兼容全文搜索、审核、删除和导出
- 可平滑扩展成更多自定义表情包

## 4. 组件设计

### 4.1 `CommentEmojiPicker`

文件位置：

`yunyu-web/app/components/content/CommentEmojiPicker.vue`

职责：

- 在评论输入框底部工具栏中展示笑脸按钮
- 点击后弹出自定义表情面板
- 选择某个表情后，把对应表情项回传给父组件

说明：

- 这是一个纯“选择器”组件，不直接持有评论正文
- 后续如果消息输入框、留言板、私信输入区也要支持这套表情，可以直接复用

### 4.2 `CommentRichContent`

文件位置：

`yunyu-web/app/components/content/CommentRichContent.vue`

职责：

- 解析评论正文中的表情编码
- 将普通文本和表情图片按顺序渲染
- 保持前台评论区与后台评论列表的展示一致

当前接入位置：

- 前台文章评论区
- 后台评论管理页

### 4.3 `CommentRichEditor`

文件位置：

`yunyu-web/app/components/content/CommentRichEditor.vue`

职责：

- 作为评论输入区内部的富表情编辑器
- 在输入过程中把表情编码渲染成可视化 token
- 对外仍然输出原始文本编码，例如 `[微笑]`
- 支持在光标位置插入表情，而不是只追加到末尾

说明：

- 用户在输入框中看到的是“图片 + 名称”的轻量 token
- 实际提交到接口的仍然是纯文本内容
- 这让输入体验更自然，同时不影响后端存储和审核逻辑

### 4.4 `CommentComposer`

文件位置：

`yunyu-web/app/components/content/CommentComposer.vue`

职责补充：

- 继续作为根评论和回复评论共用的输入组件
- 内部接入 `CommentEmojiPicker`
- 内部组合 `CommentRichEditor`
- 负责承接评论输入、表情选择、提交和取消交互

说明：

- 选择表情后，输入框中会即时显示可视化表情 token
- 评论发布后，前台和后台会自动把它渲染为对应表情图片

## 5. 渲染规则

表情解析逻辑位于：

`yunyu-web/app/utils/commentEmoji.ts`

当前规则：

- 仅识别表情配置文件中定义的编码
- 未命中的中括号文本保持原样展示
- 评论正文始终按纯文本处理，不直接拼接不受控 HTML

这样可以避免：

- 自定义输入内容造成的 XSS 风险
- 未知编码导致页面结构异常
- 表情资源变化时需要修改历史评论数据

## 6. 为什么当前选 SVG，而不是 GIF

当前阶段优先选择 SVG，原因如下：

- 体积更小，评论列表滚动更轻
- 清晰度更稳定，适配高分屏更自然
- 配色和风格更容易统一
- 不会像 GIF 一样产生持续动画干扰阅读

如果后续有明确需求，可以在第二阶段扩展：

- 某些节日或活动表情使用 GIF / WebP 动图
- 保留同样的编码方式，只替换 `src`

## 7. 后续扩展方式

如果要新增一个表情，建议按以下步骤执行：

1. 在 `yunyu-web/public/emojis/comments/` 下新增静态资源
2. 在 `yunyu-web/app/constants/commentEmoji.ts` 中追加表情配置
3. 保持 `code` 唯一，推荐继续使用中括号语义文本，例如 `[偷笑]`
4. 无需修改后端接口、数据库结构或评论实体

## 8. 注意事项

### 8.1 数据库存储

当前评论表本身使用 `utf8mb4`，但本方案保存的是普通文本编码，因此不会因为表情包资源接入而要求后端字段变更。

### 8.2 搜索与审核

由于表情以文本编码形式存储：

- 后台搜索可以命中表情编码文本
- 审核和删除逻辑无需新增特殊处理
- 导出评论时也能保留表情语义

### 8.3 未来升级方向

后续如果评论输入体验要进一步增强，可以继续演进：

- 增加“最近使用表情”
- 增加表情分类
- 增加活动限定表情包或运营表情包
