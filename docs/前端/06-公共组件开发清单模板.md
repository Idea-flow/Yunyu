# 公共组件开发清单模板

## 文档目标

本文档用于作为 `Yunyu` 公共组件的统一开发模板。

后续每新增一个 `Yunyu` 公共组件，都建议按本文档同一格式完成以下事项：

- 判断是否应该进入 `common/`
- 设计组件职责与边界
- 约定 props 与事件
- 落实代码注释与主题支持
- 补充使用示例与文档记录

本文档既是开发清单，也是组件说明模板。

## 一、使用方式

新增公共组件时，建议复制以下模板内容，按实际组件替换其中占位项：

- 组件名
- 组件职责
- props
- 事件
- 默认样式
- 暗黑模式行为
- 使用示例

建议每个公共组件至少在对应需求说明、实现代码和本文档模板实例三者之间保持一致。

## 二、开发前检查清单

在创建组件前，先逐项确认：

- 这个组件是否会被前台与后台共同复用
- 这个组件是否属于基础能力而不是业务拼装
- 这个组件是否适合进入 `yunyu-web/app/components/common/`
- 这个组件是否应该使用 `Yunyu` 前缀命名
- 这个组件后续引用是否统一使用 `~/components/common/...` 导入
- 这个组件是否已有近似实现，能否先复用或抽象
- 这个组件是否需要默认支持明亮与暗黑模式
- 这个组件是否需要加载态、空态、禁用态或错误态
- 这个组件是否需要无障碍属性支持

如果以上问题有多项回答是否定的，应重新判断它是否真的属于公共组件。

## 三、实现阶段检查清单

进入编码阶段后，逐项确认以下内容：

- 文件路径是否放在 `yunyu-web/app/components/common/`
- 文件名是否采用 `PascalCase` 且以 `Yunyu` 开头
- 导入路径是否统一使用 `~/components/common/...`
- 类注释是否写明组件功能和作用
- 方法注释是否说明方法职责与用途
- 默认 props 是否足够支撑常见使用场景
- 是否避免把文章、用户、专题等业务对象直接耦合进组件
- 是否支持明亮与暗黑模式
- 是否具备合理的过渡、反馈与状态表现
- 是否保留必要但不过量的扩展能力
- 是否保证组件在移动端下也能正常使用

## 四、提测前检查清单

组件完成后，至少自查以下内容：

- 默认态是否正常
- 加载态是否正常
- 空态或错误态是否正常
- 暗黑模式表现是否正常
- 类名覆盖能力是否正常
- 无障碍属性是否完整
- 是否存在明显的 props 过多或职责失控问题
- 是否已经补充使用示例
- 是否已经同步更新相关文档

## 六、代码实现模板

以下模板用于创建新的公共组件文件，按实际组件调整。

```vue
<script setup lang="ts">
/**
 * 云屿通用 Xxx 组件。
 * 作用：统一处理 Xxx 的展示、状态与主题表现，
 * 作为前后台共享的基础公共组件。
 */

interface YunyuXxxProps {
  /**
   * 这里填写主要属性说明。
   */
  modelValue?: string
}

const props = withDefaults(defineProps<YunyuXxxProps>(), {
  modelValue: ''
})

/**
 * 这里填写方法说明。
 * 作用：描述该方法在组件中的职责。
 */
function handleAction() {
  console.log(props.modelValue)
}
</script>

<template>
  <div class="yunyu-xxx">
    <slot />
  </div>
</template>
```

## 七、文档同步模板

新增公共组件后，建议至少同步以下内容：

1. 若组件属于首批标准基础件，更新 `05-公共组件设计规范.md`
2. 若组件成为团队默认能力，在本文件补一条实例记录
3. 若组件影响目录说明或项目结构描述，更新 `01-前端路由与页面设计.md`
4. 若组件改变全局视觉标准，更新 `ui/01-基础视觉与交互规范.md`
