# 公共组件设计规范

## 文档目标

本文档用于约束 `Yunyu` 前端项目中的公共组件目录、命名方式、分层边界、设计原则与首批基础组件范围，确保后续组件扩展时具备统一标准。

本规范面向前台与后台共享组件，不讨论只服务于单一业务页面的局部组件实现。

## 一、为什么要建立公共组件目录

`Yunyu` 当前已经按前台内容组件与后台管理组件做了分层，但随着页面增加，如果没有独立的公共组件层，容易出现以下问题：

- 同一类基础能力在多个目录重复实现
- 图片、空状态、加载态、按钮等基础交互风格逐渐失控
- 前后台虽然共享主题，却没有共享基础视觉构件
- 页面开发速度会越来越慢，维护成本不断上升

因此当前约定：

- 建立 `app/components/common/` 作为公共组件目录
- 所有前后台可复用的基础组件优先沉淀到该目录
- 业务性强、只在单一领域使用的组件不要为了“看起来通用”强行放入公共目录

## 二、目录分层约定

当前前端组件目录统一约定如下：

```text
app/components/
  common/
  content/
  admin/
  theme/
```

各目录职责如下：

- `common/`：前后台共用的基础组件、基础展示单元、通用状态组件
- `content/`：只服务前台内容展示的组件，例如文章卡片、目录树、筛选条
- `admin/`：只服务后台管理的组件，例如表格卡片、表单控件、操作按钮
- `theme/`：主题切换、主题辅助、主题上下文相关组件

判断一个组件是否应该进入 `common/`，建议使用以下标准：

- 是否会同时被前台和后台复用
- 是否属于图片、按钮、空状态、加载态、弹层容器等基础能力
- 是否不依赖某个具体业务模型
- 是否可以通过少量 props 完成风格适配，而不需要嵌入大量业务判断

如果以上条件不满足，则优先留在业务目录中。

## 三、命名规范

公共组件统一使用 `Yunyu` 作为组件名前缀。

命名规则如下：

- 组件名统一采用 `PascalCase`
- 公共组件统一以 `Yunyu` 开头
- 名称优先表达职责，不使用模糊命名
- 不使用 `Base`、`Common`、`Item` 这类过宽泛且缺少语义的名字作为主名称

示例：

- `YunyuImage`
- `YunyuAvatar`
- `YunyuEmpty`
- `YunyuLoading`
- `YunyuSectionTitle`
- `YunyuNotice`

不推荐示例：

- `BaseImage`
- `CommonCard`
- `ItemBox`
- `ImageWrapper`

这样命名的目的不是“显得正式”，而是确保项目规模变大后，组件来源和职责仍能快速辨认。

## 四、导入路径规范

公共组件在业务页面或其他组件中被引用时，统一使用 `~/` 别名导入，不使用多层级相对路径。

推荐写法：

```ts
import YunyuImage from '~/components/common/YunyuImage.vue'
```

不推荐写法：

```ts
import YunyuImage from '../../components/common/YunyuImage.vue'
import YunyuImage from '../common/YunyuImage.vue'
```

统一使用 `~/` 的目的如下：

- 路径更短，更容易阅读
- 文件移动后更不容易出现路径断裂
- 更符合 `Nuxt` 项目的常见约定
- 能降低跨目录引用时的维护成本

## 五、公共组件设计原则

### 5.1 先做薄而稳的基础层

公共组件优先提供稳定、明确、可复用的基础能力，不承担复杂业务编排。

例如：

- `YunyuImage` 负责图片加载状态、主题表现与显示一致性
- 不负责文章封面布局、作者信息、内容角标等业务拼装

### 5.2 优先统一交互体验

公共组件最重要的价值，不是减少几行代码，而是统一用户体验。

统一范围包括：

- 加载状态
- 空状态
- 禁用状态
- 暗黑模式表现
- 交互反馈节奏
- 圆角、阴影、边框与过渡的一致性

### 5.3 不把业务逻辑塞进基础组件

公共组件不应直接依赖文章、专题、标签、用户等具体业务结构。

例如：

- `YunyuImage` 接收 `src`、`alt`、尺寸和样式类
- 不直接接收 `post` 对象再自己推导封面字段

### 5.4 通过 props 做有限定制

公共组件允许可控扩展，但不鼓励变成“万能组件”。

建议：

- 保留少量核心 props
- 暴露必要的 class 定制能力
- 保持默认值足够好用
- 当条件分支越来越多时，应拆分组件而不是继续堆 props

## 六、样式规范

公共组件必须遵循全局主题系统与视觉规范。

统一要求如下：

- 优先使用语义化颜色与现有主题类
- 明亮与暗黑模式都必须具备明确表现
- 默认状态应克制，不抢页面内容重心
- 动效应短促、柔和、不过度炫技
- 无障碍属性必须完整，例如图片 `alt`、按钮 `aria-label`

其中图片组件的加载占位风格，统一称为：

- 骨架屏
- 若带有横向流光动画，可称为 `shimmer skeleton`

后续同类加载占位，建议保持相同语言与相同视觉节奏。

## 七、组件落地流程

新增公共组件时，建议遵循以下步骤：

1. 先判断组件是否真的是前后台共享能力
2. 再判断是否应该进入 `common/`
3. 命名时使用 `Yunyu` 前缀
4. 导入路径统一使用 `~/components/common/...`
5. 先定义最小可用 props，而不是一次性做成大而全
6. 默认支持明亮与暗黑模式
7. 编写必要注释，并补充使用示例
8. 若组件成为基础标准件，应同步更新本文档

## 八、首批公共组件清单

第一批建议优先沉淀以下组件：

- `YunyuImage`：统一图片展示、加载占位、暗黑模式亮度控制
- `YunyuAvatar`：统一头像尺寸、占位与默认态
- `YunyuEmpty`：统一空列表、空搜索结果、空内容提示
- `YunyuLoading`：统一局部加载与页面加载指示
- `YunyuSectionTitle`：统一内容分区标题与副标题结构
- `YunyuHero`：统一首页、文章详情页等大图首屏的背景、遮罩、氛围层与底部信息布局
- `YunyuNotice`：统一成功、错误、警告、信息等轻量提示内容的视觉表达

这些组件应优先成为前后台共享基础件，而不是在业务页面中各自重复实现。

## 九、当前落地结论

当前项目正式约定如下：

- 公共组件目录为 `yunyu-web/app/components/common/`
- 公共组件统一使用 `Yunyu` 前缀命名
- 公共组件引用统一使用 `~/components/common/...` 别名导入
- `YunyuImage` 作为首个公共组件已经建立
- `YunyuSectionTitle` 已作为首页与文章详情页的分区标题标准件落地
- `YunyuHero` 已作为首页与文章详情页的大图首屏标准件落地
- `YunyuNotice` 作为统一提示视觉基础件补充到公共组件层
- 后续新增通用基础组件时，优先在本文档补充规范后再沉淀实现
- 新增公共组件时，开发与文档落地统一参考 `06-公共组件开发清单模板.md`
