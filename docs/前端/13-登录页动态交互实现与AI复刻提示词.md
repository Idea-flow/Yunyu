# 登录页动态交互实现与 AI 复刻提示词

## 文档目标

本文档用于沉淀 `Yunyu` 前台登录页的真实开发实现，帮助后续研发成员或 AI 智能体快速理解以下内容：

- 当前登录页为什么这样设计
- 左侧四个角色动态交互是如何实现的
- 右侧登录表单的样式策略是什么
- 如果后续要再开发一个类似页面，应该按什么步骤落地
- 如果交给 AI 智能体开发，提示词应该怎么写，才能尽量一次做对

本文档不是单纯的视觉说明，而是“可继续开发、可继续细抠、可复刻”的工程实现文档。

## 二、页面目标

本登录页不只是一个普通认证表单，而是一个“前台统一认证入口 + 左侧角色动态表演区 + 移动端可收敛”的品牌化页面。

目标拆解如下：

- 前台与后台共用同一认证入口
- 左侧角色要有明显记忆点，不能是静态装饰
- 鼠标跟随、账号聚焦、密码聚焦、密码显示、报错都要有联动反馈
- 右侧表单不能有传统后台卡片感，要更轻、更平、更自然
- 移动端要自动收敛为单栏认证体验，不能为了保留动效牺牲操作效率

## 三、最终设计原则

### 3.1 总体原则

当前登录页开发时遵循了这些原则：

- 左侧是“角色表演区”，右侧是“认证操作区”
- 左侧动效可以强，右侧输入区必须克制
- 页面整体不能有明显后台卡片感
- 页面默认首屏就完整呈现，不依赖下拉
- 明亮模式优先，暗黑模式收敛

### 3.2 左侧原则

- 四个角色的相对站位、大小、神态尽量贴近参考 HTML 原码
- 角色跟随不是所有人同步同一个向量，而是按各自锚点做独立追踪
- 眼睛容器与瞳孔都要动，不能只有瞳孔动
- 特殊状态优先使用“定点姿态”，不要一边锁定情绪一边继续乱跟随

### 3.3 右侧原则

- 右侧不要大面积卡片底
- 输入框只保留轻椭圆、弱边界、弱阴影
- 图标只做轻量引导，不做抢戏装饰
- 文案尽量少，减少登录动作之外的噪音

页面大体分为两大区域：

### 4.1 左侧角色区

左侧包含：

角色类型：

- 紫色高矩形角色
- 深色中矩形角色
- 橙色半圆角色
- 黄色圆角角色

### 4.2 右侧认证区

右侧包含：

- 账号 / 邮箱 / 密码 / 确认密码表单
- 提交按钮
- 模式切换文案

## 五、左侧动态交互实现拆解

### 5.1 状态源

登录页的动态交互不是写死动画，而是通过状态驱动。

当前核心状态有：

- `focusedField`
- `isSceneError`
- `sceneErrorStage`
- `identityGreetingPhase`
- `purpleBlinking`
- `blackBlinking`
- `purplePeeking`
- `purplePeekPhase`
- `passwordVisibility`
- `scenePointer`

这些状态共同决定角色当前应该看哪里、身体怎么偏、嘴怎么动、是否进入特殊姿态。

### 5.2 鼠标跟随

当前鼠标跟随不是简单把鼠标位置映射给所有角色，而是：

1. 页面级监听 `@pointermove`
2. 将鼠标位置归一化为 `scenePointer.offsetX / offsetY`
3. 根据每个角色的锚点，调用 `resolveCharacterVector(anchorX, anchorY)`
4. 为不同角色生成不同的观察向量

这样做的好处是：

- 紫色、黑色、橙色、黄色不会“同向同步摆头”
- 四个角色更像独立个体
- 观感更接近参考 HTML 原码

### 5.3 账号 / 邮箱聚焦

参考 HTML 原码中，账号输入聚焦不是持续重复触发，而是：

- `focus` 时只触发一次
- 先让紫色和黑色角色互看
- 约 `800ms` 后结束互看
- 回到正常输入状态

在当前项目中，对应实现是：

- `startIdentityGreeting()`
- `identityGreetingPhase = 'greeting'`
- 定时器 `800ms` 后回到 `idle`

这个阶段必须注意：

- 不能在 `input` 时反复触发
- 不能出现“点击一次变化两次”的感觉
- 聚焦结束后要恢复正常跟随，不要卡死在输入框方向

### 5.4 密码聚焦

密码聚焦时采用“回避密码”的表达方式。

当前实现策略：

- `isPasswordFocused && !isShowingPassword`
- 统一切到“向左上回避”的固定姿态
- 四个角色的眼睛容器、瞳孔、嘴巴使用绝对偏移
- 不再继续按鼠标实时摇摆

这样做的原因是：

- 密码保护情绪需要更明确
- 如果密码态还叠加实时鼠标跟随，观感会很乱

### 5.5 密码显示偷看

参考页最有记忆点的一段是“密码显示时紫色角色偷看”。

当前项目实现方式：

- 当密码字段已有内容且已显示明文时
- 通过 `schedulePurplePeek()` 安排随机偷看节奏
- `purplePeeking` 控制是否正在偷看
- `purplePeekPhase` 控制 `forward / return`

偷看阶段的关键点：

- 眼睛基准点会切到固定位置
- 瞳孔在前进和返回阶段使用不同基准
- 身体和眼睛的延迟关系要自然，不能同时瞬移

### 5.6 报错反馈

报错阶段不是单纯抖动，而是三段式：

1. 先进入伤心姿态
2. 再开始抖动
3. 最后恢复

当前项目中：

- `sceneErrorStage = 'pose' | 'shake'`
- 先切 `pose`
- 定时后切到 `shake`
- 再统一恢复到 `idle`

这是为了避免：

- 一报错就立刻抖，显得很吵
- 情绪没有铺垫
- 角色像 bug，而不是反馈

### 5.7 眨眼与默认首帧

为了避免角色太呆，当前还加了：

- 紫色角色随机眨眼
- 深色角色随机眨眼
- 首次进入页面默认看向左上方“云屿”图标

当前首帧默认位置不是正中间，而是：

- `scenePointer.offsetX = -0.96`
- `scenePointer.offsetY = -1.04`

这样角色一进页就像在看左上角品牌区域。

## 六、右侧表单实现策略

### 6.1 为什么去掉卡片背景

本页面经过多轮调整后，右侧已经明确不走“后台卡片表单”路线。

原因：

- 登录页本质是一个动作页，不是信息容器页
- 页面已经有左侧强角色区，如果右侧再用大卡片，会显得厚重
- 前台内容站的认证入口更适合轻、平、柔和的表达

所以当前策略是：

- 不给整个右侧表单加大容器背景
- 只精修输入框本身

### 6.2 输入框策略

当前输入框采用：

- 椭圆形 `rounded-full`
- 轻边框
- 弱底色
- 轻微阴影
- 聚焦时只增强一点点边界和亮度

不采用的方案：

- 厚玻璃卡片
- 粗描边科技风
- 深投影浮空输入框
- 过于扁平的纯下划线

### 6.3 图标策略

当前图标最终收敛为：

- 无背景
- 小尺寸线性图标
- 只承担识别作用
- 不做图标胶囊或图标圆球

这是因为：

- 登录页本身已经有左侧高记忆点角色
- 右侧图标不需要再抢戏

## 七、移动端适配策略

当前页面的移动端适配原则是：

- 操作优先
- 表单优先
- 装饰可收敛

具体策略：

- 左侧角色区在大屏显示，在移动端隐藏
- 右侧表单在移动端独立占主区
- 输入框和按钮保持足够点击面积
- 避免横向滚动
- 不保留大面积装饰背景

## 八、如果要继续开发类似页面，建议按这个顺序做

后续如果要开发一个类似“角色联动型认证页”，建议按以下步骤实施。

### 第一步：先拆参考页，不要急着写 Vue

优先拆清楚：

- 哪些状态是鼠标态
- 哪些状态是聚焦态
- 哪些状态是密码态
- 哪些状态是错误态
- 哪些效果是绝对位置
- 哪些效果是相对跟随

如果这一步不做清楚，后面很容易写成“大量 if-else + 看起来很像，但就是不自然”。

### 第二步：先设计状态机，再写样式

建议先列状态：

- 默认
- 鼠标跟随
- 账号聚焦
- 密码回避
- 密码显示偷看
- 报错

然后定义：

- 哪些状态可以叠加
- 哪些状态互斥
- 哪些状态需要定时器

### 第三步：身体和脸分开算

不要把角色写成一个统一 transform。

建议拆为：

- 角色主体姿态
- 眼睛容器位移
- 单只眼球微调
- 瞳孔位移
- 嘴部位移

只有拆开，才能细抠到接近参考页。

### 第四步：特殊情绪使用固定姿态

对于以下状态，不建议继续叠加鼠标实时跟随：

- 密码回避
- 报错
- 互看
- 偷看

这几种状态应该尽量用固定基准点控制。

### 第五步：最后再调过渡时长

很多“看起来不自然”的问题，根源不是位置错了，而是：

- 眼睛太快
- 身体太快
- 眼睛和身体同时启动
- 报错没有停顿

所以建议最后单独调：

- 眼睛过渡
- 身体过渡
- 瞳孔过渡
- 嘴部过渡
- 抖动延迟

## 九、开发时的高风险点

### 9.1 不要重复触发账号互看

最容易犯的错误是：

- `focus` 触发一次
- `input` 又触发一次
- 每输一个字符再触发一次

这会导致角色像抽搐，不像互看。

### 9.2 不要所有角色共用同一个鼠标向量

这样会导致所有角色像一个整体在摆头，显得很呆。

### 9.3 不要只动瞳孔，不动眼睛容器

如果只是瞳孔位移，神态会明显不够。

### 9.4 不要让右侧重新变成后台卡片

一旦给右侧整块加厚背景，前台登录页的气质就会立刻偏后台。

## 十、建议 AI 智能体遵循的实现约束

如果未来让 AI 智能体继续改这个页面，建议明确约束：

- 所有回复和文档使用简体中文
- 不允许把右侧登录区改成重卡片
- 左侧角色交互要优先参考参考 HTML 原码
- 账号聚焦只能触发一次互看
- 密码回避要固定姿态
- 密码显示偷看要有 `forward / return`
- 报错必须是“伤心定格 -> 抖动 -> 恢复”
- 移动端继续保持单栏收敛
- 修改后必须构建验证

## 十一、推荐给 AI 智能体的高质量提示词

下面这份提示词可以作为后续开发类似页面时的基础模板。

```md
你现在要在一个 Nuxt + Vue 项目中实现一个“前台品牌化登录页”，目标文件是 `app/pages/login.vue`。

请严格按以下要求开发：

1. 页面定位
- 这是前台统一认证入口，不是后台控制台登录页。
- 页面整体要简洁、轻、平、自然，不要明显卡片感。
- 左侧是角色互动舞台，右侧是认证表单。
- 移动端自动收敛为单栏表单体验。

2. 参考基准
- 必须参考 `/Users/wangpenglong/Downloads/login-animated.html` 的动态逻辑。
- 重点对齐以下行为：
  - 鼠标跟随
  - 账号/邮箱输入聚焦时，紫色与深色角色先互看，再回到正常输入联动态
  - 密码聚焦时四个角色统一回避
  - 密码显示且有内容时，紫色角色随机偷看
  - 报错时先伤心定格，再抖动，再恢复

3. 代码实现要求
- 用 Vue 组合式 API 和响应式状态实现，不要写成散乱的 DOM 操作。
- 先设计状态机，再写模板。
- 身体、眼睛容器、眼球、瞳孔、嘴巴分开计算。
- 特殊状态下优先使用固定基准位移，不要继续叠加鼠标实时跟随。
- 普通鼠标态下，每个角色按自身锚点独立计算观察向量，不允许所有角色共用同一个向量。

4. 右侧表单要求
- 整个右侧不要使用厚卡片背景。
- 输入框用轻椭圆、弱边界、弱底色。
- 图标只保留简单线性图标，不要图标背景块。
- 减少无关文案，只保留必要提示。

5. 交互要求
- 账号聚焦只触发一次互看，不要 input 时重复触发。
- 鼠标移出浏览器时，保持最后一帧，不主动复位。
- 默认首帧让角色看向左上方品牌图标区域，不是中间。

6. 工程要求
- 尽量复用已有状态与结构，不要无意义大改。
- 修改完成后执行构建验证。
- 输出时说明：
  - 改了什么
  - 哪些状态是关键
  - 哪些地方最接近参考页
  - 是否通过构建
```

## 十二、建议 AI 智能体的执行步骤

如果后续让 AI 智能体继续做，建议让它严格按下面的顺序执行：

1. 先阅读参考 HTML 的交互函数与样式，不要先动项目代码。
2. 再阅读 `login.vue` 当前实现，确认哪些状态已经存在。
3. 先输出方案，再开发。
4. 开发时优先小步迭代，不要一次推翻全部结构。
5. 每次只细抠一类行为，例如“账号聚焦互看”或“密码偷看节奏”。
6. 每轮开发后都执行构建验证。

## 十三、后续可继续细抠的方向

当前页面已经可用，但如果后续还要继续追近参考页，可以优先细抠这些点：

- 眼睛容器在四角观察时的绝对位移
- 紫色角色偷看的 forward / return 停顿节奏
- 黑色角色互看结束后的回位轨迹
- 黄色角色嘴线与眼神联动的微节奏
- 明暗模式下输入框质感的一致性

## 十四、结论

当前登录页的核心经验不是“把一个 HTML 照抄进 Vue”，而是：

- 先拆状态
- 再拆部件
- 再拆时序
- 最后才是样式和数值细抠

只有这样，后续不管是人工继续迭代，还是交给 AI 智能体继续实现，才能在不推翻现有工作的前提下持续逼近目标效果。
