# 后台布局与工作台规范

## 文档目标

本文档用于定义 `Yunyu` 后台管理端的长期布局规范、视觉层级与页面骨架，解决当前后台页面“整块堆叠、装饰偏重、层级不够自然”的问题。

本规范用于指导后续后台首页、文章管理、分类管理、标签管理、专题管理、用户管理、系统监控、站点设置等页面的统一改造。

本文档重点不是描述“某一页怎么写”，而是沉淀后台管理端的共用设计原则、结构模板与验收标准。

## 一、当前问题判断

结合当前后台布局与页面实现，现阶段主要问题集中在以下几个方面：

- 容器层级过多，页面头部、筛选区、统计区、列表区大多都被独立大卡片包裹。
- 卡片圆角、阴影、背景渐变和装饰纹理使用偏多，导致工作台气质不够克制。
- 模块之间主要依赖“大卡片切割”而不是“留白、标题层级、分组关系”来组织信息。
- 页面重点不够稳定，筛选区、概览区、表格区的视觉权重经常接近。
- 同一后台内出现较多相似但不完全一致的容器样式，降低整体秩序感。

因此，后台改造的核心方向不是继续增加视觉元素，而是：

- 降低卡片依赖
- 降低装饰强度
- 强化页面骨架
- 统一密度与间距
- 让用户先看到“结构”，再看到“组件”

## 二、后台设计定位

`Yunyu` 后台不是传统控制台风格，也不是营销页风格，而是“编辑工作台”风格。

最终气质应当满足：

- 克制
- 安静
- 稳定
- 清楚
- 高可读性
- 低噪声

后台的第一原则不是“有设计感”，而是：

- 信息容易扫描
- 操作顺序明确
- 长时间使用不疲劳
- 页面之间结构一致

## 三、整体设计原则

### 3.1 布局优先于装饰

后台的简洁感主要来自结构、留白和密度控制，而不是依赖阴影、毛玻璃、渐变和大圆角。

### 3.2 页面优先于卡片

后台页面应先建立完整页面骨架，再决定哪些区域需要卡片。不能默认“每个区块都是一张卡片”。

### 3.3 标题分级优先于背景分块

用户应该通过页面标题、区块标题、辅助说明和内容排列理解结构，而不是靠一层层容器背景来判断分组。

### 3.4 一页只保留少量重点

每个后台页面只能有一个主操作焦点、一个主要内容区和少量辅助信息区，避免同时出现多个同权重模块争抢注意力。

### 3.5 统一比个性更重要

后台可以有品牌气质，但后台各页面首先必须共享一致的布局语言、容器规则、按钮层级和信息密度。

## 四、后台信息架构骨架

后台统一采用三层结构：

1. 全局框架层
2. 页面骨架层
3. 业务内容层

### 4.1 全局框架层

全局框架层由以下部分组成：

- 左侧导航：承载主导航、当前模块定位、账号信息、主题切换、退出操作。
- 顶部导航：承载当前页面标题、页面级操作、状态提示、面包屑或快捷入口。
- 主内容区：承载页面头部、筛选区、主体数据区和辅助区。

要求：

- 左侧导航负责“切模块”。
- 顶部导航负责“看当前页”和“做当前页主操作”。
- 主内容区负责“完成工作”。

### 4.2 页面骨架层

后台页面统一拆分为以下结构：

1. 页面头部
2. 轻量操作区
3. 主内容区
4. 页尾补充区

说明如下：

- 页面头部：页面标题、页面说明、主操作按钮。
- 轻量操作区：搜索、筛选、状态切换、批量操作入口。
- 主内容区：表格、列表、编辑器、配置表单、监控面板。
- 页尾补充区：分页、帮助提示、最近更新时间、说明信息。

### 4.3 业务内容层

业务内容层按页面类型拆分，不同页面采用不同主工作区模型：

- 列表页：筛选 + 列表
- 表单页：标题 + 表单主体 + 固定操作区
- 工作台页：概览 + 待办 + 快捷入口，但禁止仪表盘卡片堆叠
- 监控页：状态摘要 + 趋势区 + 明细区

## 五、后台容器策略

这是本次改造最重要的规范。

### 5.1 默认不用卡片

后台不是每个区块都必须使用卡片。默认优先级如下：

1. 直接使用页面留白分组
2. 使用浅分割线或标题区分组
3. 只有在内容需要独立边界时才使用卡片

### 5.2 哪些区域适合使用卡片

允许使用卡片的区域包括：

- 独立统计块
- 单独的表单分组
- 需要明显边界的提示信息
- 可复用的列表容器
- 弹窗与抽屉内容

### 5.3 哪些区域不建议使用卡片

以下区域原则上不再默认包大卡片：

- 页面头部
- 筛选栏
- 面包屑与页级说明
- 分页区
- 简单表格上方的标题与操作条

### 5.4 卡片样式限制

后台卡片必须克制，限制如下：

- 大型主容器圆角统一控制在 `12px` 到 `14px`
- 次级容器、表格壳层、筛选工具条圆角统一控制在 `10px` 到 `12px`
- 图标按钮、分页按钮、小徽标容器圆角统一控制在 `8px` 到 `10px`
- 页头右侧操作按钮、页头辅助按钮统一使用微圆角，小控件圆角建议控制在 `8px` 到 `10px`，禁止使用 `rounded-2xl` 及 `20px+` 的大圆角，避免按钮呈现胶囊化、椭圆化
- 后台不允许再使用 `20px+`、`24px+`、`28px+`、`30px+` 这类接近胶囊或半椭圆的圆角
- 后台需要“微圆角”，不是“强椭圆角”，视觉目标是轻柔而克制，不是软糯卡通
- 阴影仅作为弱悬浮提示，不得成为主视觉
- 优先使用浅边框，而不是强阴影
- 卡片背景使用纯色或极轻微色差，不再叠加强装饰纹理
- 同一页面内容器样式不超过 `2` 种

## 六、页面密度与间距规范

后台自然感很大程度来自节奏，而不是来自颜色。

统一使用以下间距体系：

- 页面左右内边距：`24px`
- 页面上下主区块间距：`24px`
- 区块内部常规内边距：`16px`
- 小组件之间间距：`8px`
- 标题与正文之间间距：`12px`
- 表单项之间垂直间距：`16px`
- 大模块之间间距：`32px`

响应式建议：

- 桌面端主内容区宽度优先保持稳定，不做超宽拉伸。
- 平板端减少双列区块，优先收敛为单列。
- 移动端后台若继续支持访问，应优先保证查看与简单操作，不强求复杂多列布局。

## 七、视觉层级规范

### 7.1 标题层级

后台标题统一分为四层：

- 页面标题：定义当前页面职责
- 区块标题：定义某个功能区
- 数据标题：定义卡片或指标
- 辅助文字：定义说明、时间、状态注释

建议视觉强度如下：

- 页面标题最强，但不夸张
- 区块标题稳定清晰
- 数据内容可略强于区块标题
- 辅助文字弱化，不抢焦点

### 7.2 操作层级

页面操作优先级固定如下：

1. 主操作
2. 次操作
3. 危险操作
4. 辅助操作

规则如下：

- 每个页面只保留一个主按钮高亮。
- 次级按钮优先使用描边或弱填充。
- 危险操作只在需要时使用警示色。
- 图标按钮只能做辅助，不应承担页面主操作。

### 7.3 状态层级

状态表达应轻量但清楚：

- 成功态：用于完成与发布
- 警告态：用于待处理与风险提示
- 错误态：用于失败与阻断
- 中性态：用于说明与统计

禁止将所有标签、徽标、提示都做成强对比色块。

## 八、后台各类页面的推荐模板

### 8.1 列表页模板

适用页面：

- 文章管理
- 用户管理
- 分类管理
- 标签管理
- 专题管理

推荐结构：

1. 页面头部
2. 轻量筛选区
3. 列表标题条
4. 数据表格
5. 分页区

要求如下：

- 页面头部只负责说明当前页面和提供主操作按钮。
- 筛选区采用轻量容器或直接排版，不使用厚重卡片。
- 列表标题条可与表格容器合并，但不必额外再套一层大卡片。
- 分页区与表格保持连续关系，不单独做强调容器。

### 8.2 表单页模板

适用页面：

- 新建文章
- 编辑文章
- 站点设置
- 大型配置页

推荐结构：

1. 页面头部
2. 主表单工作区
3. 辅助说明区或预览区
4. 固定底部操作栏

要求如下：

- 主表单工作区优先保证阅读和编辑连贯性。
- 表单分组按任务拆分，而不是按视觉块随意切分。
- 保存、发布、返回等操作位置必须稳定。
- 页面型表单不建议继续使用多个同权重大卡片纵向堆叠。

### 8.3 工作台首页模板

适用页面：

- 后台首页

推荐结构：

1. 页面头部
2. 简短状态摘要
3. 本周待办
4. 快捷入口
5. 系统状态补充

要求如下：

- 后台首页不是营销页，不需要 Hero 式大视觉开场。
- 不使用大段情绪化描述，不做宣传口号式文案。
- 统计信息控制在少量关键指标，避免卡片矩阵化。
- 待办区优先级高于装饰性概览区。

### 8.4 监控页模板

适用页面：

- 系统监控

推荐结构：

1. 页面头部
2. 当前运行摘要
3. 关键指标区
4. 明细列表或趋势图
5. 说明与刷新信息

要求如下：

- 用数字与趋势支撑页面，不靠装饰容器制造存在感。
- 监控信息优先保证对齐、分组与扫描效率。

## 九、结合当前项目的落地约束

结合当前实现，后续改造建议按以下方向推进。

### 9.1 `admin` 布局改造方向

当前左侧导航已有统一入口，但装饰较强。后续应调整为：

- 保留统一导航结构
- 降低背景渐变、纹理和玻璃感强度
- 缩减超大圆角与大面积阴影
- 让导航强调“稳定框架”而不是“视觉主角”

### 9.2 列表页容器改造方向

当前 `AdminFilterPanel` 与 `AdminTableCard` 都是完整大卡片容器。后续应调整为：

- 筛选区改为轻量分组容器或无卡片工具条
- 列表区保留一个主容器即可
- 列表标题、批量操作、统计数与表格合并为一个连续工作区

### 9.3 后台首页改造方向

当前首页存在较强的工作台展示感，但页面块较多。后续应调整为：

- 去掉过强的首屏包装感
- 弱化大视觉开场
- 优先突出待办、快捷入口和关键状态
- 用更少的模块承载更清楚的首页信息

### 9.4 通用后台组件改造方向

后续组件改造建议优先级如下：

1. `AdminFilterPanel`
2. `AdminTableCard`
3. `admin.vue` 布局
4. 后台首页
5. 各后台列表页
6. 各后台表单页

## 十、后台样式 Token 约束

后台页面必须共享统一语义，不允许在页面中散落大量一次性视觉参数。

建议后台优先使用以下语义：

- `admin-bg-page`
- `admin-bg-surface`
- `admin-bg-surface-soft`
- `admin-border-default`
- `admin-border-muted`
- `admin-text-strong`
- `admin-text-body`
- `admin-text-muted`
- `admin-accent`
- `admin-accent-soft`
- `admin-success`
- `admin-warning`
- `admin-danger`

实现要求：

- 后台优先基于语义 token 调整样式。
- 避免在页面组件里持续堆叠一次性 Tailwind 任意值。
- 同类组件必须复用相同边框、圆角、背景和阴影规则。

## 十一、改造执行顺序

为避免边改边乱，后台改造统一按以下顺序推进：

1. 先统一布局原则与页面模板
2. 再抽象后台通用容器样式
3. 再改后台首页
4. 再改列表页
5. 再改表单页
6. 最后统一监控页与设置页细节

每一阶段都必须遵守：

- 先做结构收敛
- 再做视觉收敛
- 最后再修细节

## 十二、验收清单

后续每个后台页面改造完成后，都必须对照以下清单检查：

- 页面是否仍然依赖多个同权重大卡片分块。
- 页面头部、筛选区、主体区、分页区是否关系清楚。
- 主操作是否只有一个高亮按钮。
- 是否通过留白和标题层级完成分组，而不是完全依赖背景块。
- 容器样式是否控制在少量统一规则内。
- 页面装饰是否明显少于当前版本。
- 内容扫描路径是否清楚，用户是否能快速找到“看什么”和“做什么”。
- 列表、表单、监控、首页四类页面是否遵循对应模板。

## 十三、最终设计判断标准

改造后的后台应让用户产生如下感受：

- 结构清楚
- 页面安静
- 信息自然
- 操作顺手
- 不花哨，但也不单调

如果一个页面去掉大部分阴影和装饰后仍然成立，说明布局是健康的。

如果一个页面必须依赖很多卡片和装饰才能显得“有设计”，说明骨架仍然不够好。
