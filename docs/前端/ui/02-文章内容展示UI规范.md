# 文章内容展示 UI 规范

## 文档目标

本文档用于定义 `Yunyu` 文章内容展示组件的 UI 规范，确保后台最终预览、前台文章详情页、专题内容页在视觉上属于同一套产品语言。

本文档只讨论“文章内容怎么显示更像产品”，不讨论后端存储和 Markdown 解析实现。

## 一、设计定位

文章内容展示不是“浏览器默认排版”，也不是“文档站模板默认样式”。

`Yunyu` 的内容展示应具备以下特征：

- 像被认真设计过的数字文章，而不是默认 HTML
- 阅读体验优先，而不是组件感优先
- 技术内容要专业，但不能有浓重开发工具感
- 后台预览与前台详情页气质一致，只在密度上区分

当前视觉方向定为：

- 主方向：`Editorial Reading Surface`
- 技术内容增强：`Refined Technical Layer`

## 二、整体视觉原则

### 2.1 内容本身是主角

- 不允许正文区域出现多余提示卡片
- 不允许大面积装饰性边框切碎阅读节奏
- 不允许代码块、引用块、表格样式过度喧宾夺主

### 2.2 层级清晰但克制

- 标题必须清楚
- 强调必须可见
- 链接必须可识别
- 代码块必须与正文分区
- 但整体不能像后台控件堆叠

### 2.3 明亮模式优先

- 首版先把明亮模式打磨到位
- 暗黑模式保持克制、低刺激、长时间阅读舒适

## 三、主题定义

### 3.1 `editorial`

适用场景：

- 前台文章详情页
- 后台弹窗最终预览

视觉特征：

- 呼吸感更强
- 标题更有编辑感
- 段落节奏更舒展
- 引用块更像杂志批注
- 图片与表格更柔和

### 3.2 `documentation`

适用场景：

- 技术方案页
- 文档型内容页
- 内容审核页

视觉特征：

- 信息密度略高
- 表格更规整
- 代码块权重更高
- 目录感更强

### 3.3 `minimal`

适用场景：

- 后台右侧实时预览
- 嵌入式预览卡片
- 小范围内容容器

视觉特征：

- 弱装饰
- 高可读
- 更紧凑
- 适合工作台环境

## 四、版式规范

### 4.1 正文宽度

- 标准阅读宽度不宜无限拉满
- 内容列应控制在舒适阅读区间
- 弹窗和后台工作台中允许更宽，但正文文本仍应有最大阅读宽度控制

### 4.2 段落节奏

- 正文行高建议保持在 `1.85-2`
- 相邻段落之间留白稳定
- 标题前后留白明显大于普通段落

### 4.3 标题层级

- `h1` 强势，但不夸张
- `h2` 是主章节切换点
- `h3` 用于内容分组
- `h4` 及以下只做辅助层级

### 4.4 目录锚点

- 标题锚点默认弱化
- hover 或聚焦时再显现
- 不要长时间占据注意力

## 五、正文元素 UI 规范

### 5.1 文本

- 正文字色应柔和但清晰
- `strong` 要明显强于正文
- `em` 要轻于 `strong`，避免与链接混淆
- 删除线、行内标记等后续扩展也要遵循同一套层级

### 5.2 链接

- 链接不能只靠颜色识别
- hover 时要有更明确反馈
- 长链接支持断行
- 外链可预留小图标位，但不强制首版上线

### 5.3 行内代码

- 行内代码必须与普通文字分离
- 背景要轻，不能像整块代码块
- 不允许使用过深的大色块破坏段落节奏

### 5.4 代码块

代码块是重点内容模块，必须具备完整视觉层级：

- 外层容器
- 顶部信息栏
- 语言标记
- 复制按钮位
- 内容滚动区
- 折叠状态预留

代码块要求：

- 要有独立边界
- 要有稳定圆角
- 头部和代码正文要分层
- 长代码允许横向滚动
- 超长代码后续支持折叠

### 5.5 引用块

- 引用块要有柔和背景和左侧强调
- 不要只是浏览器默认灰色斜体文本
- 引用内如包含列表或代码，也要保持结构清晰

### 5.6 列表

- 列表项之间要有轻微呼吸感
- 多层嵌套要保留清晰缩进
- 有序列表序号与正文之间要有足够间距

### 5.7 表格

- 表头要有识别度
- 表格边线不宜太重
- 外层容器支持横向滚动
- 单元格不允许撑破布局

### 5.8 图片

- 图片默认圆角处理
- 图片与正文之间保留足够间距
- 宽图在窄容器中不能溢出
- 后续如支持图片描述，应统一 caption 风格

### 5.9 分割线

- 使用轻线或渐变线
- 作用是章节换气，不是视觉阻断

## 六、代码主题 UI 规范

### 6.1 分离原则

内容主题与代码主题分离：

- 内容主题控制正文阅读气质
- 代码主题控制高亮色系

两者不能绑死。

### 6.2 推荐组合

- `editorial + github-light`
- `editorial + github-dark`
- `documentation + vitesse-light`
- `documentation + vitesse-dark`
- `minimal + github-light`

### 6.3 代码块头部

代码块头部建议包含：

- 左侧语言名
- 可选文件名
- 右侧复制按钮
- 后续可扩展折叠按钮

头部风格要求：

- 比代码正文更轻
- 不抢视觉焦点
- 但要足够清晰可用

## 七、后台预览与前台详情页差异

### 7.1 后台右侧实时预览

- 使用 `minimal`
- 保持清晰、紧凑、低噪声
- 更强调编辑过程中的结构检查

### 7.2 后台弹窗最终预览

- 使用 `editorial`
- 更接近用户最终看到的文章效果
- 更强调完整阅读感

### 7.3 前台文章详情页

- 使用 `editorial`
- 加入完整页面留白和目录联动
- 内容展示为页面主角

## 八、交互规范

### 8.1 复制代码

- hover 显示或始终弱显示均可
- 点击后反馈要轻量明确
- 不要弹出重型提示

### 8.2 折叠代码

- 默认只对超长代码开放
- 按钮文案简洁，避免教程式说明
- 展开后要保持滚动体验稳定

### 8.3 链接交互

- hover 有颜色或下划线变化
- focus 状态必须清楚
- 不使用夸张跳动动画

## 九、实现约束

- 正文展示样式必须走公共组件
- 不允许在页面里对 `h1`、`p`、`a`、`pre`、`code` 散写样式
- 不允许页面自己拼一套新的代码块 UI
- 内容主题和代码主题必须通过 props 或统一配置传入

## 十、当前定稿结论

文章内容展示的 UI 规范正式定为：

- 统一走公共正文渲染组件
- 支持 `editorial / documentation / minimal` 三套内容主题
- 支持独立代码主题
- 代码块采用“头部 + 内容区”的产品化结构
- 后台预览与前台详情页共用一套视觉体系，只在密度和场景上区分
