# 基础视觉与交互规范

## 文档目标

本文档用于定义 `Yunyu` 前端应用的长期设计基调，覆盖品牌气质、字体系统、明亮与暗黑主题、语义色 token、Nuxt UI 映射、组件状态、动效与页面文案规则。

本规范不是“当前页面怎么写”的临时说明，而是后续所有前台页面、后台页面、组件样式和交互实现的共同依据。

## 一、设计总纲

`Yunyu` 的目标不是做一个“普通博客模板”，而是做一个兼具审美表达、内容沉浸感与平台扩展能力的内容网站。

因此整体设计基调必须同时满足：

- 前台像一本被精心排版的当代数字杂志
- 后台像一个克制、专业、稳定的编辑工作台
- 前后台属于同一个品牌体系，而不是两套割裂产品

当前最终定调为：

- 设计模式：`Content First`
- 风格方向：`Editorial Minimalism + Swiss Modernism 2.0`
- 核心气质：克制、纯净、编辑感、秩序感、呼吸感、沉浸感
- 禁止方向：模板站感、控制台感过强、紫色默认风、过度玻璃拟物、炫技动画

## 二、设计原则

### 2.1 内容优先

组件不能抢内容的风头，排版、留白、层级、阅读节奏优先于组件展示欲。

### 2.2 品牌优先于组件库默认值

虽然项目使用 `Nuxt UI`，但 `Nuxt UI` 只是实现手段，不是审美定义者。

### 2.3 明亮模式优先

明亮模式是 `Yunyu` 的第一视觉主场，必须承担最完整的品牌表达。

### 2.4 暗黑模式克制

暗黑模式不是把页面简单反色，而是做低刺激、低反差、长时间阅读更舒适的夜间视觉。

### 2.5 用户界面不展示研发过程

联调说明、权限说明、技术状态、参数说明都不能出现在真实用户界面里，只能写在注释和文档中。

## 三、品牌视觉定位

### 3.1 视觉关键词

- Editorial
- Minimal
- Refined
- Calm
- Structured
- Atmospheric

### 3.2 页面气质

前台页面需要具备：

- 大留白
- 强标题层级
- 清晰栅格
- 柔和边界
- 低密度但高质量信息布局

后台页面需要具备：

- 高可读性
- 稳定卡片结构
- 低噪声背景
- 强调操作优先级
- 不过度修饰
- 带轻微悬浮感的编辑工作台气质

## 四、字体系统

结合 `$ui-ux-pro-max` 搜索结果，当前推荐采用：

- 标题字体：`Playfair Display`
- 正文字体：`Inter`
- 中文回退字体：`PingFang SC`、`Microsoft YaHei`

选择原因：

- `Playfair Display` 具备足够的编辑感和高级感，适合品牌标题、专题标题、文章大标题
- `Inter` 在界面、正文、表单、后台管理场景里具备更稳定的可读性
- 这组搭配适合内容站、杂志感页面、品牌化博客，不会过于花哨

### 4.1 字体使用规则

- 品牌标题、Hero 标题、文章大标题：优先使用标题字体
- 正文、导航、卡片说明、后台表单、后台表格：统一使用正文字体
- 不允许一个页面混用超过两类主字体
- 不允许为了“设计感”引入不稳定、过装饰的展示字体

### 4.2 字重建议

- Hero 标题：`600-700`
- 一级标题：`600`
- 二级标题：`600`
- 正文：`400-500`
- 辅助说明：`400`
- 按钮：`500-600`

## 五、色彩系统

## 5.1 总体策略

本项目不采用“艳丽品牌色主导全站”的策略，而采用：

- 低饱和纸感背景
- 深色文字建立阅读秩序
- 单一清透品牌色承担操作与强调
- 状态色只服务于反馈，不主导页面气质

### 5.2 明亮主题

明亮主题应呈现：

- 像纸张、不是像屏幕白板
- 有空气感，不发灰
- 有层次，不靠重阴影

推荐基础值：

- 页面背景：`#F5F8FC`
- 一级容器背景：`#FFFFFF`
- 二级容器背景：`#EEF4FA`
- 强文字：`#14213D`
- 正文文字：`#43536C`
- 弱文字：`#70819B`
- 边框：`#D8E3EE`
- 品牌主色：`#1D9BE6`
- 主色 hover：`#167FC0`
- 主色 active：`#11689E`

### 5.3 暗黑主题

暗黑主题应呈现：

- 深而不闷
- 稳定，不赛博
- 可长时间阅读

推荐基础值：

- 页面背景：`#0D1522`
- 一级容器背景：`#121C2B`
- 二级容器背景：`#182438`
- 强文字：`#EEF4FF`
- 正文文字：`#C5D1E4`
- 弱文字：`#8EA0BB`
- 边框：`#26344A`
- 品牌主色：`#4DB8F5`
- 主色 hover：`#77CBFB`
- 主色 active：`#98D8FF`

## 六、语义色 Token

页面和组件不能直接基于“蓝、灰、红”写样式，必须基于语义层工作。

当前统一语义 token：

- `bg-page`
- `bg-surface`
- `bg-surface-soft`
- `bg-surface-strong`
- `text-strong`
- `text-body`
- `text-muted`
- `text-inverted`
- `border-default`
- `border-muted`
- `primary`
- `secondary`
- `success`
- `warning`
- `danger`

### 6.1 状态色建议

- `primary`：品牌操作色
- `secondary`：次级强调或辅助入口
- `success`：保存成功、发布成功、完成态
- `warning`：待处理、风险提示、即将离开
- `danger`：删除、禁用、权限拒绝、错误反馈

建议值：

- `secondary-light: #7A8CA5`
- `secondary-dark: #9AAECC`
- `success-light: #1F9D73`
- `success-dark: #3CC693`
- `warning-light: #D99114`
- `warning-dark: #F3B54A`
- `danger-light: #D84F5F`
- `danger-dark: #FF7B88`

## 七、Nuxt UI 映射规则

结合 `$nuxt-ui` 的主题机制，当前项目统一映射如下：

- `primary`：品牌蓝系
- `neutral`：蓝灰中性色，不使用默认僵硬灰
- `success`：清晰但克制的青绿色
- `warning`：偏金黄，不用荧光黄
- `error`：偏莓红，不用刺眼纯红

### 7.1 当前建议映射

在 `app.config.ts` 中建议长期保持：

```ts
export default defineAppConfig({
  ui: {
    colors: {
      primary: 'sky',
      neutral: 'slate',
      success: 'emerald',
      warning: 'amber',
      error: 'rose'
    }
  }
})
```

### 7.2 Nuxt UI 使用规则

- 优先使用 `text-default`、`text-muted`、`bg-elevated`、`border-default` 这类语义类
- 不在业务页面大量写 `text-blue-500`、`bg-gray-100` 这类直接色
- 若某个组件默认样式不符合 `Yunyu` 气质，应通过 `app.config.ts` 或 `ui` slot 重写
- 前台页面不能过度暴露 Nuxt UI 的“默认控制台感”

## 八、布局系统

### 8.1 栅格原则

前台页面：

- 采用宽松栅格
- 模块之间保留明显呼吸空间
- 内容宽度以阅读舒适度优先，不追求大面积铺满

后台页面：

- 采用规整栅格
- 卡片布局优先保证信息清晰
- 列表、工具栏、筛选区遵循固定节奏
- 背景允许使用极轻的网格纹理和柔和径向光斑，但不能喧宾夺主

### 8.2 圆角系统

当前推荐圆角层级：

- 小组件：`12px`
- 输入框 / 按钮：`16px`
- 卡片：`24px`
- 大面板 / 页面容器：`32px`

规则：

- 不要一个页面里出现太多不同圆角等级
- 不使用过于尖锐的矩形，也不使用过于软糯的胶囊堆叠

### 8.3 阴影系统

当前推荐阴影原则：

- 轻阴影建立层级
- 大面板使用柔和长阴影
- 小组件仅在 hover 或浮层中适量增强阴影

不建议：

- 高饱和发光阴影
- 多重厚重阴影
- 玻璃卡片过度模糊

## 九、组件视觉规范

### 9.1 按钮

主按钮：

- 始终是页面中最明确的行动入口
- 需要足够强的对比度
- 亮暗主题都必须保持清晰可识别

次按钮：

- 用于次级操作
- 不与主按钮竞争
- 优先轮廓或弱填充样式

禁止：

- 主按钮颜色过浅
- 主按钮与背景接近
- hover 只改透明度导致文字不清晰

### 9.2 输入框

输入框必须满足：

- 宽度优先拉满容器
- 文字与 placeholder 对比清晰
- 聚焦时有明确 ring
- 背景与边框层级分明

输入框风格建议：

- 亮色模式：白底 + 低对比边框
- 暗色模式：深面板底 + 较柔和边框

### 9.3 卡片

卡片是 `Yunyu` 的核心容器形态。

卡片必须具备：

- 清晰边界
- 足够内边距
- 不依赖过度阴影来建立层级

前台卡片强调：

- 内容感
- 封面图与标题节奏
- 阅读欲望

后台卡片强调：

- 信息清晰
- 模块化结构
- 操作区与内容区分明
- 使用轻玻璃感或柔和半透明面板，但透明度必须足够高，不能影响阅读

### 9.4 后台侧边栏

后台侧边栏是整个后台最重要的长期识别元素，必须稳定、克制、可快速扫读。

规则：

- 侧边栏整体采用高透明度面板，不做重度实色填充
- 当前激活导航需要有明确的背景层、左侧强调条和更强文字对比
- hover 只允许轻位移、轻背景变化和轻边框变化，不允许大幅缩放
- 折叠状态下仍需保证图标居中、可识别、可点击
- 侧边栏底部允许放置主题切换和当前身份信息，但层级必须低于导航本身
- 主题切换优先采用单入口下拉或上拉菜单，不直接在侧边栏底部平铺多个按钮，便于后续扩展更多主题

### 9.5 后台表单

后台表单需要统一使用更饱满的输入面板，而不是默认的细边框控件。

规则：

- 输入框、选择器、文本域统一使用较高可见度背景和柔和阴影
- focus 时使用统一品牌色 ring，不允许页面各自发明不同焦点样式
- 搜索区表单与弹窗表单共享同一套输入风格
- placeholder 只作为辅助，不替代表单 label
- 长文本输入优先使用更舒展的圆角和更大的行高，保证编辑舒适度

### 9.6 后台表格操作按钮

后台表格右侧的操作按钮必须统一成紧凑型图标按钮，不允许页面各自混用文字按钮、幽灵按钮和默认图标按钮。

规则：

- 编辑、删除、预览等表格操作统一使用相同尺寸的图标按钮
- 默认态使用高可见度边框和轻面板背景
- hover 只允许轻微上浮、边框增强和背景变化
- 危险操作使用独立 danger 语义，不与普通操作共用同一视觉反馈
- 所有图标按钮必须具备 `aria-label` 或等价可访问描述

### 9.7 后台弹窗尺寸与层级

后台新增与编辑弹窗必须采用统一层级，不允许每个页面自己定义宽度、留白和头部结构。

规则：

- 简单表单弹窗使用标准宽度
- 包含多字段或双列表单使用宽弹窗
- 包含正文编辑或长文本区域时使用编辑器级宽弹窗
- 弹窗头部统一包含 eyebrow、标题、说明和语义图标区
- 弹窗底部统一固定为“取消 + 主操作按钮”的操作区结构

### 9.8 后台列表层级

后台列表的表头、空状态和分页区必须保持统一层级，避免列表区域像多个独立组件拼接。

规则：

- 表头统一使用较轻但清晰的分层背景，与列表正文形成稳定区隔
- 空状态统一包含语义图标、标题和一行辅助说明，不直接只放一段文字
- 分页区统一位于列表卡片底部，左侧展示当前页信息，右侧展示分页控件
- 列表正文 hover 反馈应低于主按钮强度，不与主要操作争抢视觉焦点

## 十、交互状态规范

所有交互组件必须定义：

- 默认态
- hover
- active
- focus
- disabled
- loading
- error

### 10.1 统一要求

- hover：150ms 到 220ms 的轻量变化
- active：更明确但不跳动
- focus：清晰焦点环，不能省略
- disabled：弱化但仍可读
- loading：按钮内文案和状态同时变化
- error：局部反馈，不放大全页惊扰

## 十一、动效规范

当前推荐动效方向：

- subtle fade
- micro elevation
- gentle reveal

建议时长：

- 按钮 / 输入框 / 小 hover：`150ms - 180ms`
- 卡片 / 抽屉 / 轻层级切换：`180ms - 240ms`
- 页面进入：`220ms - 320ms`

禁止：

- bounce
- 大幅 scale
- 高速旋转
- 与内容站调性不符的强科技风粒子动画

## 十二、用户可见文案规则

所有用户可见页面必须遵循：

- 不展示联调说明
- 不展示技术状态
- 不展示接口、角色、回跳参数、权限字段
- 不展示“当前阶段 / 演示中 / 后续接入”类文案
- 只展示用户完成当前操作所必须知道的内容

技术说明统一写在：

- 代码注释
- 前端文档
- 后端文档

## 十三、禁止事项

后续开发禁止出现以下问题：

- 大面积使用默认 Nuxt UI 风格不做调整
- 页面直接写死色值而不走 token
- 一个页面出现多套视觉语言
- 明亮模式和暗黑模式像两个产品
- 把研发过程说明暴露给最终用户
- 用动画弥补布局或排版问题

## 十四、当前阶段执行标准

从本文件开始，后续所有前端页面开发都必须遵循这套基调。

### 14.1 样式实现优先级

后续前端样式实现统一遵循以下优先级：

1. 优先使用 `Tailwind CSS` 工具类完成样式
2. 次优先使用 `Nuxt UI` 已提供的 `variant`、`size`、`color`、`ui` slot 等能力完成样式调整
3. 如果 `Nuxt UI` 已经有对应组件，优先基于该组件实现，不重复手写同类结构
4. 除非用户明确说明，否则不要额外手写自定义 CSS 来完成页面级或组件级样式

补充规则：

- 能用 `Tailwind CSS` 表达的样式，不进入自定义样式文件
- 能用 `Nuxt UI` 组件和配置完成的样式，不再额外包一层自定义 CSS 方案
- 自定义 CSS 只用于以下场景：用户明确要求、`Tailwind CSS` 与 `Nuxt UI` 都难以合理表达、或涉及全局 design token / 主题变量
- 如需新增自定义 CSS，应先判断是否真的无法通过 `Tailwind CSS` 与 `Nuxt UI` 落地

在进入新的页面设计或组件开发前，至少先检查：

- 是否使用统一字体系统
- 是否使用统一语义色
- 是否遵循亮暗主题对应关系
- 是否符合 Nuxt UI 的映射规则
- 是否减少了无必要提示文案
- 是否保持了“内容优先、克制表达”的整体气质

后续如基础色调、字体系统、组件气质或主题策略发生变化，必须先更新本文档，再进入实现阶段。
