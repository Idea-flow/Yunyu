# 前端路由与页面设计

## 文档目标

本文档用于明确 `Yunyu` 前端项目在第一阶段的技术边界、主题系统、UI 分层策略、前后台页面边界与后续扩展方向，避免前端工程在实现过程中失去统一性。

## 一、当前技术结论

当前前端项目采用以下组合：

- 前端框架：`Nuxt 4`
- 样式体系：`Tailwind CSS`
- 组件体系：`@nuxt/ui`
- 主题机制：`@nuxtjs/color-mode`
- 运行模式：前台以 `SSR` 为主，后台以登录后管理界面为主

当前阶段的核心原则不是“为了简单快速堆页面”，而是为“最美博客 / 内容站”建立长期可维护的设计底层。

## 二、前后台 UI 设计边界

### 2.1 统一的部分

前台与后台必须统一以下内容：

- 主题切换机制
- 设计 token
- 基础语义色
- 间距、圆角、阴影、边框层级规则
- 交互状态规则

统一这些内容的目的是：

- 保证整个产品属于同一品牌体系
- 让明亮、暗黑、系统模式能全局统一运作
- 降低后续维护成本

### 2.2 区分的部分

前台与后台不应追求“长得完全一样”。

前台更强调：

- 品牌感
- 内容表达
- 排版节奏
- 沉浸感
- 阅读体验

后台更强调：

- 信息密度
- 操作效率
- 稳定一致
- 长时间使用的舒适度

因此当前结论是：

- 前后台共享一套设计底层
- 前后台在页面表达和信息组织上分别设计

## 三、主题系统设计

### 3.1 支持模式

全站统一支持三种模式：

- `light`
- `dark`
- `system`

其中：

- `system` 表示跟随设备系统主题
- 默认策略为 `system`

### 3.2 技术实现

当前主题机制使用 `@nuxtjs/color-mode` 统一管理。

原因：

- `Nuxt 4` 采用服务端渲染，主题切换不能只依赖客户端临时处理
- `color-mode` 更适合处理主题持久化、系统模式与 SSR 场景的首屏体验
- 能作为前台与后台统一主题机制的基础

### 3.3 主题规则

主题系统必须遵循以下规则：

- 不允许在业务组件中大面积写死颜色值
- 优先使用语义类名与设计 token
- 明亮与暗黑模式必须共享同一套语义层
- 系统模式只是主题来源，不是独立第四套设计

## 四、UI 组件策略

### 4.1 为什么使用 `@nuxt/ui`

当前项目决定引入 `@nuxt/ui`，原因是：

- 与 `Nuxt 4` 集成自然
- 与 `Tailwind CSS` 配合顺畅
- 能提供稳定的基础组件能力
- 后续前后台都可以复用同一套组件层

### 4.2 使用原则

虽然引入了 `@nuxt/ui`，但不意味着直接使用默认风格就结束。

必须遵循以下原则：

- `@nuxt/ui` 是组件基础设施，不是最终品牌视觉
- 最终视觉仍需通过主题变量、token、布局与页面设计进行定制
- 前台页面要避免“控制台风”
- 后台页面要避免“内容站风”

### 4.3 用户可见文案规则

所有用户可见页面必须遵循以下规则：

- 不展示开发联调说明
- 不展示接口、权限、回跳参数等技术细节
- 不展示“当前阶段 / 后续接入 / 演示中”这类研发过程文案
- 页面只保留用户完成操作所必需的提示
- 技术说明、联调说明、实现细节只写在代码注释和文档中

## 五、前端目录建议

当前 `yunyu-web` 建议逐步演进为以下结构：

```text
app/
  app.vue
  assets/
    css/
      main.css
  components/
    common/
    content/
    admin/
    theme/
  layouts/
    default.vue
    admin.vue
  pages/
    index.vue
    posts/
    topics/
    admin/
  composables/
  middleware/
```

目录设计原则：

- `common/`：前后台共用组件
- `content/`：前台内容展示组件
- `admin/`：后台管理组件
- `theme/`：主题切换与主题辅助能力

## 六、第一阶段页面范围

第一阶段优先页面建议如下：

- 首页
- 文章详情页
- 专题页基础能力
- 共享登录页
- 后台首页
- 后台文章管理页

这些页面构成第一阶段前台体验和后台管理的最小闭环。

## 七、路由约定

当前第一阶段优先约定以下页面路由：

- `/`：前台首页
- `/login`：前台与后台共用登录页
- `/admin`：后台首页
- `/admin/posts`：后台文章管理页
- `/admin/users`：后台用户管理页

其中登录页为共享入口，后续通过 `redirect` 参数和真实身份信息决定回跳目标，例如：

- `/login?redirect=/admin`

## 八、页面设计文档

### 8.0 后台页面通用规则

#### 布局规则

- 后台页面进入主内容区后的第一块区域必须是搜索区域
- 搜索区域只承载搜索、筛选和搜索按钮，不混入统计卡片或说明卡片
- 后台页面的搜索区域必须默认支持换行，不能假设所有筛选项永远在一行内排完
- 搜索框、筛选项组、操作按钮组都需要使用可伸缩布局，宽度不足时自动折到下一行
- 搜索区域下方才进入操作区域，例如增加、批量操作、导出等
- 列表区域位于操作区域之后，分页区域位于列表底部
- 搜索、增加、保存等后台主动作统一使用高可见主按钮组件，确保明亮与暗黑模式下都清晰可见
- 用户界面不展示研发说明、结构说明和阶段性占位说明

### 8.1 共享登录页

#### 页面目标

统一承接前台与后台登录入口，避免维护两套登录页和重复交互逻辑。

#### UI 设计

- 左侧为品牌说明与登录语义展示
- 右侧为登录表单卡片
- 页面需兼顾品牌感与通用性

#### 布局设计

- 桌面端采用双栏布局
- 移动端自动折叠为单栏纵向结构
- 登录表单保持独立卡片容器，提升聚焦感

#### 交互设计

- 输入账号与密码后提交
- 若信息不完整，给出明确反馈
- 登录请求由前端统一接口层直接访问 Spring Boot 认证接口
- 前端统一通过 `useApiClient` 管理接口请求入口
- 登录成功后根据 `redirect` 参数回跳
- 登录成功后将访问令牌写入前端 Cookie，供当前用户恢复和后台鉴权复用
- 当回跳目标为后台且当前用户不是站长时，只保留必要的失败反馈
- 未指定回跳地址时默认进入 `/admin`

### 8.2 后台首页

#### 页面目标

作为后台管理工作台的第一屏，为站长提供运营总览、快捷操作、待处理事项和系统状态信息。

#### UI 设计

- 使用 `Nuxt UI` 的 dashboard 结构作为后台骨架
- 左侧为可折叠导航
- 主内容区展示总览卡片、快捷操作、进度状态和待处理事项
- 页面整体强调效率、层级和稳定性

#### 布局设计

- 使用后台独立布局 `admin`
- 左侧导航固定承载后台入口与主题切换
- 右侧内容区采用多卡片模块化布局
- 移动端优先保证信息模块可纵向堆叠

#### 交互设计

- 进入页面前先通过后台中间件检查登录态与站长角色
- 首页中的文章相关快捷入口统一跳转到 `/admin/posts`
- 页面只展示与管理工作台相关的必要信息，不展示研发过程说明

### 8.3 后台文章管理页

#### 页面目标

作为站长的核心内容运营页面，承载文章检索、状态筛选、发布时间判断、内容质量巡检与进入编辑流程的主入口。

#### UI 设计

- 页面采用后台统一工作台骨架，延续左侧导航 + 主内容区结构
- 首行为搜索区域，包含关键词、状态、排序和显式搜索按钮
- 搜索区域下方为操作区域，保留“增加”按钮作为主动作入口
- 主内容区下方为文章列表区，使用稳定的表格化卡片样式呈现标题、状态、专题、最近更新时间与操作入口
- 列表底部统一保留分页区域

#### 布局设计

- 桌面端采用纵向结构，依次为搜索区、操作区、列表区和分页区
- 列表区宽度优先，保证文章标题、状态与更新时间具备舒适阅读长度
- 搜索区采用可换行的弹性布局，桌面端宽度不足时也必须自然折行
- 移动端退化为纵向堆叠，先显示搜索与操作，再显示列表内容

#### 交互设计

- 进入页面前先通过后台中间件检查登录态与站长角色
- 搜索框用于按标题和 Slug 快速过滤文章
- 状态筛选支持在“全部 / 草稿 / 已发布 / 已下线”之间切换
- 搜索按钮触发文章列表查询
- 操作区中的“增加”按钮打开文章创建弹窗
- 列表右侧提供编辑和删除操作
- 编辑按钮会先加载文章详情，再打开编辑弹窗
- 删除按钮使用二次确认弹窗
- 列表底部通过分页组件切换页码
- 用户可见提示仅保留内容管理所需信息，不出现接口、权限、联调等技术描述

### 8.4 后台用户管理页

#### 页面目标

为站长提供统一的用户查询、创建、编辑、状态调整与删除入口，用于支撑后台账号体系和基础权限管理。

#### UI 设计

- 页面延续后台工作台基调，采用稳定、克制的管理界面风格
- 首行为搜索区域，包含关键词输入、筛选条件和显式搜索按钮
- 搜索区域下方为操作区域，保留“增加”按钮作为主动作入口
- 主区域使用后台统一表格容器展示这个页面相关的数据,表格最右侧有个操作栏, 里面有修改,删除按钮
- 新建和修改动作统一使用弹窗窗口承接

#### 布局设计

- 桌面端采用纵向结构，依次为搜索区、操作区、列表区和分页区
- 搜索区采用支持换行的弹性布局，关键词、筛选项和按钮区在宽度不足时自动折行
- 移动端同样保持纵向结构，只在列表行内压缩信息密度
- 弹窗表单独立于列表区，避免编辑状态打断列表阅读

#### 交互设计

- 支持按邮箱和用户名进行搜索
- 支持按角色和状态进行筛选
- 搜索条件变更后由搜索按钮触发查询
- 新建用户时点击“增加”按钮打开弹窗，填写邮箱、用户名、密码、角色和状态
- 编辑用户时点击列表右侧“修改”按钮打开弹窗，允许更新基础信息、角色、状态和密码
- 删除用户前使用二次确认弹窗，避免误删
- 列表底部支持分页切换
- 页面不展示技术性说明，仅保留对管理动作有帮助的必要反馈
- 未登录时自动跳转到共享登录页
- 非站长用户访问后台时自动跳转回共享登录页并展示禁止提示
- 后台权限判定依赖当前用户角色 `SUPER_ADMIN`

## 九、当前落地结论

当前已经明确：

- 前端使用 `Nuxt 4`
- 使用 `Tailwind CSS` 作为样式基础
- 使用 `@nuxt/ui` 作为组件基础设施
- 使用 `@nuxtjs/color-mode` 统一管理主题
- 默认支持 `light / dark / system`
- 前后台共享主题底层，但页面表达分开设计
- 登录页为前台与后台共用入口
- 后台首页优先作为第一批后台页面落地
- 前端已接入真实后端认证接口
- 前端统一通过 `useApiClient` 访问后端
- 后端已开放全局跨域策略，允许前后端分离开发阶段直接联调

后续如有设计系统、路由层级、布局体系变化，必须先同步更新本文档，再进入实现阶段。
