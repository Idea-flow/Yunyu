# 后台评论树形审核改版方案

## 1. 文档目标

本文档用于明确后台评论管理页从“扁平表格列表”升级为“按文章分组的树形审核页”的落地方案。

本文档重点回答 3 个问题：

- 后台评论接口应该返回什么结构
- 前端页面应该如何展示评论主楼与回复流
- 按状态筛选时应该如何搜索和呈现结果

本文档是后续接口开发、前端改版和交互验收的执行依据，不是纯视觉灵感记录。

## 2. 当前问题

当前后台评论页采用扁平表格形式，每一行展示一条评论。

当前形式存在以下问题：

- 评论上下文割裂，难以一眼看出“这是谁回复谁”
- `评论 ID`、`回复 ID`、`楼层根 ID` 同时出现，对运营和站长不友好
- 同一篇文章下的评论被打散到分页列表里，不利于集中审核
- 即使后端已有 `replyCommentId` 和 `rootId`，前端也没有把它真正组织成更适合管理的结构

因此后台评论页需要从“数据表格”改成“审核视图”。

## 3. 改版目标

后台评论管理页本次改版目标如下：

1. 以“文章”为第一层分组单位
2. 每篇文章下先展示一级评论
3. 一级评论下统一展示回复流
4. 后台视觉上只做两层，不继续做无限缩进
5. 所有二级及更深回复，都以“回复 @某人”的形式挂在对应主评论下
6. 每条评论都清晰展示当前审核状态
7. 每条评论右侧保留审核、驳回、删除等操作
8. 评论内部 ID 降级为辅助信息，不再作为主展示列

## 4. 设计原则

### 4.1 为什么视觉上只做两层

评论系统的数据层级仍然允许：

- 一级评论
- 二级回复
- 三级回复
- 更深层回复

但是后台管理页不建议继续无限树形缩进，原因如下：

- 审核场景关注的是关系与状态，而不是论坛式楼中楼沉浸阅读
- 层级越深，后台内容区域越窄
- 多级缩进会显著降低扫描效率
- 对运营来说，“这条回复是回给谁的”比“它在第几层缩进”更重要

因此本次方案采用：

- 数据结构保留真实回复关系
- 视觉结构只展示为两层
- 所有非根评论统一挂在对应根评论下的回复区
- 每条回复通过 `replyToUserName` 表达“回复 @谁”

### 4.2 和前台评论展示的关系

后台评论页的展示思路可以参考前台文章详情页评论区，但不能完全照搬。

后台需要在前台评论结构的基础上增强以下能力：

- 显示评论审核状态
- 提供评论审核、驳回、删除操作
- 在状态筛选时保留必要上下文
- 弱化内部 ID、强化审核信息

因此可以理解为：

- 前台评论结构
- 加上后台审核能力
- 加上文章级分组

## 5. 总体结构

后台评论页改版后的结构建议如下：

1. 顶部页面标题区
2. 筛选区
3. 按文章分组的评论审核卡片列表
4. 分页区

其中内容区层级为：

- 文章卡片
- 一级评论卡片
- 一级评论下的回复流
- 每条评论右侧操作区

## 6. 后台接口设计

## 6.1 为什么不能只靠前端把现有列表凑成树

当前后台接口返回的是“按评论分页”的扁平列表。

如果仅由前端把当前页评论拼成树，会有以下问题：

- 某条回复在当前页，但它的父评论不在当前页
- 同一篇文章的评论会被分页切散
- 状态筛选后更容易丢失上下文
- 前端需要额外补查数据，逻辑会越来越复杂

因此最稳妥的方式是新增一个后台接口，直接返回“按文章分组 + 主评论 + 回复流”的审核视图数据。

## 6.2 新接口建议

建议新增接口：

`GET /admin/comments/thread-groups`

用途：

- 面向后台评论审核页
- 返回按文章分组后的评论树视图
- 支持关键词、文章、状态、分页筛选

## 6.3 查询参数设计

```ts
/**
 * 后台评论树形审核查询参数。
 * 作用：统一描述后台评论审核页在按文章分组视图下使用的筛选条件与分页参数。
 */
export interface AdminCommentThreadGroupQuery {
  keyword?: string
  postId?: number
  status?: 'PENDING' | 'APPROVED' | 'REJECTED'
  pageNo?: number
  pageSize?: number
}
```

字段说明：

- `keyword`
  支持搜索评论内容、文章标题、文章 slug、用户名、邮箱，以及纯数字 ID
- `postId`
  精确筛选某篇文章
- `status`
  按评论状态筛选
- `pageNo`
  当前页码
- `pageSize`
  每页文章分组数，不再是每页评论数

## 6.4 返回结构设计

```ts
/**
 * 后台评论树形审核页响应。
 * 作用：承接按文章分组后的评论审核视图及分页信息。
 */
export interface AdminCommentThreadGroupListResponse {
  list: AdminCommentThreadGroup[]
  total: number
  pageNo: number
  pageSize: number
  totalPages: number
}

/**
 * 后台评论文章分组类型。
 * 作用：描述一篇文章下的评论审核概览和对应评论树。
 */
export interface AdminCommentThreadGroup {
  postId: number
  postTitle: string
  postSlug: string
  totalCommentCount: number
  pendingCommentCount: number
  approvedCommentCount: number
  rejectedCommentCount: number
  latestCommentTime: string | null
  roots: AdminCommentThreadRootItem[]
}

/**
 * 后台评论主评论类型。
 * 作用：描述后台审核视图中的一级评论及其回复流。
 */
export interface AdminCommentThreadRootItem {
  id: number
  postId: number
  userId: number
  userName: string
  userEmail: string
  content: string
  status: 'PENDING' | 'APPROVED' | 'REJECTED'
  ip: string
  createdTime: string
  updatedTime: string
  visibleOnSite: boolean
  hasMatchingDescendant: boolean
  replies: AdminCommentThreadReplyItem[]
}

/**
 * 后台评论回复项类型。
 * 作用：描述根评论下面的回复流项，并保留“回复 @谁”的关系信息。
 */
export interface AdminCommentThreadReplyItem {
  id: number
  postId: number
  rootId: number
  replyCommentId: number | null
  replyToUserName: string | null
  userId: number
  userName: string
  userEmail: string
  content: string
  status: 'PENDING' | 'APPROVED' | 'REJECTED'
  ip: string
  createdTime: string
  updatedTime: string
  visibleOnSite: boolean
  isMatchedByFilter: boolean
}
```

## 6.5 字段设计说明

### 6.5.1 文章分组字段

- `postId`
  用于文章筛选、跳转和操作关联
- `postTitle`
  文章标题
- `postSlug`
  文章 slug，用于跳转前台详情页
- `totalCommentCount`
  当前文章下评论总数
- `pendingCommentCount`
  当前文章下待审核评论数
- `approvedCommentCount`
  当前文章下已通过评论数
- `rejectedCommentCount`
  当前文章下已驳回评论数
- `latestCommentTime`
  当前文章最近一条评论时间

### 6.5.2 主评论字段

- `id`
  评论 ID，仅作辅助信息展示
- `visibleOnSite`
  是否前台可见
  规则建议为：`status === 'APPROVED'`
- `hasMatchingDescendant`
  在状态筛选或关键词筛选时，标记该主评论下是否存在命中的回复

### 6.5.3 回复字段

- `replyCommentId`
  实际回复的评论 ID
- `replyToUserName`
  实际回复对象用户名
- `isMatchedByFilter`
  当前回复是否是本次筛选的直接命中项

说明：

- 回复项不再继续嵌套 `children`
- 即使数据库存在三级或更深回复，也统一压平到 `replies`
- 真实关系通过 `replyToUserName` 表达

## 6.6 后端组装规则

后端组装建议如下：

1. 先按筛选条件找到命中的评论集合
2. 取出这些评论涉及的文章 ID
3. 按文章分页，而不是按评论分页
4. 对每篇文章：
   - 找出所有根评论
   - 找出所有回复评论
   - 根评论挂到 `roots`
   - 所有非根评论挂到各自根评论的 `replies`
5. 如果当前存在状态筛选或关键词筛选：
   - 直接命中的评论标记为命中
   - 命中的回复所在根评论保留作为上下文
   - 非必要上下文可以不返回

## 7.2 页面信息层级

### 7.2.1 文章卡片头部

建议展示：

- 文章标题
- `slug`
- 评论总数
- 待审核数
- 最近评论时间
- 跳转前台文章链接

### 7.2.2 一级评论卡片

建议展示：

- 作者名
- 邮箱
- 状态 badge
- 时间
- 评论正文
- IP
- 辅助信息：评论 ID
- 右侧操作按钮

### 7.2.3 回复流

建议展示：

- `回复 @某某`
- 当前作者
- 状态 badge
- 时间
- 回复正文
- 辅助信息：评论 ID
- 右侧操作按钮

### 7.2.4 内部 ID 的展示策略

内部 ID 不再作为主列展示。

建议处理方式：

- 放在作者信息下方或底部灰字中
- 样式弱化
- 仅供问题排查时使用

不建议再单独保留：

- 评论 ID 列
- 回复 ID 列
- 楼层根 ID 列

## 7.3 操作区设计

每条评论最右侧保留操作区。

建议操作如下：

- `通过`
- `驳回`
- `删除`

按钮状态规则：

- `PENDING`
  - 可 `通过`
  - 可 `驳回`
  - 可 `删除`
- `APPROVED`
  - `通过` 置灰
  - 可 `驳回`
  - 可 `删除`
- `REJECTED`
  - 可 `通过`
  - `驳回` 置灰
  - 可 `删除`

## 7.4 审核可见性提示

建议每条评论增加“前台可见性”辅助文案：

- `待审核，前台暂不可见`
- `已通过，前台可见`
- `已驳回，前台不可见`

这样比单独展示状态 badge 更直观。

## 8. 状态筛选交互规则

## 8.1 筛选目标

状态筛选的核心目标不是简单过滤行，而是：

- 让站长快速看到需要处理的评论
- 同时保留最少必要的上下文

因此状态筛选不建议做成“只剩一堆孤立评论”。

## 8.2 推荐筛选模式

本次后台评论页建议采用以下模式：

- 页面仍按文章分组
- 分组内优先展示命中的评论
- 命中的回复所在主评论保留作为上下文
- 非必要的非命中回复不展示

这是后台审核最实用的一种模式。

## 8.3 不同状态下的展示规则

### 8.3.1 状态为 `ALL`

- 展示所有文章分组
- 每篇文章下展示完整主评论和回复流
- 适合整体浏览

### 8.3.2 状态为 `PENDING`

- 仅展示包含待审核评论的文章
- 如果一级评论本身是待审核，正常展示
- 如果一级评论已通过，但其下有待审核回复：
  - 该一级评论保留
  - 标记为“上下文评论”
  - 待审核回复高亮展示
- 非命中的通过/驳回回复可隐藏

### 8.3.3 状态为 `APPROVED`

- 仅展示包含已通过评论的文章
- 适合排查前台当前可见互动内容
- 若命中的是回复，则保留主评论上下文

### 8.3.4 状态为 `REJECTED`

- 仅展示包含已驳回评论的文章
- 适合复查被驳回内容
- 若命中的是回复，则保留主评论上下文

## 8.4 命中与上下文的样式建议

筛选时建议区分两种评论：

- 直接命中评论
- 上下文评论

样式建议：

- 直接命中评论
  - 正常不透明
  - 高亮边框或状态强调
- 上下文评论
  - 降低背景强调
  - 附加“上下文”小标记

这样既能保留关系，也不会让用户误以为所有评论都需要处理。

## 8.5 关键词搜索规则

关键词搜索建议保持与现有后台一致，但展示方式改为树形：

- 匹配评论内容
- 匹配文章标题
- 匹配文章 slug
- 匹配用户名
- 匹配邮箱
- 如果是纯数字，匹配评论 ID、文章 ID、用户 ID

展示规则：

- 如果命中主评论，主评论及其回复流展示
- 如果命中回复，则保留根评论上下文，并高亮命中的回复

## 9.2 视觉重点

这版页面视觉重点应该是：

- 文章分组清晰
- 评论正文可读
- 回复关系明确
- 操作区固定且稳定

而不是：

- ID 信息突出
- 表格列过多
- 字段解释过重

## 10. 开发分期建议

## 10.1 第一阶段

目标：完成可用版本

范围：

- 新增后台树形审核接口
- 评论页从表格改成文章分组审核视图
- 支持关键词、文章 ID、状态筛选
- 支持评论通过、驳回、删除
- 支持分页

## 10.2 第二阶段

目标：提升审核效率

范围：

- 增加“仅看待审核”快捷切换
- 增加“折叠非命中回复”
- 增加“上下文评论”样式优化
- 增加文章卡片折叠

## 11. 方案结论

本次后台评论页改版采用以下最终方案：

1. 新增后台评论树形审核接口，不复用原扁平列表接口直接拼装
2. 页面改为“按文章分组”
3. 视觉上只展示两层
4. 所有非根评论统一放入根评论下的回复流
5. 回复关系通过“回复 @谁”表达
6. 每条评论都展示当前审核状态和前台可见性
7. 状态筛选保留必要上下文，不做孤立列表
8. 评论 ID、回复 ID、根 ID 降级为辅助信息

后续开发时，应以本方案为准，不再回到扁平表格方向。
