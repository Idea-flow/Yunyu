# 部署与运行架构

## 文档目标

本文档用于确定 `云屿 / Yunyu` 的前后端部署位置、反向代理策略、资源消耗预期与后续扩展路线。

## 一、当前阶段推荐部署方案

### 方案 A：推荐方案

- `Nuxt 4`：部署到 Cloudflare
- `Spring Boot 4`：部署到云服务器 / 容器
- `PostgreSQL` 或 `MySQL 8`：托管数据库或独立数据库实例
- `Nginx`：部署在后端主机前作为反向代理

### 方案说明

这套方案适合“前台访问多、后台写操作较少、需要 SEO 与全球访问速度”的内容平台项目。

## 二、流量路径

### 前台访问路径

1. 用户访问站点域名
2. Cloudflare 处理 CDN、缓存与边缘 SSR
3. Nuxt 渲染页面
4. 页面需要业务数据时调用 Spring Boot API
5. Spring Boot 访问数据库并返回数据

### 后台访问路径

1. 管理员访问后台域名
2. 请求进入 Nuxt 后台应用
3. 后台通过认证接口与业务接口访问 Spring Boot
4. Spring Boot 完成权限校验与业务处理

## 三、资源消耗判断

### Nuxt 4

如果前台主要承担 SSR、内容展示和轻量 BFF 聚合，资源消耗通常可控。

### Spring Boot 4

资源消耗主要来自：

- JVM 常驻内存
- 认证过滤链
- 业务查询
- 连接池

### 数据库

数据库的稳定性和 IO 质量通常比“多省几百 MB 内存”更重要。

## 四、初期资源建议

如果是项目第一阶段，推荐：

- 应用服务器：`2C / 4G`
- 数据库：托管服务优先

如果预算较紧：

- 也可从 `2C / 2G` 起步，但需要严格控制 JVM 参数、连接池与后台任务数量

## 五、扩展路线

### 第一阶段

- 单体 Spring Boot
- 单 Nuxt 应用
- 单数据库

### 第二阶段

- 增加对象存储
- 增加搜索服务
- 增加缓存

### 第三阶段

- 视业务量拆分鉴权、内容、交易等服务

## 六、数据库选择结论

本项目长期更偏向 `PostgreSQL`，原因是更适合内容平台与商品平台的复杂查询和扩展属性设计。

但如果团队对 `MySQL 8` 更熟悉，第一阶段可以直接使用 `MySQL 8`，不会影响项目起步。
