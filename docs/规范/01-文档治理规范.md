# 文档治理规范

## 文档目标

本文档用于规定本项目未来所有产品文档、技术文档、架构文档的目录分类、命名规则与变更同步要求。

## 一、目录分类规范

### 1.1 固定目录

- `docs/架构/`：技术选型、部署架构、系统边界、认证方案
- `docs/决策记录/`：重要技术决策记录
- `docs/产品/`：产品规划、需求、页面结构、业务流程、运营方案
- `docs/前端/`：前端路由、前端认证、UI 规范、主题系统、交互与页面设计
- `docs/技术/`：数据库设计、领域模型、通用技术方案
- `docs/后端/`：后端模块结构、接口设计、认证权限、依赖说明
- `docs/运维/`：部署、备份、监控、发布、应急流程
- `docs/规范/`：文档规范、命名规范、协作规则

### 1.2 新文档放置规则

- 讨论业务价值、用户流程、页面结构，放到 `产品/`
- 讨论前端页面结构、前端认证联调、UI 规范、主题系统，放到 `前端/`
- 讨论数据库、领域模型、内容渲染等通用实现，放到 `技术/`
- 讨论后端项目结构、接口协议、权限设计、依赖选型，放到 `后端/`
- 讨论全局方案和跨模块决策，放到 `架构/`
- 讨论为什么选择某个方案而不是另一个方案，新增到 `决策记录/`
- 讨论上线、部署、环境、监控，放到 `运维/`

## 二、命名规范

文档文件名统一采用：

- `序号-中文名称.md`

例如：

- `01-总体技术架构.md`
- `02-部署与运行架构.md`
- `03-认证与权限设计.md`
- `01-基础视觉与交互规范.md`

这样做的目的是：

- 方便排序
- 方便逐步扩写
- 方便长期维护

## 三、变更同步要求

以下任一内容发生变化时，必须同步更新相关文档：

- 页面结构变化
- 登录方式变化
- 权限模型变化
- 接口字段变化
- 数据表结构变化
- 部署方式变化
- 技术栈变化
- 商品模型变化
- 虚拟产品交付方式变化

## 四、文档更新原则

### 4.1 先改文档还是先改代码

对于以下类型变更，优先改文档再改代码：

- 架构层变更
- 模块边界变更
- 权限模型变更
- 数据模型变更
- 目录结构变更

对于以下类型变更，可以代码与文档同一轮完成：

- 小字段新增
- 样式调整
- 小型交互优化

### 4.2 一次变更至少检查的文档范围

如果后续修改某项逻辑，至少检查以下文档是否需要更新：

- 当前功能对应的产品文档
- 当前模块对应的技术文档或后端文档
- 若影响系统边界，则更新架构文档
- 若改变技术决策，则新增或更新决策记录
- 若新增、删除或调整项目目录结构，则同步更新对应结构说明文档

### 4.3 目录结构变更同步规则

如果本次修改涉及以下内容：

- 仓库目录变化
- 后端包结构变化
- 模块目录变化
- 运维脚本目录变化

则必须同时完成：

1. 更新实际目录结构
2. 更新对应说明文档
3. 在本次回复中明确告知用户“本次同步更新到了哪篇文档”

当前后端目录结构说明主文档为：

- `docs/后端/01-后端结构与依赖说明.md`

## 五、推荐维护流程

每次需求或逻辑变更时，按以下顺序执行：

1. 判断变更属于产品、技术、架构还是运维层
2. 更新对应文档
3. 如涉及重大决策，补一篇 ADR
4. 再进入代码修改
5. 代码完成后回查文档是否与实现一致
6. 若涉及目录结构变更，明确告知用户已同步更新到哪篇文档
