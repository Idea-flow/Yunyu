# Yunyu draw.io 配图实施方案

## 文档目标

本文档用于说明 `Yunyu` 项目里 draw.io 图如何落地，包括：

- 哪些文档适合配图
- 图源和图片放哪里
- 文件怎么命名
- 什么时候需要导出图片
- 文档里如何插图

## 一、适合配图的文档类型

优先给以下文档配图：

- 产品需求文档
- 技术实现方案
- 数据模型设计
- 接口链路说明
- 架构与权限边界说明

当前 `Yunyu` 最适合配图的内容包括：

- 内容访问控制
- 附件上传链路
- 认证与权限边界
- Markdown 渲染与存储链路
- 后台配置项与前台行为映射

## 二、正式目录规范

正式 draw.io 产物统一放在“对应文档的资源目录”里。

规则如下：

- 图源目录：`资源/文档名/drawio/`
- 图片目录：`资源/文档名/img/`

例如：

- `docs/技术/资源/04-内容访问控制技术实现方案/drawio/`
- `docs/技术/资源/04-内容访问控制技术实现方案/img/`

## 三、正式产物建议

推荐保留以下产物：

1. `.drawio`
2. `.spec.yaml`
3. `.arch.json`
4. `.png` 或 `.svg`

说明：

- `.drawio` 是正式图源
- `.spec.yaml` 是可维护的结构化规范
- `.arch.json` 是 CLI 生成的中间结构
- `.png` 或 `.svg` 用于插入 Markdown

## 四、命名规范

建议统一采用：

- `01-图名.drawio`
- `01-图名.spec.yaml`
- `01-图名.arch.json`
- `01-图名.png`

对于同一文档下的多张图，按顺序编号即可。

例如：

- `01-内容访问控制整体结构图.drawio`
- `02-文章访问与尾部隐藏内容判定流程图.drawio`

## 五、插图建议

Markdown 中建议按“图标题 + 图片 + 简短说明”的方式插入。

示例：

```md
## 配图：内容访问控制整体结构

![内容访问控制整体结构](./资源/04-内容访问控制技术实现方案/img/01-内容访问控制整体结构图.png)

上图用于说明文章权限、尾部隐藏内容权限、授权记录和前后端交互之间的关系。
```

## 六、风格建议

`Yunyu` 的文档配图优先追求：

- 信息密度高
- 边界清楚
- 命名准确
- 留白适中
- 颜色克制

不建议：

- 使用过多品牌图标
- 使用高饱和大面积渐变
- 节点文案写成长句

## 七、当前项目推荐配图数量

单篇文档推荐：

- 1 到 3 张关键图

通常足够。

原则：

- 用图补齐结构理解成本
- 不用图替代整篇正文

## 八、当前结论

对于 `Yunyu` 当前阶段，最好的配图策略不是“多”，而是“准”。

每篇核心文档只补最关键的 1 到 3 张图，就能显著降低理解成本，同时保持维护成本可控。
