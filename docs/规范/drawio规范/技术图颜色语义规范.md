# 技术图颜色语义规范

> 本文件用于统一技术图中的颜色语义。
>
> 技术图里的颜色，不是为了“看起来丰富”，而是为了帮助读者更快理解结构、归属和重点。

## 一、目的

本规范主要解决下面这些常见问题：

- 一张图里颜色太多，读者抓不住重点
- 同一种颜色表达了两种不同含义
- 同一个含义在一张图里用了多套颜色
- 颜色虽然好看，但会让人误解模块归属
- 颜色成了装饰，不再承担信息表达

目标是让颜色做到：

- 有语义
- 有边界
- 有层级
- 有克制

## 二、总原则

### 1. 颜色先服务于理解，再服务于美观

优先级：

1. 语义正确
2. 归属清晰
3. 层级明确
4. 视觉舒服

如果“更好看”和“更好理解”冲突，优先选择更好理解。

### 2. 一张图的颜色要少

推荐：

- 主语义色控制在 `2` 到 `4` 类
- 默认优先采用“`3` 个语义色 + `1` 个中性色”

不要这样做：

- 每个节点一个颜色
- 同级节点全部换不同色
- 为了显得丰富硬加颜色

### 3. 同一种颜色尽量只表达一种语义

例如：

- 蓝色就稳定表示“用户侧 / 应用侧 / 发起方”
- 绿色就稳定表示“系统侧 / 承载侧 / 执行侧”
- 橙色就稳定表示“事件 / 提醒 / 风险 / 关键变化”

不要出现：

- 蓝色一会儿表示用户，一会儿表示缓存
- 绿色一会儿表示成功，一会儿表示内核
- 橙色一会儿表示切换，一会儿表示普通模块

### 4. 同一种语义不要换多套颜色

如果已经定义：

- 用户侧 = 蓝色

那同一张图里就不要再出现：

- 一部分用户侧是蓝色
- 一部分用户侧是紫色
- 另一部分用户侧是青色

### 5. 颜色语义不能破坏空间归属

这是技术图里特别重要的一条。

即使某种颜色的“语义解释”说得过去，只要它跨分区之后会让读者误以为：

- 这是同一层
- 这是同一类节点
- 这是同一个模块体系

那这套颜色使用就是不合格的。

这时应优先：

- 降级成灰白中性色
- 或直接并入所在分区主色

## 三、推荐颜色模型

本项目默认采用下面这套颜色模型：

### 1. 蓝色系

**语义**

- 用户侧
- 应用侧
- 发起方
- 客户端模块

**推荐场景**

- 用户程序
- 客户端请求
- 应用服务入口
- 用户态区域

**推荐配色**

- 填充色：`#DCEBFF`
- 边框色：`#1F2937`
- 文字色：`#111827`

### 2. 绿色系

**语义**

- 系统侧
- 执行侧
- 承载侧
- 内核侧

**推荐场景**

- 内核执行模块
- 服务承载模块
- 核心处理模块
- 系统资源处理区

**推荐配色**

- 填充色：`#DDF5E7`
- 边框色：`#1F2937`
- 文字色：`#111827`

### 3. 橙色系

**语义**

- 关键事件
- 触发点
- 提醒
- 风险
- 注意事项

**推荐场景**

- 触发事件
- 异常入口
- 面试易错点
- 风险提示块

**推荐配色**

- 填充色：`#FFE9B8`
- 边框色：`#1F2937`
- 文字色：`#111827`

### 4. 灰白系

**语义**

- 中性信息
- 过渡节点
- 附属能力
- 总结块
- 非重点说明

**推荐场景**

- 过渡步骤
- 子能力项
- 总结语句
- 普通说明块

**推荐配色**

- 主填充色：`#F8FAFC`
- 边框色：`#D1D5DB`
- 文字色：`#374151` 或 `#111827`

## 四、颜色使用优先级

画图时，颜色建议按下面顺序分配：

1. 先给分区定色
2. 再给主流程节点定色
3. 再给核心执行模块定色
4. 再决定是否需要橙色提示
5. 最后把剩余中性内容统一收进灰白系

原则：

- 先分大类，再分重点
- 不要从“哪个好看”开始想

## 五、不同组件的颜色建议

### 1. 主流程节点

- 优先继承所属分区主色
- 不建议单独再发明颜色

### 2. 过渡节点

- 默认灰白系
- 只有在确实需要强调“关键事件”时，才考虑橙色

### 3. 核心执行节点

- 使用所属执行体系主色
- 用户侧用蓝
- 系统侧用绿

### 4. 附属能力节点

- 默认灰白系
- 不抢主色

### 5. 分区标签

- 使用本分区主语义色
- 保持稳定，不随意变化

### 6. 总结块

- 默认灰白系

### 7. 注释块

- 默认灰白系
- 如果必须提示，可少量使用橙色小标签

### 8. 风险块

- 浅橙或浅红
- 使用要克制

## 六、颜色数量控制规则

### 1. 推荐数量

一张图里建议：

- `1` 个主分区色 + `1` 个系统色 + `1` 个强调色 + `1` 个中性色

或者：

- `2` 个主色 + `1` 个中性色

### 2. 警戒线

如果一张图里出现：

- `5` 类以上明显不同的主色

通常就要警惕是否已经过载。

### 3. 可以不用满

不是每张图都必须有蓝、绿、橙、灰四种颜色。

例如：

- 对比图可能只需要两类颜色
- 分层图可能只需要一类主色加灰白

## 七、何时应该降级为中性色

下面这些情况，优先改成灰白系：

### 1. 颜色会导致归属误解

例如：

- 用户区一个橙色节点
- 内核区也一个橙色节点

如果读者会误以为这两个是同类模块，就应该降级。

### 2. 节点只是过渡，不是主归属

例如：

- 切换步骤
- 桥接步骤
- 中间态节点

这类节点通常更适合用灰白系。

### 3. 节点只是补充，不是重点

例如：

- 普通说明
- 附属能力
- 辅助信息

### 4. 图已经够丰富了

如果已经有分区主色和核心主色，就不要再给无关节点上新颜色。

## 八、颜色与位置的关系

技术图里，颜色不能脱离位置单独理解。

必须同时看：

- 这个节点是什么颜色
- 它处在哪个分区
- 它和谁在一组
- 它是不是在主流程上

原则：

- 颜色语义要和空间分布一起成立
- 如果位置会破坏颜色语义，就先调整颜色

## 九、不同图类型的颜色建议

### 1. 主流程图

推荐：

- 主流程按分区主色走
- 过渡节点灰白
- 触发事件可少量橙色
- 总结块灰白

### 2. 分层结构图

推荐：

- 以层级为主，颜色更克制
- 同一层尽量同色或同体系浅深变化

### 3. 对比图

推荐：

- 对比双方保持对等配色
- 不要一边很重，一边很轻

### 4. 状态流转图

推荐：

- 状态节点尽量统一体系
- 只有异常状态或终止状态才考虑单独强调

## 十、常见错误示例

### 1. 把橙色当万能强调色

问题：

- 一会儿表示风险
- 一会儿表示切换
- 一会儿表示重要模块

结果：

- 读者不知道橙色到底在说什么

### 2. 按好看配色，不按语义配色

问题：

- 看起来彩色丰富
- 实际没有信息增量

### 3. 同级节点不同色

问题：

- 会让人误以为它们语义不同

### 4. 不同级节点同色

问题：

- 会让人误以为它们属于同一层

### 5. 色彩过淡或过灰

问题：

- 图会显得发旧、没精神

解决：

- 用浅色填充，但不要脏灰
- 用深灰黑边保持精神度

## 十一、推荐检查清单

出图前检查：

- 这张图一共有几类主色
- 每种颜色是不是只表达一种语义
- 同一种语义是不是只用一套颜色
- 有没有颜色跨分区后造成误解
- 有没有节点其实应该降级成灰白
- 有没有不必要的强调色

## 十二、推荐搭配关系

推荐搭配：

- 浅色填充 + 深灰黑边
- 主语义色 + 灰白中性色
- 少量强调色 + 大量中性色

不推荐搭配：

- 高饱和边框 + 高饱和底色
- 蓝边、绿边、橙边同时大量出现
- 深色大底块铺满整张图
