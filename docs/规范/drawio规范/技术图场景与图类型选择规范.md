# 技术图场景与图类型选择规范

> 本文件用于解决一个更上游的问题：
>
> 不是“这张图怎么画”，而是“这段内容到底应该画成什么图”。

## 一、目的

很多技术图不好看、不好懂，根本原因不是画得差，而是**图选错了**。

最常见的问题就是：

- 明明是结构关系，却硬画成流程图
- 明明是状态变化，却硬画成模块图
- 明明是时序交互，却画成普通箭头流程
- 明明是数据关系，却不用 ER 图

所以这份规范的核心目标是：

- 先判断内容类型
- 再选择合适图种
- 避免“什么都画成流程图”

## 二、总原则

### 1. 先问内容在表达什么

画图前先判断，你要表达的重点到底是哪一种：

- 结构
- 流程
- 时序
- 状态
- 分工
- 数据关系
- 时间计划
- 对比分析

先判断内容本质，再选图。

### 2. 一张图优先只承担一个主表达目标

不要把下面这些混在一张图里：

- 结构图 + 流程图
- 状态图 + 时序图
- ER 图 + 架构图

如果必须都讲，优先拆成两张图。

### 3. 图类型要服从认知成本

原则：

- 用户一眼能理解的，优先用简单图
- 只有内容真的需要时，才上更专业的图种

例如：

- 简单模块关系，不一定非要上 UML 类图
- 简单时间顺序，不一定非要上严格时序图

## 三、先做内容分类

开始画图前，先把内容归到下面某一类：

### 1. 结构关系类

你要表达的是：

- 有哪些模块
- 模块怎么分层
- 模块之间如何归属

适合图种：

- 架构图
- 分层结构图
- 组织结构图
- 类图

### 2. 执行流程类

你要表达的是：

- 事情按什么步骤发生
- 先做什么，后做什么
- 某个请求怎么流转

适合图种：

- 流程图
- 泳道图
- 时序图

### 3. 状态变化类

你要表达的是：

- 对象处于哪些状态
- 状态之间怎么切换

适合图种：

- 状态图

### 4. 数据关系类

你要表达的是：

- 表和表的关系
- 实体和实体的关系
- 字段归属和约束

适合图种：

- ER 图
- 类图

### 5. 时间与计划类

你要表达的是：

- 时间顺序
- 里程碑
- 任务排期

适合图种：

- 时间轴
- 甘特图

### 6. 分析与决策类

你要表达的是：

- 优势与劣势
- 外部环境分析
- 决策维度拆解

适合图种：

- SWOT 分析
- PEST 分析

## 四、各图类型的使用场景

下面按图种说明“什么时候该用”。

### 1. 架构图

**适合表达**

- 系统整体结构
- 模块分层
- 服务之间的关系
- 上下游依赖

**常见场景**

- 微服务架构
- Spring Cloud 组件关系
- 缓存、数据库、MQ 的整体关系
- JVM 整体结构

**不适合表达**

- 详细执行步骤
- 精确时间顺序

**一句判断**

- 如果你想回答“系统由哪些部分组成”，优先考虑架构图。

**本项目补充规则**

- 系统设计题里的“整体图”，默认优先先画架构图，而不是流程图。
- 默认优先采用“容器级系统架构图”思路，也就是先画：
  - 外部角色
  - 接入层
  - 核心业务层
  - 基础设施层
  - 治理能力
- 如果一张图更像在回答“步骤怎么走”，那它更接近流程图，不是系统架构图。
- 系统架构图的专项规范，继续看：
  - [技术图系统架构图规范.md](/Users/wangpenglong/Obsidian/JavaInterview/项目规范/技术图系统架构图规范.md)

### 2. 流程图

**适合表达**

- 单条业务流程
- 请求处理链路
- 固定步骤流转

**常见场景**

- 用户态与内核态切换
- 一次 HTTP 请求处理流程
- Bean 初始化流程
- SQL 执行流程

**不适合表达**

- 多角色并行协作
- 严格消息时序
- 状态切换全集

**一句判断**

- 如果你想回答“这件事一步一步怎么发生”，优先考虑流程图。

### 3. 时序图

**适合表达**

- 多个对象之间按时间顺序交互
- 谁先调用谁
- 请求和响应如何往返

**常见场景**

- 用户 -> 网关 -> 服务 -> 缓存 -> 数据库
- RPC 调用链
- 分布式事务参与者交互

**不适合表达**

- 模块静态结构
- 数据表关系

**一句判断**

- 如果你想回答“谁在什么时间点调用了谁”，优先考虑时序图。

### 4. 泳道图

**适合表达**

- 多角色、多模块协同完成一个流程
- 每一步由谁负责

**常见场景**

- 用户、网关、服务、数据库协作流程
- 开发、测试、发布流程
- 多服务参与的审批流

**不适合表达**

- 纯结构关系
- 复杂对象消息交互细节

**一句判断**

- 如果你想回答“流程没错，但还要看每步是谁做的”，优先考虑泳道图。

### 5. ER 图

**适合表达**

- 表与表关系
- 实体关系
- 主键、外键、关联关系

**常见场景**

- 用户表、订单表、商品表关系
- 权限模型
- 博客系统数据库设计

**不适合表达**

- 服务调用
- 执行流程

**一句判断**

- 如果你想回答“这些数据对象之间是什么关系”，优先考虑 ER 图。

### 6. 组织结构图

**适合表达**

- 归属关系
- 上下级结构
- 树状模块结构

**常见场景**

- 包结构
- 模块树
- 权限树
- Spring 容器中的层级归属

**不适合表达**

- 流程顺序
- 状态流转

**一句判断**

- 如果你想回答“谁属于谁、谁挂在谁下面”，优先考虑组织结构图。

### 7. 时间轴

**适合表达**

- 事件在时间上的先后
- 版本演进
- 故障发生与排查过程

**常见场景**

- JVM 发展版本演进
- 故障时间线
- 系统启动阶段时间线

**不适合表达**

- 模块关系
- 执行条件分支

**一句判断**

- 如果你想回答“这些事件按时间怎么排”，优先考虑时间轴。

### 8. 类图

**适合表达**

- 类之间的继承、实现、依赖、关联
- 面向对象结构

**常见场景**

- 设计模式讲解
- Spring 核心接口与实现类关系
- 集合框架继承结构

**不适合表达**

- 业务流程
- 数据表设计细节

**一句判断**

- 如果你想回答“这些类之间是什么面向对象关系”，优先考虑类图。

### 9. 甘特图

**适合表达**

- 任务排期
- 里程碑计划
- 项目推进安排

**常见场景**

- 学习计划安排
- 项目交付计划

**不适合表达**

- 技术原理
- 模块结构

**一句判断**

- 如果你想回答“这件事在什么时间做完”，优先考虑甘特图。

### 10. 状态图

**适合表达**

- 状态集合
- 状态切换条件

**常见场景**

- Java 线程状态
- TCP 状态机
- 订单状态流转

**不适合表达**

- 服务模块结构
- 普通线性流程

**一句判断**

- 如果你想回答“对象会经历哪些状态以及如何切换”，优先考虑状态图。

### 11. SWOT 分析

**适合表达**

- 优势、劣势、机会、威胁

**常见场景**

- 技术方案对比
- 项目决策分析

**不适合表达**

- 技术原理知识点

**一句判断**

- 如果你想回答“这个方案值不值得选”，才考虑 SWOT。

### 12. PEST 分析

**适合表达**

- 政策、经济、社会、技术环境分析

**常见场景**

- 商业分析
- 战略分析

**不适合表达**

- Java 面试知识主体内容

**一句判断**

- 对本项目主知识库来说，PEST 基本不是常用图种。

## 六、本项目常见知识点推荐图种

### 1. 操作系统

推荐：

- 流程图：用户态与内核态切换
- 状态图：进程状态、线程状态
- 结构图：进程与线程关系

### 2. 计算机网络

推荐：

- 流程图：TCP 三次握手、四次挥手
- 时序图：客户端与服务端交互
- 分层结构图：TCP/IP 分层

### 3. Java 基础

推荐：

- 类图：集合框架、继承关系
- 对比图：重载 vs 重写、抽象类 vs 接口

### 4. Java 并发

推荐：

- 状态图：线程状态流转
- 流程图：CAS、自旋、锁升级
- 时序图：线程协作时序

### 5. JVM

推荐：

- 架构图：JVM 整体结构
- 分层图：类加载、运行时数据区
- 流程图：类加载流程、GC 流程

### 6. Spring / Spring Boot

推荐：

- 流程图：Bean 生命周期
- 架构图：Spring 容器结构
- 时序图：请求进入 Spring MVC 的处理链路
- 类图：核心接口体系

### 7. 数据库 / MySQL

推荐：

- ER 图：表关系
- 流程图：SQL 执行流程
- 结构图：索引结构
- 时序图：事务执行与锁竞争

### 8. Redis / 缓存

推荐：

- 架构图：缓存体系
- 流程图：缓存读写策略
- 时序图：缓存击穿、缓存一致性交互

### 9. 系统设计与实战

推荐：

- 架构图：整体系统设计
- 时序图：核心链路交互
- 泳道图：多角色协作流程
- 时间轴：故障时间线

### 2. 你想讲“过程”

### 3. 你想讲“变化”

用：

- 状态图

### 4. 你想讲“关系”

### 5. 你想讲“计划”

用：

- 甘特图
- 时间轴

## 八、何时不该用流程图

下面这些情况，不建议默认画流程图：

### 1. 内容本质是静态结构

例如：

- JVM 结构
- Spring 容器层次
- 数据结构分类

更适合：

- 架构图
- 分层结构图
- 类图

### 2. 内容本质是状态切换

例如：

- 线程状态
- TCP 状态机

更适合：

- 状态图

### 3. 内容本质是对象关系

例如：

- 数据表关系
- 类关系

更适合：

- ER 图
- 类图

### 4. 内容本质是多对象按时间交互

例如：

- 用户请求经过多个系统

更适合：

- 时序图
- 泳道图

## 九、推荐工作流

后续开始画图时，建议先做这 4 步：

1. 先问：我要表达的是组成、过程、变化、关系，还是计划
2. 再问：这是单对象还是多对象
3. 再问：重点是结构、顺序、状态，还是归属
4. 最后再选图种
