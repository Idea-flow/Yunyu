# 技术图组件规范

> 本文件用于统一本项目技术图的组件设计方式。
>
> 项目级总规则以根目录下的 `AGENTS.md` 为准；本文件更偏“组件字典”和“落地使用说明”。

## 一、目的

这份规范主要解决三个问题：

- 技术图里到底应该用哪些组件
- 每种组件分别适合表达什么
- 如何避免一张图里样式混乱、颜色混乱、语义混乱

目标不是把图画复杂，而是让图：

- 一眼能读懂
- 组件有固定职责
- 颜色有稳定语义
- 后续整套知识库风格统一

## 二、使用原则

每次画图前，先按下面顺序判断：

1. 先判断这张图是什么类型
2. 再确定主流程或主结构
3. 再选择需要的组件
4. 最后才决定颜色、位置和连线

原则：

- 一张图优先只表达一个主重点
- 组件不要滥用
- 颜色不要滥用
- 提示信息不要抢主流程
- 样式必须为理解服务，而不是为装饰服务

## 三、组件总表

本项目技术图推荐使用以下组件：

1. 主流程节点
2. 过渡节点
3. 判断节点
4. 核心执行节点
5. 附属能力节点
6. 分区标签
7. 轻分隔线
8. 总结块
9. 对比块
10. 注释块
11. 风险块
12. 容器分组
13. 箭头组件

## 四、组件明细

### 1. 主流程节点

**作用**

- 表达主流程中的关键步骤
- 读者应该优先看这一类节点

**适用场景**

- 请求处理流程
- 状态切换流程
- 系统执行链路
- 生命周期流程

**默认样式**

- 形状：圆角矩形
- 圆角：`12px` 到 `14px`
- 边框：深灰黑边
- 线宽：`1.4px` 到 `1.6px`
- 文字：加粗

**颜色建议**

- 属于用户侧：浅蓝底
- 属于系统侧：浅绿底
- 如果只是流程中的普通步骤，也可以使用同分区主色

**禁忌**

- 不要把补充说明也做成主流程节点
- 不要在主流程节点里塞太多文字
- 不要让主流程节点颜色随意变化

### 2. 过渡节点

**作用**

- 表达“切换”“桥接”“进入下一阶段”这类过渡动作
- 它是流程中的中转站，不是最终归属模块

**适用场景**

- 用户态切换到内核态
- 请求进入网关
- 任务被调度到执行器
- 数据从一层流入另一层

**默认样式**

- 形状：圆角矩形
- 风格：比主流程节点更克制
- 边框：浅灰边或深灰细边
- 文字：正常或轻加粗

**颜色建议**

- 默认优先使用灰白中性色
- 只有在不会产生归属误解时，才允许使用语义强调色

**禁忌**

- 不要把过渡节点画得比主流程节点还抢眼
- 不要跨分区重复使用同一种强调色，避免读者误以为它们属于同一层

### 3. 判断节点

**作用**

- 表达是否成立、是否通过、是否满足条件这类分支判断
- 明确告诉读者流程会在这里分叉

**适用场景**

- 是否通过校验
- 是否存在库存
- 是否支付成功
- 是否命中缓存
- 是否需要降级

**必须使用菱形的场景**

下面这些情况，默认必须优先使用菱形判断节点，而不是普通矩形：

- 文案里出现“是否”
- 存在“成功 / 失败”分叉
- 存在“是 / 否”分叉
- 一个节点后面要分成两条以上互斥路径

一句判断：

- 如果这个节点本质上是在“做判断”，而不是“做处理”，就应该优先用菱形。

**默认样式**

- 形状：菱形
- 颜色：浅橙底
- 边框：深灰黑边
- 文字：短句，通常控制在两行内

**配套规则**

- 判断节点后至少应有一条 `是` 分支
- 如果存在失败路径，也应有明确 `否` 分支
- 分支线应从菱形边缘自然引出，不要像从空气中突然出现
- `是 / 否` 标签要贴近对应分支，而不是漂在远处

**禁忌**

- 不要把“普通处理动作”画成菱形
- 不要一个判断节点后分出太多条复杂路径
- 不要省略失败分支，导致读者不知道“不满足时去了哪里”

### 4. 核心执行节点

**作用**

- 表达真正承担执行职责的核心模块
- 通常是图中最重要的“处理者”

**适用场景**

- 操作系统内核执行服务
- JVM 执行引擎
- Spring 容器
- 数据库执行器
- 消息队列 Broker

**默认样式**

- 形状：较大的圆角矩形
- 字号：比普通节点稍大
- 边框：深灰黑边
- 位置：一般位于流程中心或分区核心位置

**颜色建议**

- 用户侧核心模块：蓝色系
- 系统侧核心模块：绿色系

**禁忌**

- 不要和普通附属能力节点画得一样轻
- 不要在同一张图里出现多个视觉上完全等重的“核心中心”

### 5. 附属能力节点

**作用**

- 表达围绕核心节点展开的能力项或子能力
- 它们是补充理解，不是主流程重点

**适用场景**

- 文件 IO
- 网络 IO
- 设备访问
- 进程调度
- 缓存
- 日志
- 监控

**默认样式**

- 形状：小号圆角矩形
- 风格：白底或浅灰底
- 边框：浅灰边
- 文字：正常字重

**颜色建议**

- 默认灰白中性色
- 不抢主流程颜色

**禁忌**

- 不要把附属能力节点做得比主流程节点更亮
- 不要给每个附属能力单独上不同颜色

### 6. 分区标签

**作用**

- 表达一大片区域的身份
- 帮助读者理解“这里是哪一层、哪一类”

**适用场景**

- 用户态 / 内核态
- 应用层 / 网络层 / 传输层
- 上游 / 下游
- 生产者 / Broker / 消费者

**默认样式**

- 形状：小尺寸圆角标签
- 位置：区域左侧、上方或明显边界处
- 视觉：简洁、稳定、轻量

**颜色建议**

- 用户侧分区标签可用蓝色系
- 系统侧分区标签可用绿色系
- 标签颜色应与分区主语义一致

**禁忌**

- 不要把分区标签画得像主流程节点
- 不要为了标签制造大块厚重背景

### 7. 轻分隔线

**作用**

- 表达分区边界或层级边界
- 提供阅读辅助

**适用场景**

- 上下层划分
- 左右区域划分
- 模块边界提示

**默认样式**

- 颜色：浅灰
- 线宽：`1px`
- 风格：轻、弱、克制

**禁忌**

- 不要把分隔线画得像主流程线
- 不要用粗黑线切割整张图

### 8. 总结块

**作用**

- 用一句话收束这张图的核心结论
- 帮助读者在看完图后形成记忆点

**适用场景**

- 概念性知识图
- 面试解释图
- 对比图

**默认样式**

- 形状：扁平长条圆角块
- 风格：灰白中性色
- 位置：图底部优先

**颜色建议**

- 灰白系

**禁忌**

- 不要做成高饱和强调块
- 不要写成一大段文字

### 8. 对比块

**作用**

- 用于表达两个或多个对象的区别

**适用场景**

- 进程 vs 线程
- BIO vs NIO vs AIO
- 乐观锁 vs 悲观锁
- 聚簇索引 vs 二级索引

**默认样式**

- 形状：左右对照块或并列卡片
- 结构：同尺寸、同层级、同对齐

**颜色建议**

- 同组对比优先用同一体系下的深浅变化
- 只有在确实要表达差异时，才轻度换色

**禁忌**

- 不要让对比双方样式不对等
- 不要一边是重视觉，一边是轻视觉

### 9. 注释块

**作用**

- 补充说明某个节点或某段逻辑
- 帮助理解，但不参与主流程执行

**适用场景**

- 注意事项
- 面试易错点
- 补充解释
- 概念说明

**默认样式**

- 推荐采用“标签 + 说明卡片”两段式
- 标签：小号提示标签
- 卡片：白底或浅灰底说明卡片
- 连线：细虚线或细引导线

**颜色建议**

- 默认灰白系
- 如果必须提示，可少量使用橙色标签

**禁忌**

- 不要压主流程线
- 不要卡在箭头通道中间
- 不要让注释块比主流程还醒目
- 如果内容可删而不影响理解，优先删掉而不是硬加

### 10. 风险块

**作用**

- 表达风险、异常、易错点、故障点

**适用场景**

- 缓存穿透
- 死锁风险
- 空指针风险
- 服务雪崩
- SQL 慢查询风险

**默认样式**

- 形状：圆角矩形
- 视觉：比注释块稍强，但仍然不能压过主流程

**颜色建议**

- 浅红底或浅橙底

**禁忌**

- 不要把普通说明也画成风险块
- 不要在同一张图里出现过多风险块

### 11. 容器分组

**作用**

- 表达一组组件属于同一模块或同一子系统

**适用场景**

- JVM 内存区域
- Spring 容器内部结构
- 微服务模块边界
- 数据流处理阶段

**默认样式**

- 容器要轻
- 可以无边框，或使用极浅边框
- 可以用很浅的底色辅助分组

**颜色建议**

- 默认极浅灰或极浅语义色

**禁忌**

- 不要使用厚重的大背景框
- 不要让容器抢走节点本身的注意力

### 12. 箭头组件

**作用**

- 表达方向、执行、依赖、补充说明

**类型规范**

- **实体细箭头**：主流程、真实执行路径、明确调用关系
- **虚线箭头**：补充说明、弱依赖、附属能力连接
- **粗箭头 / 大方向箭头**：阶段强调、总体方向；默认少用

**通用要求**

- 箭头和框必须分开
- 箭头优先走空白通道
- 箭头标签不要压在线上
- 转折次数尽量少

**禁忌**

- 不要让箭头穿过节点
- 不要在同一张图里混用太多箭头风格

## 五、颜色语义约定

推荐使用“`3` 个语义色 + `1` 个中性色”模型：

- **蓝色系**：用户侧、应用侧、发起方
- **绿色系**：系统侧、执行侧、承载侧
- **橙色系**：关键事件、提醒、风险、注意
- **灰白系**：中性说明、过渡节点、附属能力、总结

补充规则：

- 一张图主色尽量控制在 `2` 到 `4` 类
- 同一种颜色尽量只表达一种语义
- 如果同色跨分区会造成归属误读，优先改用灰白中性色
- 不是所有图都必须用满所有颜色

## 六、常见组合模板

### 1. 分区流程图

推荐组件组合：

- 分区标签
- 轻分隔线
- 主流程节点
- 过渡节点
- 核心执行节点
- 附属能力节点
- 总结块

### 2. 分层结构图

推荐组件组合：

- 分区标签
- 容器分组
- 核心执行节点
- 附属能力节点
- 轻分隔线

### 3. 对比图

推荐组件组合：

- 对比块
- 总结块
- 少量注释块

### 4. 状态流转图

推荐组件组合：

- 主流程节点
- 过渡节点
- 实体箭头
- 风险块或注释块

## 七、选择组件时的快速判断

如果你在画图时拿不准该用什么组件，可以直接按下面判断：

- 这是主执行步骤吗：用主流程节点
- 这是阶段切换吗：用过渡节点
- 这是承担主要处理职责的中心模块吗：用核心执行节点
- 这是外围能力吗：用附属能力节点
- 这是区域身份说明吗：用分区标签
- 这是一句总结吗：用总结块
- 这是补充解释吗：用注释块
- 这是危险点或易错点吗：用风险块

## 八、组件使用禁忌

- 不要把所有内容都画成同一种节点
- 不要为了好看增加无意义组件
- 不要让颜色脱离语义单独存在
- 不要让提示信息进入主流程通道
- 不要让容器比节点更显眼
- 不要让一张图有多个视觉中心

## 九、推荐落地方式

后续本项目里的技术图，建议统一按下面方式执行：

1. 先判断图类型
2. 先列出需要的组件
3. 先定义颜色语义
4. 再开始摆放节点
5. 最后处理箭头、留白和总结块
