# 技术图系统架构图规范

> 本文件专门解决一个常见问题：
>
> 很多“系统设计”内容，本来应该画成系统架构图，结果被误画成了流程图。

## 一、目的

系统架构图的目标，不是回答：

- 第一步做什么
- 第二步做什么
- 请求一共走了几跳

而是回答：

- 系统由哪些模块组成
- 每个模块负责什么
- 模块之间怎么协作
- 哪些是业务服务，哪些是基础设施，哪些是治理能力

所以系统架构图更偏**结构表达**，不是**步骤表达**。

## 二、本项目默认采用什么思路

本项目后续画“系统设计题”的整体图时，默认采用：

- **主标准：C4 模型中的 Container Diagram 思路**
- **落地方式：中文化、面试化、简洁圆润技术图风**

这里的意思不是机械照搬英文模板，而是借鉴它最核心的思想：

- 先明确系统边界
- 再明确有哪些主要容器 / 模块
- 再明确它们之间的关系
- 每个盒子不仅写名字，还要写职责

对于本项目这种“Java 面试知识库”，最适合的通常不是超抽象 Context 图，也不是太细的组件图，而是：

- **容器级系统架构图**

也就是：

- 用户入口
- 接入层
- 核心业务层
- 基础设施层
- 数据层
- 治理能力

这几个层次之间的关系图。

## 三、系统架构图必须回答什么

一张合格的系统架构图，至少应该让读者一眼看出下面 5 件事：

1. 系统边界在哪里
2. 有哪些核心模块
3. 每个模块的职责是什么
4. 模块之间如何通信
5. 最终数据落在哪

如果一张图画完以后，读者只能看出“流程往哪走”，却看不出“系统由什么组成”，那它更像流程图，不像架构图。

## 四、系统架构图必须包含的元素

### 1. 外部角色

例如：

- 用户
- 浏览器
- App
- 第三方系统

作用：

- 告诉读者系统服务的是谁

### 2. 接入模块

例如：

- CDN
- API 网关
- 负载均衡
- 鉴权层

作用：

- 表达请求先经过哪些入口能力

### 3. 核心业务模块

例如：

- 秒杀接入服务
- 资格校验服务
- 库存服务
- 订单服务

作用：

- 表达系统真正的业务骨架

### 4. 基础设施模块

例如：

- Redis
- MQ
- MySQL
- ES

作用：

- 表达业务服务依赖哪些基础能力

### 5. 治理能力

例如：

- 监控告警
- 熔断降级
- 幂等补偿
- 限流防刷

作用：

- 表达系统不是只有功能模块，还有稳定性与工程化能力

## 五、系统架构图里不该塞什么

下面这些内容，不应该硬塞进系统架构图：

- 详细 1、2、3、4 步骤
- 一次请求的精确时间顺序
- 每个异常分支怎么回滚
- 太细的 SQL、Lua、事务细节
- 太多面试追问答案

这些内容应该拆到：

- 流程图
- 时序图
- 异常处理图
- 文字说明

一句话：

- **架构图讲“谁和谁”，流程图讲“先和后”。**

### 补充规则：正式图中不要出现“给作者自己看的说明”

系统架构图是给读者直接看的正式图，不是画图过程说明板。

所以图中默认不要出现这类内容：

- `表达方式：系统架构图`
- `为什么这里用这个图`
- `这张图不是流程图`
- `本图强调模块关系`

这些说明适合写在：

- 辅助规则文档
- Markdown 正文解释

不适合直接写进正式架构图里。

## 六、本项目系统架构图推荐版式

后端系统设计题，默认优先采用下面的版式：

### 1. 左侧：外部角色

- 用户
- App
- 浏览器

### 2. 中左：接入层

- CDN
- 网关
- 鉴权

### 3. 中部：核心业务层

- 核心服务模块横向展开
- 每个模块都写清职责

### 4. 中下：基础设施层

- Redis
- MQ
- MySQL

### 5. 右侧：治理能力

- 监控
- 熔断
- 补偿

这样做的优点是：

- 中间是核心业务
- 左边是入口
- 下方是依赖
- 右边是兜底与治理

读图路径自然，也更像系统架构，而不是单条流程。

## 七、节点文案规则

系统架构图里的每个核心节点，建议写成两行：

- 第一行：模块名
- 第二行：职责说明

例如：

- `秒杀接入服务`
- `请求收口 / 活动路由`

不要只写：

- `服务 A`
- `服务 B`
- `模块 C`

否则图会很空，也不够面试化。

## 八、节点尺寸规则

系统架构图最容易出问题的地方，就是框太小、字太长。

本项目默认建议：

- 核心模块宽度：`180px` 到 `220px`
- 核心模块高度：`84px` 到 `96px`
- 双行文案优先，不强行单行塞满
- 文案超过两层意思时，优先缩文案，不继续缩框

原则：

- 宁可框大一点，也不要让字挤爆

### 节点尺寸处理优先级

如果出现“文字快要超框、已经超框、读起来发堵”的情况，按下面顺序处理：

1. 先拆成两行
2. 再缩短文案
3. 再增大框宽度
4. 必要时再增加框高度
5. 最后才考虑整体重新排版

默认不要优先做这些事：

- 为了塞进去把字体压得很小
- 把行距压得很紧
- 继续往节点里塞第三层、第四层解释

原则：

- 先保证可读性，再保证紧凑

### 节点留白下限

正式图中，节点文字与边框之间必须留出明确内边距。

建议最低保证：

- 左右内边距至少 `16px`
- 上下内边距至少 `12px`
- 两行文字之间要有明显行距

如果读者一眼觉得“字贴边了”，哪怕还没真正溢出，也视为不合格。

### 系统架构图专项宽度建议

对于“模块名 + 职责”两行结构，推荐宽度进一步细化为：

- 接入层节点：`220px` 到 `260px`
- 核心业务节点：`210px` 到 `250px`
- 基础设施节点：`190px` 到 `230px`
- 治理能力节点：`220px` 到 `260px`

原因：

- 系统架构图里的文案通常比流程图更长
- 如果还沿用太窄的流程节点宽度，就很容易出现挤压和溢出

## 九、箭头语义规则

系统架构图里，箭头不要像流程图那样“每一步都写一条路”。

默认语义：

- **实线箭头**：同步调用、查询、直接依赖
- **虚线箭头**：异步消息、事件投递、弱依赖

补充规则：

- 不重要的边不要硬连
- 非关键关系不必全部画出
- 标签只标注非显然关系

例如：

- `缓存 / 预扣减`
- `异步投递`
- `最终落库`

这类标签是有价值的。

而：

- `请求`
- `调用`
- `执行`

如果没有额外信息价值，就可以不写。

## 十、什么时候应该拆成第二张图

如果内容同时需要回答下面两个问题，就不要只用一张架构图：

1. 系统由哪些模块组成
2. 一次请求详细怎么流转

正确做法是拆成两张：

- 第一张：系统架构图
- 第二张：核心流程图 / 时序图

例如秒杀系统很适合这样拆：

- 架构图：看整体模块关系
- 核心链路流程图：看一次秒杀请求怎么走

## 十一、本项目落地结论

从现在开始，系统设计类主题默认这样处理：

1. 先判断是不是系统架构问题
2. 如果是，优先先画系统架构图
3. 架构图默认采用“容器级表达”
4. 节点写“模块名 + 职责”
5. 同步和异步关系分开表达
6. 需要详细执行过程时，再补流程图或时序图

## 十二、可参考的标准来源

下面这些来源的思路，可以作为后续继续统一风格时的参考：

- C4 Model 的 Container Diagram：强调系统中的主要容器、职责和关系
- Microsoft Azure Architecture Center 的架构图实践：强调模块边界、依赖关系、可读性和一致性

参考链接：

- [C4 Model - Container diagrams](https://c4model.com/diagrams/container)
- [Azure Architecture Center](https://learn.microsoft.com/azure/architecture/)
