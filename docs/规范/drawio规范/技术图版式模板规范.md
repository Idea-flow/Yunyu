# 技术图版式模板规范

> 本文件用于统一技术图的版式结构。
>
> 如果说《技术图组件规范》解决的是“用什么组件”，那么本文件解决的是“这些组件应该怎么摆”。

## 一、目的

技术图难看、难懂，很多时候不是组件有问题，而是版式有问题。

常见问题：

- 主流程不明显
- 节点虽然好看，但摆放混乱
- 箭头和框抢通道
- 提示块压在主流程上
- 分区有了，但没有层次
- 一张图里同时出现多个阅读方向

所以本规范的目标是：

- 固定读图方向
- 固定主流程通道
- 固定注释区位置
- 固定总结区位置
- 固定不同图类型的基本排版模板

## 二、总原则

### 1. 一张图先确定唯一主阅读方向

推荐优先级：

1. 从左到右
2. 从上到下
3. 左右对比

原则：

- 一张图里不要同时让读者既从左到右读，又从上到下读
- 如果主流程已经是从左到右，补充信息就不要再形成第二条主流程

### 2. 先给主流程留通道

主流程区域必须先留出来，再摆其他元素。

推荐做法：

- 先摆主流程节点
- 再留箭头走线空白
- 最后再放注释块、总结块、分区标签

### 3. 版式优先级固定

一张图中，视觉优先级应当是：

1. 主流程
2. 核心执行模块
3. 分区结构
4. 附属能力
5. 注释与补充说明
6. 总结块

### 4. 空白不是浪费

技术图必须有呼吸感。

原则：

- 节点之间必须留箭头通道
- 主流程周围要有空白缓冲区
- 不要为了“利用空间”把提示块塞进主流程缝隙里

## 三、标准区域划分

推荐把一张常规技术图分成以下区域：

### 1. 标题区

**位置**

- 顶部居中

**规则**

- 标题优先纯文字
- 不建议加大底板、胶囊背景、椭圆背景

### 2. 主图区

**位置**

- 整张图中间区域

**规则**

- 主流程节点或主结构必须放在这里
- 这是整张图最重要的区域

### 3. 分区标识区

**位置**

- 左侧或顶部

**规则**

- 用于标记“用户态 / 内核态”“应用层 / 网络层”等区域身份
- 分区标签要轻，不要抢主流程

### 4. 附属能力区

**位置**

- 主模块下方、右下方或外围

**规则**

- 用于放次级能力项
- 不参与主流程主通道

### 5. 注释区

**位置**

- 右侧优先
- 下侧次优先

**规则**

- 用于放说明卡片、提醒、补充解释
- 默认不得进入主流程通道

### 6. 总结区

**位置**

- 底部居中优先

**规则**

- 一句结论即可
- 用于收束，不用于展开细节

## 四、主流程图模板

### 1. 适用场景

- 请求处理流程
- 用户态 / 内核态切换
- 一次 SQL 执行流程
- Bean 生命周期
- 消息发送消费流程

### 2. 推荐版式

- 标题在上
- 主流程横向展开
- 分区标签放左侧
- 核心执行节点放主流程下方或中下部
- 附属能力节点放底部一排
- 总结块放最底部

### 3. 推荐结构

```text
标题

分区标签     主流程节点 -> 主流程节点 -> 过渡节点 -> 主流程节点

分区标签              ↓
                 核心执行节点
              ↙   ↓   ↓   ↘
           附属能力 附属能力 附属能力 附属能力

                  总结块
```

### 4. 布局要求

- 主流程节点优先保持同一水平线
- 向下的分支只允许少量存在
- 核心执行节点与主流程之间要有明显垂直通道
- 附属能力节点之间要横向对齐

## 五、分层结构图模板

### 1. 适用场景

- JVM 整体结构
- TCP/IP 分层
- Spring 容器结构
- 微服务系统结构

### 2. 推荐版式

- 自上而下分层
- 每层内部再横向排布
- 每层之间用轻分隔线或轻容器区分

### 3. 推荐结构

```text
标题

第一层标签
  核心节点  核心节点  核心节点

第二层标签
  核心节点  附属节点  附属节点

第三层标签
  承载节点  承载节点  承载节点

总结块
```

### 4. 布局要求

- 分层图优先上下阅读
- 每层内部保持对齐
- 层与层之间不要出现混乱回线
- 分层图不要再强行混入复杂流程箭头

## 六、模块交互图模板

### 1. 适用场景

- 微服务调用关系
- 系统间交互
- 模块协作关系
- 组件通信关系

### 2. 推荐版式

- 先摆核心模块
- 周边围绕放上下游模块
- 交互线尽量走外围

### 3. 推荐结构

```text
标题

上游模块   上游模块
     \      /
      核心模块
     /   |   \
下游模块 模块 模块

总结块
```

### 4. 布局要求

- 主模块尽量放中心
- 周边模块不要挤在一侧
- 如果交互很多，优先模块分区，而不是硬连线

## 七、对比图模板

### 1. 适用场景

- 进程 vs 线程
- 乐观锁 vs 悲观锁
- 聚簇索引 vs 二级索引
- BIO vs NIO vs AIO

### 2. 推荐版式

- 左右并列对比
- 三对象对比时采用三栏结构
- 各栏样式必须对齐

### 3. 推荐结构

```text
标题

对象 A      对象 B
要点 1      要点 1
要点 2      要点 2
要点 3      要点 3

总结块
```

### 4. 布局要求

- 每一列宽度尽量一致
- 同一层级内容必须横向对齐
- 对比图优先少箭头

## 八、状态流转图模板

### 1. 适用场景

- 线程状态流转
- TCP 状态机
- 生命周期状态变化

### 2. 推荐版式

- 优先环形或单向状态流
- 状态节点围绕主通道展开

### 3. 布局要求

- 主状态路径必须明显
- 回环箭头应走外围
- 状态切换条件要少而清楚

## 九、总结块与注释块的位置规范

### 1. 总结块

推荐位置：

- 底部居中

不要这样做：

- 放在主流程中间
- 贴在某个节点旁边像注释一样

### 2. 注释块

推荐位置：

- 右侧空白区
- 下侧空白区

不要这样做：

- 压在箭头上
- 放在两节点中间
- 和主流程节点混成一排

## 十、箭头通道规范

### 1. 主流程箭头

- 走最直观的主通道
- 优先水平直连
- 必要时少量折线

### 2. 向下分支箭头

- 只在确实需要表达“进入下层/进入执行模块”时使用
- 落点要明显

### 3. 回流箭头

- 优先走外围
- 不要从图中央穿回去

### 4. 附属连接箭头

- 默认使用虚线
- 不和主流程抢粗细与颜色

## 十一、间距建议

推荐使用下面的简单节奏：

- 标题与主图区：留较大空白
- 主流程节点之间：留标准箭头通道
- 主流程与下方核心模块：留明显垂直距离
- 核心模块与附属能力：留一层次级距离
- 图底部与总结块：留稳定边距

原则：

- 近表示相关
- 远表示层级差异
- 空白表示结构

## 十二、常见错误版式

### 1. 所有东西都堆在中间

问题：

- 没有层次
- 箭头容易打架

### 2. 主流程和提示块混排

问题：

- 主次不分
- 读者不知道先看哪个

### 3. 左右上下同时都在读

问题：

- 阅读方向混乱

### 4. 分区有标签，但没有空间边界

问题：

- 名字上分区了，视觉上却没分开

### 5. 过度追求对称

问题：

- 有些图逻辑本来不对称，硬对称反而不自然

## 十三、推荐落地流程

后续画图时建议按下面步骤执行：

1. 判断图类型
2. 选择版式模板
3. 先摆主流程或主结构
4. 再摆核心模块
5. 再摆附属能力与分区标签
6. 最后再放注释块、总结块和箭头标签
