# AI 公共系统需求方案（一期）

## 文档目标

本文档用于定义 `Yunyu` AI 公共系统的一期需求，重点围绕以下结论：

1. `Chat` 与 `Responses` 协议都要在本期实现，不再占位。
2. 需提供两套 OpenAI 兼容接口，字段与交互尽量保持一致：
  - `/v1/chat/completions`
  - `/v1/response`（同时建议兼容 OpenAI 标准路径 `/v1/responses`）
3. `posts/ai/meta/generate` 底层固定调用 Chat Completions 服务方法，并通过 `stream` 判断流式/非流式。
4. 全链路参数、交互、错误结构尽量贴合 OpenAI API。

## 一、需求背景

后台文章编辑页（`/admin/posts/create`）在新增/编辑时，`slug`、摘要、`seoTitle`、`seoDescription` 依赖人工填写，存在效率与一致性问题。

为避免后续多业务重复接入 AI，本期需要先建设统一 AI 公共系统，且接口协议尽量 OpenAI 化，方便未来迁移与复用。

## 二、一期目标

1. 建立统一 AI 网关能力，支持多提供商配置，单启用治理。
2. 实现两类协议能力：
  - Chat Completions
  - Responses
3. 对外提供 OpenAI 兼容接口，支持流式与非流式。
4. 落地文章元信息生成能力（`slug`、摘要、SEO），优先服务后台文章编辑流程。
5. 保持模块化、解耦、可扩展，新增协议或上游厂商时不影响业务层。

## 三、范围定义

### 3.1 本期范围（In Scope）

1. AI 提供商配置管理（多配置、单启用、连接测试）。
2. OpenAI 兼容接口：
  - `POST /v1/chat/completions`
  - `POST /v1/response`（兼容 `/v1/responses`）
3. Chat 与 Responses 两协议都支持流式/非流式。
4. 文章元信息生成接口：`POST /api/admin/posts/ai/meta/generate`。
5. 调用治理：超时、重试、错误映射、日志。

### 3.2 暂不纳入（Out of Scope）

1. 评论审核、翻译、改写、配图等其他业务场景。
2. 多提供商并发路由与成本自动优化。
3. Prompt 可视化编排平台。

## 四、核心概念

1. `OpenAI 兼容接口`：请求字段、响应字段、流式事件形态对齐 OpenAI。
2. `上游协议选择`：每个配置项需声明调用上游时使用 `completions` 或 `responses`。
3. `场景接口`：业务语义接口（如 posts 元信息生成），底层复用协议服务。
4. `流式响应`：`stream=true` 时按 SSE 分片返回。

## 五、协议兼容需求

### 5.1 Chat Completions 兼容要求

1. 路径：`POST /v1/chat/completions`。
2. 入参字段：遵循 OpenAI Chat Completions 协议字段。
3. 出参字段：遵循 OpenAI Chat Completions 响应结构。
4. `stream=true` 时：遵循 Chat Completions 流式交互与分片格式。

### 5.2 Responses 兼容要求

1. 路径：`POST /v1/response`，并建议兼容标准路径 `POST /v1/responses`。
2. 入参字段：遵循 OpenAI Responses 协议字段。
3. 出参字段：遵循 OpenAI Responses 响应结构。
4. `stream=true` 时：遵循 Responses 流式事件交互。

### 5.3 OpenAI 化约束

1. 请求字段命名、结构、类型尽量不做二次改写。
2. 响应主结构与错误结构尽量对齐 OpenAI。
3. 仅在业务场景接口（如 posts）中做必要业务包装。

## 六、上游服务配置需求

1. 每个 AI 配置项需包含 `upstreamProtocol`，可选值：
  - `COMPLETIONS`
  - `RESPONSES`
2. 同一时刻仅允许一个配置启用。
3. 内部调用时根据启用配置决定走哪种协议服务。
4. 若请求协议与当前启用配置不匹配，返回明确错误提示。

## 七、文章元信息生成需求

### 7.1 接口定义

1. 接口：`POST /api/admin/posts/ai/meta/generate`。
2. 底层固定调用 Chat Completions 服务层方法。
3. `stream=true` 走流式；`stream=false` 走一次性返回。

### 7.2 入参与交互要求

1. 该接口入参采用业务语义 DTO，至少支持：
  - `title`
  - `contentMarkdown`
  - `stream`
  - `model`
  - `temperature`
  - `maxTokens`
2. 标题和正文至少填写一项；后端统一组装提示词，不要求调用方手工拼装 `messages`。
3. 返回结构尽量贴合 Chat Completions 响应，同时保证可提取业务结果。

### 7.3 结果要求

1. 可生成并回填：`slug`、`summary`、`seoTitle`、`seoDescription`。
2. 服务端需做结果兜底：
  - `slug` 规范化与唯一化候选。
  - 长度不超过后端校验限制。
3. AI 异常不影响文章手工编辑与保存。

## 八、验收标准

### 8.1 协议验收

1. Chat 与 Responses 协议接口都可用。
2. 两类协议都支持 `stream=true/false`。
3. 请求与响应字段符合 OpenAI 协议预期。

### 8.2 配置验收

1. 可维护多配置且单启用。
2. 启用配置可指定上游协议。
3. 协议不匹配时可返回明确错误。

### 8.3 业务验收

1. `posts/ai/meta/generate` 可按 `stream` 正常分流。
2. 底层确实调用 Chat Completions 服务方法。
3. 可稳定生成并回填文章元信息。

## 九、风险与约束

1. OpenAI 协议更新可能带来字段演进，需预留兼容策略。
2. 流式链路受网络影响更明显，需强化超时与中断处理。
3. 上游协议选择与业务接口依赖关系要清晰，否则容易配置误用。

## 十、评审重点

1. 是否确认 Chat 与 Responses 本期都实现。
2. 是否确认 `/v1/chat/completions` 与 `/v1/response(s)` 采用 OpenAI 兼容契约。
3. 是否确认 `posts/ai/meta/generate` 固定依赖 Chat 服务层。
4. 是否确认 `stream` 作为核心分流开关。
