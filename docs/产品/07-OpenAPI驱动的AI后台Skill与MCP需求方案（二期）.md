# OpenAPI 驱动的 AI 后台 Skill / MCP 需求方案（二期）

## 文档目标

本文档用于重新定义 `Yunyu` 后台 AI 能力的建设方向。

本期不再优先建设“站内自研 AI 操作平台页面 + 大量业务编排逻辑”，而是改为以下方向：

1. 以现有 `OpenAPI` 文档作为后台能力的单一事实来源。
2. 以 `MCP` 作为后台执行层，让任意支持 MCP 的 agent 都能复用同一套后台操作能力。
3. 以 `skill` 作为操作规范层，让任意 agent 都知道应该如何搜索接口、理解参数、确认风险、再执行。
4. 后续新增后台功能时，尽量做到“不改 AI 代码”或“只补少量接口描述信息”，AI 即可默认支持。

## 一、为什么要调整方向

上一版思路的问题在于：

1. 维护点太多
2. AI 逻辑过重
3. 容易和现有后台接口体系重复建设
4. 后续每新增一个后台功能，都可能要补一层 AI 适配

而当前项目其实已经具备一个非常关键的基础：

1. 后端已有 `OpenAPI` 文档入口 `/v3/api-docs`
2. 后台接口已经集中在 `/api/admin/**`
3. 后台接口已经受 `SUPER_ADMIN` 权限保护
4. 前端已存在稳定的登录态与 Bearer Token 机制

这意味着：

真正需要建设的，不应该是“再造一套 AI 后台业务层”，而应该是“把现有后台能力标准化暴露给 agent 使用”。

## 二、目标重述

### 2.1 总体目标

让 `Yunyu` 后台能力可以被任意 agent 安全、统一、低维护地调用，而不是只服务某一个页面、某一种模型或某一条 prompt。

### 2.2 核心目标

1. 任意 agent 可以通过统一的 `MCP` 能力访问后台管理接口。
2. 任意 agent 可以通过统一的 `skill` 学会如何正确使用这些能力。
3. 站长权限继续由后端现有 JWT + `SUPER_ADMIN` 机制控制。
4. 后续新增后台功能时，优先通过 `OpenAPI` 文档自动纳入，不做重复适配。
5. 写操作默认具备风险提示与确认流程。

## 三、核心判断

## 3.1 不是“skill 或 MCP 二选一”

这件事的正确结构应该是：

1. `OpenAPI`：能力来源
2. `MCP`：执行能力
3. `skill`：操作方法

三者职责不同：

### `OpenAPI`

负责回答：

1. 后台有哪些接口
2. 每个接口参数是什么
3. 返回值是什么
4. 哪些接口是新增、修改、删除

### `MCP`

负责回答：

1. agent 如何真正调用这些后台接口
2. 如何统一带上认证 token
3. 如何把接口能力暴露成稳定工具

### `skill`

负责回答：

1. agent 拿到需求后应该先做什么
2. 什么时候先搜索接口
3. 什么时候需要先预览或确认
4. 什么时候直接执行
5. 失败后如何解释

## 3.2 为什么单靠 skill 不够

单独做 `skill` 只能告诉 agent“应该怎么做”，但不能给 agent 提供真正稳定、跨模型复用的执行工具。

如果没有 `MCP`，skill 最终通常只能依赖：

1. shell + curl
2. 浏览器自动化
3. 手工拼 HTTP 请求

这会带来三个问题：

1. 不稳定
2. 不够通用
3. 不同 agent 之间复用性差

因此：

`skill` 适合做“通用操作规范”，`MCP` 才适合做“真正执行层”。

## 四、推荐方向

推荐将本需求定义为：

> 构建一个基于 `OpenAPI` 自动发现后台接口能力的 `Yunyu Admin MCP`，并配套一个 `yunyu-admin-operator` skill，供任意 agent 通过标准方式完成后台操作。

## 五、系统边界

### 5.1 本方案覆盖范围

优先覆盖所有 `/api/admin/**` 接口，包括但不限于：

1. 站点配置
2. 首页配置
3. 文章管理
4. 分类管理
5. 标签管理
6. 专题管理
7. 评论管理
8. 友链管理
9. 附件管理
10. 用户管理
11. AI 提供商配置
12. S3 配置
13. 后续新增后台模块

### 5.2 不覆盖范围

1. 直接操作数据库
2. 绕过后台现有权限系统
3. 直接替代后台页面的全部可视化体验
4. 把所有复杂业务判断都塞进 prompt

## 六、用户价值

### 6.1 对站长

站长可以直接对 agent 说：

1. “把站点标题改成云屿，副标题改成记录技术与生活。”
2. “帮我新建一篇文章草稿，标题叫《Spring Boot 与 Nuxt 联调记录》。”
3. “把《Nuxt 主题化实践》这篇文章摘要缩短一点。”
4. “把待审核评论里今天新增的垃圾评论都驳回，但先给我看命中结果。”

agent 不需要知道项目源码细节，只需要：

1. 通过 MCP 查询后台接口能力
2. 按 skill 约定完成操作

### 6.2 对后续扩展

以后只要后台新增一个新接口，并出现在 `OpenAPI` 文档里：

1. MCP 可以重新索引到它
2. skill 不需要重写
3. agent 理论上就具备调用能力

这比每次都给新功能做一层 AI 适配成本低很多。

## 七、鉴权要求

### 7.1 权限原则

AI 绝不是超级管理员。

AI 能做什么，完全取决于当前提供给它的 Bearer Token 拥有什么权限。

当前项目已有事实基础：

1. `/api/admin/**` 受 `SUPER_ADMIN` 保护
2. 普通用户 token 访问后台接口会失败
3. 因此只要使用站长 token，AI 才能调用后台能力

### 7.2 推荐 token 获取方式

因为当前前端已经把 token 存在浏览器中，建议支持三种方式：

#### 方式一：浏览器存储读取

优先从浏览器中读取 `yunyu_access_token`，这是当前项目已有的前端存储键。

可读取来源：

1. `localStorage.getItem('yunyu_access_token')`
2. Cookie `yunyu_access_token`

#### 方式二：手动粘贴 token

当 agent 无法直接访问浏览器上下文时，由站长手动提供 token。

#### 方式三：环境变量注入

在本地运行 MCP 时，通过环境变量注入 token。

### 7.3 token 使用约束

1. token 只用于当前站长自己的后台操作
2. MCP 每次启动后应先验证 `/api/auth/me`
3. 若返回的角色不是 `SUPER_ADMIN`，则拒绝提供后台写能力
4. token 失效时，提示重新登录并重新获取

## 八、为什么这条路更容易维护

## 8.1 能力来源统一

后台能力的来源不再是：

1. prompt 里手写
2. 文档里手写
3. agent 代码里手写

而是统一收敛为：

1. `OpenAPI` 文档

## 8.2 skill 稳定

skill 不需要记住每个接口的细节，它只负责一套稳定流程：

1. 搜索能力
2. 理解参数
3. 判断风险
4. 必要时确认
5. 执行
6. 总结结果

因此 skill 基本不会随着某个具体后台功能新增而频繁改动。

## 8.3 MCP 通用

MCP 提供的是通用后台访问能力，不绑定某个单一页面。

也就是说：

1. Codex 能用
2. 其他支持 MCP 的 agent 也能用
3. 后续即使更换模型，能力层也不需要重做

## 九、对“智能”的定义要调整

真正智能，不是“把所有后台逻辑写死在一个 AI 页面里”，而是：

1. agent 能自己发现可用能力
2. agent 能自己读接口参数和 schema
3. agent 能自己判断接下来该调用哪个操作
4. agent 能在高风险操作前先做确认

也就是说：

“智能”应该来自：

1. 更标准的能力暴露
2. 更好的字段描述
3. 更清晰的 skill 工作流

而不是来自越来越厚的手写业务 prompt。

## 十、需求分层

### 10.1 第一层：OpenAPI 可用

后台必须持续输出可供机器消费的接口文档：

1. 路径清晰
2. 参数完整
3. DTO 字段尽量有描述
4. 后台接口最好带 `summary` / `tag`

### 10.2 第二层：MCP 能消费 OpenAPI

MCP 需要做到：

1. 自动加载 `/v3/api-docs`
2. 自动筛选 `/api/admin/**`
3. 自动建立操作索引
4. 统一执行 HTTP 请求
5. 统一附带 Bearer Token

### 10.3 第三层：skill 教会 agent 怎么做

skill 需要明确：

1. 遇到模糊需求时如何先搜索接口
2. 遇到写操作时如何先预览和确认
3. 遇到批量删除时如何降级为安全流程
4. 遇到失败时如何解释错误

## 十一、推荐的交互原则

### 11.1 读操作

像查询文章、查看站点配置、查评论列表等，只要权限满足，可直接执行。

### 11.2 普通写操作

像更新站点标题、修改文章摘要、调整首页文案等，应默认先给出变更预览，再执行。

### 11.3 高风险操作

像删除文章、批量删除评论、覆盖敏感配置，应强制二次确认。

## 十二、后续新增功能默认支持的要求

这部分是本方案最核心的价值。

以后新增后台功能时，希望做到：

1. 只要新接口进入 `/v3/api-docs`
2. 且接口路径符合后台规范
3. 且字段描述基本完整

那么：

1. MCP 可以自动发现这个能力
2. skill 无需修改
3. 任意 agent 默认就能开始使用这个能力

如果是特殊高风险能力，则只补少量扩展描述，而不是重做整个 AI 功能。

## 十三、建议的最终交付形态

建议最终交付为一个本地插件或能力包，包含：

1. `Yunyu Admin MCP`
2. `yunyu-admin-operator` skill
3. 可选的浏览器 token 获取辅助能力

这样任意 agent 接入后，都能以一致方式操作你的后台。

## 十四、验收标准

### 14.1 基础验收

1. agent 能通过 MCP 列出后台可用操作
2. agent 能读取后台配置与文章详情
3. agent 能完成至少一条文章修改和一条站点配置修改

### 14.2 权限验收

1. 非 `SUPER_ADMIN` token 无法执行后台写操作
2. token 失效时会被正确识别
3. 未提供 token 时不能误执行后台操作

### 14.3 维护性验收

1. 后续新增后台接口后，无需重写 skill 主流程
2. 后续新增标准后台 CRUD 功能后，MCP 能自动发现并使用
3. 只有特殊高风险接口才需要额外补充少量描述

## 十五、结论

这次方案的主方向应明确调整为：

1. 不再优先做“重型站内 AI 操作平台”
2. 改为做“`OpenAPI` 驱动的后台 `MCP` + `skill`”
3. `MCP` 负责让任意 agent 都能真正操作后台
4. `skill` 负责让任意 agent 都会正确操作后台
5. `OpenAPI` 负责成为后台能力的唯一维护入口

这是当前项目里维护成本最低、扩展性最好、也最符合“任意 agent 可复用”的方案。
