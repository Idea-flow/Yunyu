# 前端路由与页面设计

## 文档目标

本文档用于明确 `Yunyu` 前端项目在第一阶段的技术边界、主题系统、UI 分层策略、前后台页面边界与后续扩展方向，避免前端工程在实现过程中失去统一性。

## 一、当前技术结论

当前前端项目采用以下组合：

- 前端框架：`Nuxt 4`
- 样式体系：`Tailwind CSS`
- 组件体系：`@nuxt/ui`
- 主题机制：`@nuxtjs/color-mode`
- 运行模式：前台以 `SSR` 为主，后台以登录后管理界面为主

当前阶段的核心原则不是“为了简单快速堆页面”，而是为“最美博客 / 内容站”建立长期可维护的设计底层。

## 二、前后台 UI 设计边界

### 2.1 统一的部分

前台与后台必须统一以下内容：

- 主题切换机制
- 设计 token
- 基础语义色
- 间距、圆角、阴影、边框层级规则
- 交互状态规则

统一这些内容的目的是：

- 保证整个产品属于同一品牌体系
- 让明亮、暗黑、系统模式能全局统一运作
- 降低后续维护成本

### 2.2 区分的部分

前台与后台不应追求“长得完全一样”。

前台更强调：

- 品牌感
- 内容表达
- 排版节奏
- 沉浸感
- 阅读体验

后台更强调：

- 信息密度
- 操作效率
- 稳定一致
- 长时间使用的舒适度

因此当前结论是：

- 前后台共享一套设计底层
- 前后台在页面表达和信息组织上分别设计

## 三、主题系统设计

### 3.1 支持模式

全站统一支持三种模式：

- `light`
- `dark`
- `system`

其中：

- `system` 表示跟随设备系统主题
- 默认策略为 `system`

### 3.2 技术实现

当前主题机制使用 `@nuxtjs/color-mode` 统一管理。

原因：

- `Nuxt 4` 采用服务端渲染，主题切换不能只依赖客户端临时处理
- `color-mode` 更适合处理主题持久化、系统模式与 SSR 场景的首屏体验
- 能作为前台与后台统一主题机制的基础

### 3.3 主题规则

主题系统必须遵循以下规则：

- 不允许在业务组件中大面积写死颜色值
- 优先使用语义类名与设计 token
- 明亮与暗黑模式必须共享同一套语义层
- 系统模式只是主题来源，不是独立第四套设计

## 四、UI 组件策略

### 4.1 为什么使用 `@nuxt/ui`

当前项目决定引入 `@nuxt/ui`，原因是：

- 与 `Nuxt 4` 集成自然
- 与 `Tailwind CSS` 配合顺畅
- 能提供稳定的基础组件能力
- 后续前后台都可以复用同一套组件层

### 4.2 使用原则

虽然引入了 `@nuxt/ui`，但不意味着直接使用默认风格就结束。

必须遵循以下原则：

- `@nuxt/ui` 是组件基础设施，不是最终品牌视觉
- 最终视觉仍需通过主题变量、token、布局与页面设计进行定制
- 前台页面要避免“控制台风”
- 后台页面要避免“内容站风”

## 五、前端目录建议

当前 `yunyu-web` 建议逐步演进为以下结构：

```text
app/
  app.vue
  assets/
    css/
      main.css
  components/
    common/
    content/
    admin/
    theme/
  layouts/
    default.vue
    admin.vue
  pages/
    index.vue
    posts/
    topics/
    admin/
  composables/
  middleware/
```

目录设计原则：

- `common/`：前后台共用组件
- `content/`：前台内容展示组件
- `admin/`：后台管理组件
- `theme/`：主题切换与主题辅助能力

## 六、第一阶段页面范围

第一阶段优先页面建议如下：

- 首页
- 文章详情页
- 专题页基础能力
- 共享登录页
- 后台首页
- 后台文章管理页

这些页面构成第一阶段前台体验和后台管理的最小闭环。

## 七、路由约定

当前第一阶段优先约定以下页面路由：

- `/`：前台首页
- `/login`：前台与后台共用登录页
- `/admin`：后台首页

其中登录页为共享入口，后续通过 `redirect` 参数和真实身份信息决定回跳目标，例如：

- `/login?redirect=/admin`

## 八、页面设计文档

### 8.1 共享登录页

#### 页面目标

统一承接前台与后台登录入口，避免维护两套登录页和重复交互逻辑。

#### UI 设计

- 左侧为品牌说明与登录语义展示
- 右侧为登录表单卡片
- 顶部保留主题切换入口
- 页面需兼顾品牌感与通用性

#### 布局设计

- 桌面端采用双栏布局
- 移动端自动折叠为单栏纵向结构
- 登录表单保持独立卡片容器，提升聚焦感

#### 交互设计

- 输入账号与密码后提交
- 若信息不完整，给出明确反馈
- 登录请求通过 Nuxt 服务端代理层转发到后端认证接口
- 登录成功后根据 `redirect` 参数回跳
- 当回跳目标为后台且当前用户不是站长时，明确提示“不是站长，不允许进入后台”
- 未指定回跳地址时默认进入 `/admin`

### 8.2 后台首页

#### 页面目标

作为后台管理工作台的第一屏，为站长提供运营总览、快捷操作、待处理事项和系统状态信息。

#### UI 设计

- 使用 `Nuxt UI` 的 dashboard 结构作为后台骨架
- 左侧为可折叠导航
- 主内容区展示总览卡片、快捷操作、进度状态和待处理事项
- 页面整体强调效率、层级和稳定性

#### 布局设计

- 使用后台独立布局 `admin`
- 左侧导航固定承载后台入口与主题切换
- 右侧内容区采用多卡片模块化布局
- 移动端优先保证信息模块可纵向堆叠

#### 交互设计

- 进入页面前先通过后台中间件检查登录态与站长角色
- 未登录时自动跳转到共享登录页
- 非站长用户访问后台时自动跳转回共享登录页并展示禁止提示
- 快捷操作区点击后进入后续业务流程
- 当前阶段先使用交互反馈占位
- 后续接入真实业务页面时保持布局和卡片结构不变

## 九、当前落地结论

当前已经明确：

- 前端使用 `Nuxt 4`
- 使用 `Tailwind CSS` 作为样式基础
- 使用 `@nuxt/ui` 作为组件基础设施
- 使用 `@nuxtjs/color-mode` 统一管理主题
- 默认支持 `light / dark / system`
- 前后台共享主题底层，但页面表达分开设计
- 登录页为前台与后台共用入口
- 后台首页优先作为第一批后台页面落地

后续如有设计系统、路由层级、布局体系变化，必须先同步更新本文档，再进入实现阶段。
