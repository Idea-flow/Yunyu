# 数据库设计（V2 可落地 DDL 方案）

## 文档目标

本文档用于沉淀 `云屿 / Yunyu` 当前阶段可直接落地的 `MySQL 8` 数据库设计方案。

本版目标不是继续扩展概念，而是明确：

- 当前阶段到底落哪些表
- 每张表的正式 DDL 是什么
- 真实查询场景下需要哪些复合索引
- 软删除、唯一约束、内容拆表如何统一

## 一、当前阶段范围

当前阶段正式落地以下表：

- `user`
- `user_auth`
- `category`
- `tag`
- `topic`
- `post`
- `post_content`
- `post_tag`
- `topic_post`
- `comment`
- `site_config`

当前阶段先不落以下表：

- `author`
- `media_file`
- `post_content_version`
- `product`
- `product_content`
- `digital_asset`
- `order`
- `order_item`
- `user_entitlement`

## 二、基础约定

### 2.1 数据库选型

当前阶段统一使用：

- `MySQL 8`

### 2.2 命名规范

当前阶段表名统一使用单数：

- `user`
- `post`
- `comment`
- `category`
- `tag`
- `topic`

关联表保持中划线语义拆分后的下划线形式：

- `post_tag`
- `topic_post`

### 2.3 通用字段规范

除关联表与配置表外，业务主表统一尽量具备以下字段：

- `id`
- `created_time`
- `updated_time`
- `deleted`

说明：

- `deleted` 统一表示软删除：`0` 为未删除，`1` 为已删除
- 涉及唯一业务字段时，优先采用“业务字段 + deleted”的联合唯一索引
- 所有时间字段统一使用 `DATETIME`
- 通用创建时间字段统一命名为 `created_time`
- 通用更新时间字段统一命名为 `updated_time`

### 2.4 当前阶段建模原则

- 文章主信息与正文内容拆表
- `post` 当前继续关联 `user_id`
- 认证方案第一版使用 `JWT access_token`
- `JWT` 第一版有效期固定为 `7 天`
- 当前阶段不引入 `refresh_token`
- 当前阶段不引入 `user_token`
- 第三方登录绑定统一收口到 `user_auth`
- 当前阶段不引入 `author` 独立建模
- 当前阶段不引入媒体中心
- 当前阶段不引入内容历史版本表
- 优先保证列表页、详情页、后台管理页查询性能

## 三、真实查询场景与索引策略

### 3.1 首页 / 文章列表页

常见查询条件：

- `status = 'PUBLISHED'`
- `deleted = 0`
- 按 `published_at DESC` 排序
- 部分场景叠加 `is_top = 1`
- 部分场景叠加 `is_recommend = 1`
- 分类页按 `category_id` 过滤

因此 `post` 需要重点补以下复合索引：

- `(status, deleted, published_at DESC, id DESC)`
- `(status, deleted, is_recommend, published_at DESC, id DESC)`
- `(category_id, status, deleted, published_at DESC, id DESC)`

### 3.2 后台文章管理页

常见查询条件：

- 按作者查：`user_id`
- 按状态查：`status`
- 排除已删除：`deleted = 0`
- 按更新时间倒序

因此 `post` 需要：

- `(user_id, status, deleted, updated_time DESC, id DESC)`

### 3.3 专题详情页

常见查询条件：

- 根据 `topic_id` 查文章列表
- 按 `sort_order ASC, id ASC` 排序

因此 `topic_post` 需要：

- `(topic_id, sort_order, id)`

### 3.4 评论列表页

常见查询条件：

- 按 `post_id` 查询
- 只取审核通过评论
- 排除软删除
- 根评论和楼中回复分层展示

因此 `comment` 需要：

- `(post_id, status, deleted, root_id, created_time, id)`

## 四、正式 DDL

### 4.1 用户表 `user`

```sql
CREATE TABLE `user` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '用户ID',
  `email` VARCHAR(128) NOT NULL COMMENT '邮箱，登录账号',
  `user_name` VARCHAR(64) NOT NULL COMMENT '用户名称/展示名称',
  `avatar_url` VARCHAR(255) DEFAULT NULL COMMENT '头像URL',
  `password` VARCHAR(255) NOT NULL COMMENT '密码原值',
  `password_hash` VARCHAR(255) NOT NULL COMMENT '密码哈希值',
  `role` VARCHAR(32) NOT NULL DEFAULT 'USER' COMMENT '角色：SUPER_ADMIN / USER',
  `status` VARCHAR(32) NOT NULL DEFAULT 'ACTIVE' COMMENT '状态：ACTIVE / DISABLED',
  `last_login_at` DATETIME DEFAULT NULL COMMENT '最后登录时间',
  `last_login_ip` VARCHAR(64) DEFAULT NULL COMMENT '最后登录IP',
  `created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `deleted` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否删除：0否 1是',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_email_deleted` (`email`, `deleted`),
  KEY `idx_user_status_deleted_created_time` (`status`, `deleted`, `created_time`, `id`),
  KEY `idx_user_created_time` (`created_time`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用户表';
```

设计说明：

- 保留 `password` 与 `password_hash`
- `email` 唯一约束调整为 `(email, deleted)`，避免软删除后无法重建同邮箱账号
- `status + deleted + created_time` 便于后台筛选用户列表

### 4.2 用户认证方式表 `user_auth`

```sql
CREATE TABLE `user_auth` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '认证ID',
  `user_id` BIGINT UNSIGNED NOT NULL COMMENT '用户ID',
  `auth_type` VARCHAR(32) NOT NULL COMMENT '认证类型：LOCAL / GITHUB / GOOGLE / LINUX_DO',
  `auth_identity` VARCHAR(191) NOT NULL COMMENT '认证唯一标识，本地可存邮箱，第三方存平台唯一用户ID',
  `auth_name` VARCHAR(100) DEFAULT NULL COMMENT '第三方平台用户名或展示名',
  `auth_email` VARCHAR(128) DEFAULT NULL COMMENT '第三方平台邮箱',
  `email_verified` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '邮箱是否已验证：0否 1是',
  `raw_user_info` JSON DEFAULT NULL COMMENT '第三方原始用户信息JSON',
  `created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_user_auth_type_identity` (`auth_type`, `auth_identity`),
  UNIQUE KEY `uk_user_auth_user_id_type` (`user_id`, `auth_type`),
  KEY `idx_user_auth_email` (`auth_email`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci COMMENT='用户认证方式表';
```

设计说明：

- `user` 负责用户主体，`user_auth` 负责登录方式绑定
- 本地登录使用 `LOCAL`，`auth_identity` 可直接保存邮箱
- 第三方登录统一使用平台唯一用户ID作为 `auth_identity`
- `auth_email` 仅作为辅助关联依据，不作为第三方主绑定键
- 通过 `(auth_type, auth_identity)` 保证同一第三方账号只绑定一个系统用户

```sql

  `created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `deleted` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否删除：0否 1是',
```

### 4.6 文章主表 `post`

```sql
CREATE TABLE `post` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT COMMENT '文章ID',
  `title` VARCHAR(200) NOT NULL COMMENT '文章标题',
  `slug` VARCHAR(220) NOT NULL COMMENT '文章唯一标识',

  `category_id` BIGINT UNSIGNED DEFAULT NULL COMMENT '分类ID',
  `status` VARCHAR(32) NOT NULL DEFAULT 'DRAFT' COMMENT '状态：DRAFT / PUBLISHED / OFFLINE',

  `is_recommend` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否推荐：0否 1是',
  `has_video` TINYINT(1) NOT NULL DEFAULT 0 COMMENT '是否有视频：0否 1是',
  `allow_comment` TINYINT(1) NOT NULL DEFAULT 1 COMMENT '是否允许评论：0否 1是',
  `seo_title` VARCHAR(255) DEFAULT NULL COMMENT 'SEO标题',
  `seo_description` VARCHAR(500) DEFAULT NULL COMMENT 'SEO描述',
  `published_at` DATETIME DEFAULT NULL COMMENT '发布时间',
  `sort_order` INT NOT NULL DEFAULT 0 COMMENT '排序值',
  `view_count` BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '浏览量',
  `like_count` BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '点赞数',
  `comment_count` BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '评论数',
  `created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `updated_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_post_slug_deleted` (`slug`, `deleted`),
  KEY `idx_post_publish_list` (`status`, `deleted`, `published_at` DESC, `id` DESC),
  KEY `idx_post_recommend_list` (`status`, `deleted`, `is_recommend`, `published_at` DESC, `id` DESC),
  KEY `idx_post_category_publish` (`category_id`, `status`, `deleted`, `published_at` DESC, `id` DESC),
  KEY `idx_post_user_manage` (`user_id`, `status`, `deleted`, `updated_time` DESC, `id` DESC),
  KEY `idx_post_top_publish` (`status`, `deleted`, `is_top`, `sort_order`, `published_at` DESC, `id` DESC),
```

设计说明：

- 当前阶段文章归属继续使用 `user_id`
- `slug` 唯一约束改为 `(slug, deleted)`
- 重点补齐前台列表、推荐列表、分类列表、后台管理的复合索引
- `is_top` 场景单独给出复合索引，避免首页置顶查询退化

## 六、当前阶段不做的能力

为避免第一阶段范围失控，以下能力明确不在本版 DDL 中落地：

- 多作者独立体系
- 媒体资源中心
- 内容版本回滚
- 商品与订单系统
- 用户权益与虚拟资产交付

## 七、当前方案结论

本版 `V2` 方案的核心特点是：

- 明确采用 `MySQL 8`
- 保留 `user.password`
- `post` 明确关联 `user_id`
- 删除当前阶段未落地的 `author`、`media_file`、`post_content_version`
- 补齐适合真实查询场景的复合索引
- 统一软删除下的唯一约束写法

该版本可以直接作为当前阶段建库基线使用。
