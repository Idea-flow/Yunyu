# AI 公共系统技术方案（一期）

## 文档目标

本方案用于定义 `Yunyu` AI 公共系统的一期技术落地，满足以下约束：

1. `OPENAI_CHAT` 与 `OPENAI_RESPONSES` 两协议都实现。
2. 对外提供 OpenAI 兼容接口：
  - `/v1/chat/completions`
  - `/v1/response`（兼容 `/v1/responses`）
3. 协议接口入参与出参尽量与 OpenAI 一致。
4. `posts/ai/meta/generate` 底层调用 Chat Completions 服务，并通过 `stream` 分流。

## 一、总体架构

采用六层结构：

1. `Controller Layer`：OpenAI 兼容接口 + 业务场景接口。
2. `Gateway Layer`：统一网关门面，负责路由、治理、流式调度。
3. `Protocol Service Layer`：Chat Service 与 Responses Service。
4. `Adapter Layer`：上游厂商协议适配（OpenAI Compatible）。
5. `Config Layer`：提供商配置与单启用治理。
6. `Scene Layer`：文章元信息生成等业务场景。

## 三、配置模型设计

### 3.1 提供商配置

```java
class AiProviderProfile {
    String profileKey;
    String name;
    Boolean enabled;
    String upstreamBaseUrl;
    String apiKey;
    String model;
    UpstreamProtocolType upstreamProtocol; // COMPLETIONS | RESPONSES
    Integer connectTimeoutMs;
    Integer readTimeoutMs;
    Integer writeTimeoutMs;
```

### 3.2 单启用规则

1. `enabled=true` 必须且只能一条。
2. `activeProfileKey` 必须命中启用配置。
3. 上游协议通过 `upstreamProtocol` 明确声明。

### 3.3 配置存储

沿用 `site_config`：

1. `ai.provider.profiles`
2. `ai.scene.post-meta`

## 四、OpenAI 兼容接口设计

## 4.1 Chat 接口

1. `POST /v1/chat/completions`
2. 请求体：OpenAI Chat Completions 字段结构。
3. 响应体：OpenAI Chat Completions 结构。
4. `stream=true`：返回 `text/event-stream`，按 Chat Chunk 格式输出。

## 4.2 Responses 接口

1. `POST /v1/response`
2. 兼容 `POST /v1/responses`
3. 请求体：OpenAI Responses 字段结构。
4. 响应体：OpenAI Responses 结构。
5. `stream=true`：按 Responses 事件流输出。

## 4.3 协议一致性策略

1. DTO 以 OpenAI 协议字段为主，不引入业务命名替换。
2. 透传未知字段（`extraFields` 或 `JsonNode`）避免字段丢失。
3. 错误响应尽量对齐 OpenAI `error` 结构。

## 五、统一网关 + 协议服务详细设计（Java 设计模式）

## 5.1 `Facade`（门面模式）

类：`AiGatewayFacade`

职责：

1. 统一入口：`invokeChat`、`streamChat`、`invokeResponses`、`streamResponses`。
2. 屏蔽配置读取、协议匹配、适配器选择、拦截器链细节。

## 5.2 `Strategy`（策略模式）

接口：`UpstreamProtocolAdapter`

实现：

1. `OpenAiCompletionsAdapter`
2. `OpenAiResponsesAdapter`

职责：按 `UpstreamProtocolType` 动态选择上游协议实现。

## 5.3 `Registry`（注册表模式）

类：`UpstreamProtocolAdapterRegistry`

职责：维护 `protocolType -> adapter` 映射，避免网关写大分支。

## 5.4 `Template Method`（模板方法）

抽象类：`AbstractOpenAiAdapter`（可选）

统一流程：

1. 构建请求。
2. 附加鉴权头。
3. 发起 HTTP。
4. 解析响应。
5. 异常映射。

子类处理协议差异：

1. Completions 请求/响应。
2. Responses 请求/响应。

## 5.5 `Chain of Responsibility`（责任链）

接口：`AiInvokeInterceptor`

链路建议：

1. `LoggingInterceptor`
2. `TimeoutInterceptor`
3. `RetryInterceptor`
4. `MetricsInterceptor`

职责：将治理能力与协议逻辑解耦。

## 5.6 `Observer`（观察者模式）

接口：`AiStreamListener`

事件：`START`、`DELTA`、`COMPLETE`、`ERROR`

职责：适配器上报流式分片，`SseBridge` 负责转换为 SSE 输出。

## 5.7 `Factory Method`（工厂方法）

类：`AiHttpClientFactory`

职责：按配置创建 HTTP 客户端并注入超时、连接池、通用 Header。

## 七、流式交互细节

### 7.1 Chat 流式

1. 触发条件：`request.stream == true`。
2. 输出格式：Chat Completion Chunk + 结束标识。
3. SSE 事件建议：`data: {chunk-json}`，结束时 `data: [DONE]`。

### 7.2 Responses 流式

1. 触发条件：`request.stream == true`。
2. 输出格式：Responses 事件流结构。
3. 结束事件：按 Responses 协议事件结束。

### 7.3 统一事件桥接

1. 适配器输出 `AiStreamEvent`。
2. `SseBridge` 按目标协议格式编码后写入 `SseEmitter`。

## 八、`posts/ai/meta/generate` 详细设计

### 8.1 接口职责

1. 接口：`POST /api/admin/posts/ai/meta/generate`。
2. 该接口是业务场景接口，不替代 OpenAI 兼容接口。
3. 底层固定调用 `ChatCompletionsService`。

### 8.2 参数与流程

1. 入参尽量贴合 Chat Completions 请求结构，重点包含：
  - `model`
  - `messages`
  - `stream`
  - `temperature`
  - `max_tokens` 等
2. 若 `stream=true`：
  - 调用 `chatCompletionsService.stream(...)`
  - SSE 逐步返回
3. 若 `stream=false`：
  - 调用 `chatCompletionsService.create(...)`
  - 一次性返回

### 8.3 输出处理

1. 模型输出内容要求为结构化 JSON（`slug/summary/seoTitle/seoDescription`）。
2. 服务端执行后处理：
  - `slug` 规范化与唯一化候选。
  - 字段长度截断。
3. 返回结构尽量保留 Chat 协议风格，便于统一消费。

## 九、协议与上游配置匹配规则

1. `/v1/chat/completions` 请求默认要求启用配置支持 `COMPLETIONS`。
2. `/v1/response(s)` 请求默认要求启用配置支持 `RESPONSES`。
3. 若协议不匹配：返回可读错误（配置与请求协议不一致）。
4. `posts/ai/meta/generate` 固定走 Chat，要求启用配置可用 `COMPLETIONS`。

## 十、错误与降级策略

1. 配置缺失：返回“未配置可用 AI 提供商”。
2. 协议不匹配：返回“当前启用配置不支持目标协议”。
3. 上游超时/限流：按重试策略处理，失败后返回标准错误。
4. AI 调用失败不影响文章正常手工编辑与保存。

## 十一、审计与可观测

建议日志表 `ai_invoke_log` 增加字段：

1. `protocol_type`
2. `is_stream`
3. `stream_chunk_count`
4. `first_token_latency_ms`
5. `upstream_profile_key`
6. `error_code` / `error_message`

## 十二、测试策略

### 12.1 单元测试

1. Chat 与 Responses DTO 编解码兼容性。
2. 网关协议路由与配置匹配校验。
3. 适配器流式分片解析。
4. `posts/ai/meta/generate` 的 `stream` 分流。

### 12.2 集成测试

1. `/v1/chat/completions` 非流式。
2. `/v1/chat/completions` 流式。
3. `/v1/response(s)` 非流式。
4. `/v1/response(s)` 流式。
5. `posts/ai/meta/generate` 非流式与流式。

### 12.3 回归测试

1. AI 不可用不影响文章编辑保存。
2. `slug` 唯一化在流式与非流式一致。

## 十三、实施顺序

1. 完成提供商配置与协议选择字段。
2. 完成 OpenAI 兼容接口 Controller（chat + response）。
3. 完成网关门面 + 拦截器链 + 适配器注册表。
4. 完成 Chat 与 Responses 两协议服务实现（含流式）。
5. 完成 `posts/ai/meta/generate` 场景服务并接入 Chat Service。
6. 完成日志、监控与测试。

## 十四、一期默认假设

1. 一期仅服务后台管理侧。
2. 文章元信息是首个业务场景。
3. OpenAI 兼容接口优先保证字段与交互一致性。
4. `/v1/response` 与 `/v1/responses` 在实现上保持等价处理。
