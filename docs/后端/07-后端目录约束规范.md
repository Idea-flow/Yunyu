# 后端目录约束规范

## 文档目标

本文档用于约束 `yunyu-server` 后端模块目录的日常新增与修改行为。

重点回答三个问题：

- 新增能力时应该放到哪个业务模块。
- 同一业务模块下应该进入 `admin`、`site`、`internal`、`publicapi` 还是 `bootstrap`。
- 哪些代码必须共享，哪些代码不应该为了“方便”跨边界堆在一起。

本文档偏向“落地规则”，默认基于当前仓库已经完成的真实目录结构给出约束。

本文档重点回答：

- 新增一个 controller / service / dto / vo 时应该放在哪里。
- `admin`、`site`、`internal`、`publicapi`、`bootstrap` 各自的边界是什么。
- 哪些共享代码可以留在根层，哪些不能继续混放。

本文档不负责：

- 详细解释这轮目录重构的历史背景与迁移过程。
- 穷举当前仓库每个模块的真实落地清单。
- 替代基础项目结构、依赖与启动骨架说明。

## 一、总原则

### 1.1 先按业务域放置，再按接口面细分

后端目录优先表达两层语义：

- 第一层：这是什么业务
- 第二层：这段能力服务于谁

示例：

```text
module/post/site/controller
module/post/admin/service
module/ai/internal/controller
module/system/bootstrap
```

不要反过来先想“这是 controller 还是 service”，再去找地方塞。

### 1.2 同一资源的前后台能力尽量回到同一业务域

例如：

- 文章后台管理能力放在 `module/post/admin`
- 文章前台公开能力放在 `module/post/site`

而不是：

- 后台文章在 `module/post`
- 前台文章又散落在 `module/site`

### 1.3 站点级聚合和资源级能力必须分开

例如首页、站点基础配置、版本信息属于站点整体视角，应放在：

```text
module/site/publicapi
```

而文章、分类、标签、专题的公开接口属于资源视角，应放回各自模块：

```text
module/post/site
module/category/site
module/tag/site
module/topic/site
```

### 1.4 共享持久化层优先保留在业务域根下

以下目录默认是业务域内共享层：

- `entity`
- `mapper`
- `convert`
- 部分 `model`

不要为了目录整齐把同一张表拆成多份 `entity/mapper`。

## 二、目录语义定义

### 2.1 `admin`

`admin` 表示后台管理接口面。

适合放入：

- 站长或后台管理员使用的接口
- 后台表单 DTO
- 后台列表与详情 VO
- 后台用例编排 Service

典型目录：

```text
module/post/admin/controller
module/post/admin/service
module/post/admin/dto
module/post/admin/vo
```

判断标准：

- 是否要求后台登录或更高权限
- 是否以创建、更新、删除、审核、配置为主
- 是否允许返回更完整的管理字段

### 2.2 `site`

`site` 表示前台站点接口面。

适合放入：

- 前台页面直接调用的公开接口
- 普通登录用户可访问接口
- 面向具体资源的只读查询和少量前台动作

典型目录：

```text
module/post/site/controller
module/post/site/service
module/post/site/dto
module/post/site/vo
```

判断标准：

- 是否主要服务于站点前台页面
- 是否围绕某个资源本身展开，而不是站点整体聚合
- 是否需要与同资源的后台能力成对维护

### 2.3 `publicapi`

`publicapi` 表示站点整体视角的公开聚合接口面。

当前只建议在 `module/site` 下使用。

适合放入：

- 首页聚合接口
- 站点基础信息接口
- 公开版本信息接口

典型目录：

```text
module/site/publicapi/controller
module/site/publicapi/service
module/site/publicapi/vo
```

不适合放入：

- 文章详情
- 分类详情
- 标签详情
- 专题详情

这些能力已经属于资源级，不属于站点整体聚合。

### 2.4 `internal`

`internal` 表示系统内部接口面。

适合放入：

- OpenAI 兼容协议入口
- 内部同步接口
- 回调接口
- 不直接开放给普通前后台页面的受控接口

典型目录：

```text
module/ai/internal/controller
module/ai/internal/service
module/storage/internal
```

判断标准：

- 是否不应由普通前台页面直接调用
- 是否更像系统能力、协议能力、回调能力
- 是否未来可能被 MCP、skill、agent 或 webhook 使用

### 2.5 `bootstrap`

`bootstrap` 表示启动引导与初始化基础设施。

当前只建议在 `module/system` 下使用。

适合放入：

- 启动阶段数据库检查
- 自动建库建表
- 初始化状态判断
- 启动 Runner
- 初始化脚本解析工具

典型目录：

```text
module/system/bootstrap
module/system/bootstrap/util
```

不适合放入：

- 普通业务 service
- 面向前端的系统信息接口
- 管理端系统设置接口

### 2.6 根层 `service`

不是所有 `service` 都必须继续细分到接口面。

以下情况允许保留在业务域根层：

- 明显是共享基础服务
- 同时被 `admin` 与 `site` 复用
- 不属于具体某个接口面，而属于业务域内基础支撑

当前典型示例：

```text
module/siteconfig/service
module/homepage/service
module/contentaccess/service
```

但要注意：

- 根层 `service` 不应无限膨胀
- 一旦明显偏后台或偏前台，就应迁到对应接口面

## 三、当前推荐放置规则

### 3.1 文章相关能力

应放在：

```text
module/post/admin
module/post/site
```

其中：

- 后台创建、编辑、发布、下线、管理查询：`post/admin`
- 前台列表、详情、访问校验、浏览量上报：`post/site`

不要再新增到 `module/site/publicapi`。

### 3.2 分类、标签、专题相关能力

应放在：

```text
module/category/admin | module/category/site
module/tag/admin      | module/tag/site
module/topic/admin    | module/topic/site
```

其中：

- 列表与详情等前台查询回到各自 `site`
- 后台增删改查保持在各自 `admin`

### 3.3 站点整体聚合能力

应放在：

```text
module/site/publicapi
```

适合新增：

- 首页聚合新字段
- 站点基础信息新字段
- 公开版本信息
- 未来全站导航、全站公共入口等站点级聚合返回

### 3.4 认证能力

应放在：

```text
module/auth/site
```

适合新增：

- 登录
- 注册
- 当前用户
- 刷新令牌（如果未来新增）
- 站点普通用户认证相关接口

不建议把这类能力挪到 `user`。

### 3.5 用户后台管理能力

应放在：

```text
module/user/admin
```

适合新增：

- 创建用户
- 更新用户
- 封禁/启用用户
- 后台用户列表

### 3.6 AI 能力

当前约束如下：

```text
module/ai/admin
module/ai/internal
module/ai/adapter
module/ai/model
module/ai/stream
```

规则：

- 后台 AI 配置、后台文章 AI 能力：`ai/admin`
- OpenAI 兼容协议、内部受控入口：`ai/internal`
- 上游协议适配：`ai/adapter`
- 流式事件：`ai/stream`
- 配置模型与协议枚举：`ai/model`

### 3.7 系统启动初始化能力

应放在：

```text
module/system/bootstrap
```

规则：

- 和应用启动强绑定的初始化逻辑进 `bootstrap`
- 普通系统业务不要混进 `bootstrap`

## 四、DTO / VO / Service 放置规则

### 4.1 DTO 放置规则

默认规则：DTO 跟着接口面走。

示例：

```text
AdminPostCreateRequest      -> module/post/admin/dto
SitePostQueryRequest        -> module/post/site/dto
AdminSiteAiProviderConfigUpdateRequest -> module/ai/admin/dto
```

不要再把同一业务域下所有 DTO 全塞回根层 `dto`。

### 4.2 VO 放置规则

默认规则：VO 跟着消费方走。

示例：

```text
AdminPostItemResponse       -> module/post/admin/vo
SitePostDetailResponse      -> module/post/site/vo
SiteHomeResponse            -> module/site/publicapi/vo
```

如果一个 VO 明显只服务于前台文章详情，就不要继续放在 `module/site/publicapi/vo`。

### 4.3 Service 放置规则

优先判断它属于哪种 service：

- 后台用例编排 service
- 前台资源查询 service
- 站点级聚合 service
- 共享基础 service
- 内部协议/系统引导 service

示例：

- `PostSiteService`：前台资源查询 service
- `CategorySiteService`：前台资源查询 service
- `SitePublicService`：站点级聚合 service
- `ContentAccessGrantService`：共享基础 service
- `DatabaseBootstrapService`：系统引导 service

## 五、禁止事项

### 5.1 禁止把资源级前台接口重新塞回 `module/site/publicapi`

以下能力不应再新增到 `site/publicapi`：

- 文章详情
- 分类详情
- 标签详情
- 专题详情
- 资源级交互动作

### 5.2 禁止为了省事跨接口面混放 DTO / VO

例如不要新增：

```text
module/post/dto/SitePostQueryRequest
module/post/vo/AdminPostItemResponse
```

只要它明显属于某个接口面，就应该放在对应子目录。

### 5.3 禁止把 `bootstrap` 当成普通系统模块

`bootstrap` 只处理启动引导与初始化，不处理常规业务。

### 5.4 禁止把共享持久化层复制多份

不要为了目录整齐复制：

- 多份 `entity`
- 多份 `mapper`
- 多份相同表模型

## 六、AI / Agent 落地约束

为了让 AI agent 后续改代码时更稳定，新增代码时建议遵循以下顺序：

1. 先判断业务域
2. 再判断接口面
3. 最后判断技术层目录

例如新增“前台文章点赞”：

- 业务域：`post`
- 接口面：`site`
- 技术层：`controller` + `service` + 必要 `dto/vo`

因此目录应优先考虑：

```text
module/post/site/controller
module/post/site/service
module/post/site/dto
module/post/site/vo
```

而不是先想“这是前台接口，所以往 `site/publicapi` 塞”。

## 七、文档与代码同步规则

只要目录边界发生变化，至少同步以下内容之一：

- `docs/后端/06-后端模块目录重构方案.md`
- 本文档
- 相关 `package-info.java`

如果新增了新的接口面目录，例如未来新增：

- `module/xxx/internal`
- `module/xxx/publicapi`
- `module/xxx/site`

则应同步补充：

- 该目录的包说明
- 该目录的使用场景
- 不该放什么

## 八、推荐检查清单

新增或迁移代码前，建议快速自查：

1. 这段能力属于哪个业务域？
2. 这段能力是给后台、前台、内部系统还是站点整体聚合用？
3. DTO / VO 是否已经放进对应接口面？
4. 是否误把资源级能力塞进了 `site/publicapi`？
5. 是否误把启动引导逻辑塞进了普通 `service`？
6. 是否需要更新 `package-info` 或目录文档？

## 九、当前默认结论

基于当前仓库结构，后续新增能力默认按下面的方向落位：

- 站长后台能力：`admin`
- 资源前台能力：`site`
- 站点整体公开聚合：`site/publicapi`
- 内部协议/受控入口：`internal`
- 启动初始化基础设施：`system/bootstrap`
- 共享持久化与基础服务：业务域根层共享目录

如果一个能力同时具备“资源属性”和“站点聚合属性”，优先遵循：

- 资源本身的查询与动作回资源模块
- 聚合编排回 `site/publicapi`
