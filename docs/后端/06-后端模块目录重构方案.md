# 后端模块目录重构方案

## 文档目标

本文档用于说明 `yunyu-server` 当前后端模块目录的主要问题、推荐的目标结构、`admin` 与前台 `site` 的边界划分方式，以及建议的分期迁移策略，作为后续后端目录治理与模块演进的落地依据。

## 一、背景与问题

当前后端主要采用如下组织方式：

- 顶层按业务域分包，例如 `module/post`、`module/comment`、`module/site`
- 每个业务域下继续按技术分层拆分，例如 `controller`、`service`、`dto`、`vo`、`entity`、`mapper`
- 后台接口与前台接口主要通过类名前缀区分，例如 `AdminPostController`、`SiteCommentController`

这种结构在项目早期成本低、落地快，但随着模块增多，会出现以下问题：

### 1.1 `admin` 与前台 `site` 的边界只体现在类名，不体现在目录

例如当前存在：

- `module/comment/controller/AdminCommentController`
- `module/comment/controller/SiteCommentController`

它们虽然语义不同，但仍然处于同一层目录下，容易出现以下问题：

- 管理端和前台端 DTO / VO 混用
- 权限边界只能靠类名和代码习惯维持
- OpenAPI 文档和 AI skill 难以按接口面自然分组

### 1.2 `service` 层同时承担“应用编排”和“业务规则”职责

当前存在以下混合情况：

- `AdminPostService` 更偏向后台用例编排
- 前台公开内容服务更偏向前台查询与聚合
- `HomepageConfigService` 更偏向配置聚合读写

它们都叫 `service`，但职责层次并不一致，后续容易出现：

- 同一个资源在不同服务里重复写规则
- `admin` 和 `site` 各维护一套相近逻辑
- 后续新增 `internal` 场景时无处安放

### 1.3 `site` 模块过大，容易持续膨胀

当前 `module/site` 实际承载了多类不同语义的能力，例如：

- 站点基础配置
- 首页配置
- S3 存储配置
- 站点公开内容查询
- 站点版本信息

这会导致 `site` 模块逐步演化为“超级模块”，降低可维护性。

### 1.4 DTO / VO 不按调用侧拆分，后续维护成本高

当前大多数模块为：

```text
module/post/dto
module/post/vo
```

这会导致：

- 后台 DTO 和前台 DTO 共处一个目录
- VO 仅通过命名区分用途
- 后续 OpenAPI 文档生成与 AI 消费时，需要额外靠命名规则识别

## 二、重构目标

本次目录重构不追求一次性引入完整 DDD 分层，而是优先达成以下四个目标：

### 2.1 明确接口边界

让以下三类接口在目录层面就能一眼区分：

- `admin`：后台管理接口
- `site`：前台公开接口或普通登录用户接口
- `internal`：系统内部接口、AI 内部接口、回调接口

### 2.2 控制模块膨胀

把过大的业务域拆开，尤其是 `site`，避免把不同职责长期堆积在一个模块下。

### 2.3 保持当前项目风格可迁移

仍然保留：

- 单体 Spring Boot 项目
- `module/<业务域>` 作为第一层业务域目录
- `entity / mapper / convert` 的现有风格

避免一次性引入过重的架构复杂度。

### 2.4 让 AI 与人都更容易理解目录职责

重构后，AI agent、OpenAPI 文档生成脚本、skill 文档都可以更自然地基于目录结构判断：

- 哪些接口属于后台
- 哪些 DTO / VO 只能用于前台
- 哪些能力属于系统内部，不应直接暴露给前端

- 如业务规则开始复杂，再增加 `domain` 子目录承载公共规则

## 四、各目录职责说明

### 4.1 `admin`

用于承载后台管理能力。

典型特征：

- 面向站长或后台管理员
- 读写操作较多
- 返回字段更完整
- 权限校验更严格

典型目录：

```text
module/post/admin/controller
module/post/admin/service
module/post/admin/dto
module/post/admin/vo
```

适合放入：

- `AdminPostController`
- `AdminPostService`
- `AdminPostCreateRequest`
- `AdminPostDetailResponse`

### 4.2 `site`

用于承载前台公开接口和普通用户可访问接口。

典型特征：

- 以查询为主
- 写操作较少
- 返回字段经过裁剪
- 不能暴露后台管理态字段

适合放入：

- 前台文章列表与详情
- 前台评论提交与查询
- 前台友链列表
- 站点公开基础信息

### 4.3 `internal`

用于承载不直接开放给普通前后台页面的系统内部能力。

典型场景：

- OpenAI 兼容协议入口
- 对象存储回调接口
- 内部同步接口
- 未来 MCP / skill 专用受控接口

这层目录可以让项目后续更容易支持：

- AI 自动化操作
- 内部令牌认证
- Webhook / callback

而不会把这些能力硬塞到 `admin` 或 `site` 目录中。

### 4.4 `entity / mapper / convert`

这三类目录继续保留在业务域根下，不按 `admin/site` 重复拆分。

原因：

- 数据表通常是共享的，不因接口面不同而拆分
- `mapper` 是持久化基础设施，天然应共享
- `convert` 主要承担对象转换职责，适合统一收口

推荐原则：

- `controller` 不直接访问 `mapper`
- `admin/service` 与 `site/service` 通过共享的基础查询服务、领域服务或转换器复用底层能力

## 五、模块拆分建议

### 5.1 `site` 模块建议拆分

当前 `module/site` 语义过大，建议拆成以下几个业务域：

#### 5.1.1 `siteconfig`

用于存放：

- 站点名称、副标题、Logo、Favicon
- 默认标题与默认描述
- 公众号验证码访问配置
- 品牌色配置

#### 5.1.2 `homepage`

用于存放：

- 首页布局配置
- 首页模块顺序
- 首页展示开关
- 首页视觉配置

#### 5.1.3 `storage`

用于存放：

- S3 配置
- 上传参数配置
- 存储策略
- 未来对象存储回调能力

#### 5.1.4 `site` 或 `publicsite`

用于存放公开聚合查询能力，例如：

- 首页公开内容聚合
- 公开文章流
- 公开站点版本接口

如果后续觉得 `site` 命名仍然过泛，也可以将其进一步改名为 `publicsite` 或 `publiccontent`。

### 5.2 `auth` 与 `user` 保持职责分离

建议规则：

- `auth/site` 负责登录、注册、当前用户、令牌
- `user/admin` 负责后台用户管理、用户信息维护、角色状态调整

避免把“身份认证流程”和“用户管理流程”混在同一模块中持续膨胀。

### 5.3 `ai` 模块建议显式分为“配置、场景、内部接口”

推荐形态：

```text
module/ai
├── admin
│   ├── controller
│   ├── service
│   ├── dto
│   └── vo
├── internal
│   ├── controller
│   ├── service
│   ├── dto
│   └── vo
├── adapter
├── scene
├── stream
└── model
```

各子目录职责：

- `admin`：后台 AI 提供商配置、提示词模板配置、面向后台的 AI 场景入口
- `internal`：OpenAI 兼容协议、内部受控调用接口
- `adapter`：上游 AI 供应商协议适配
- `scene`：文章 SEO、摘要、标签推荐等业务场景
- `stream`：流式输出支持

补充说明：

- 当前代码已落地为 `ai/admin` + `ai/internal` + `adapter/model/stream`
- `scene/post/PostMetaGenerateService` 目前暂放在 `ai/admin/scene/post`，因为它明确服务于后台文章编辑场景
- 后续如果出现更多跨接口面的 AI 业务场景，再考虑把公共场景沉到 `ai/scene`

## 六、控制器与服务层职责重构建议

### 6.1 Controller 职责

`controller` 只负责：

- 接收请求
- 参数校验
- 调用应用服务
- 返回统一响应

不建议在 `controller` 中承担：

- 复杂业务判断
- 多实体写入编排
- 直接访问 `mapper`

### 6.2 Admin Service 职责

`admin/service` 负责后台用例编排，例如：

- 创建文章
- 更新文章
- 上下线文章
- 更新站点配置
- 审核评论

它表达的是“后台管理意图”。

### 6.3 Site Service 职责

`site/service` 负责前台查询与前台交互，例如：

- 获取文章列表
- 获取文章详情
- 提交评论
- 获取公开友链

它表达的是“公开站点访问意图”。

### 6.4 公共业务规则承载位置

对于既会被后台调用、也会被前台调用的公共规则，建议逐步下沉到：

- 模块公共 service
- `domain` 子目录
- 共享 convert / query helper

例如：

- `slug` 唯一性校验
- 文章状态流转校验
- 内容访问权限判定
- 标签、分类、专题关联关系维护

这样可以避免：

- `AdminPostService` 和前台文章/公开内容服务各维护一套近似逻辑

## 七、DTO / VO 命名与目录规范建议

### 7.1 推荐目录组织

推荐改为：

```text
module/post/admin/dto
module/post/admin/vo
module/post/site/dto
module/post/site/vo
```

而不是继续使用：

```text
module/post/dto
module/post/vo
```

### 7.2 命名建议

迁移初期建议：

- 类名先保持现有 `AdminXXXCreateRequest`、`SiteXXXDetailResponse` 风格不变
- 先迁目录，再决定是否逐步简化命名

原因：

- 风险更低
- 减少一次性大规模重命名带来的调用链改动
- 与当前前端类型、OpenAPI 文档、skill 文档更容易平滑兼容

中后期如果目录已经稳定，可考虑简化为：

- `post/admin/dto/PostCreateRequest`
- `post/site/vo/PostDetailResponse`

但这一步不建议在本轮目录重构时同步推进。

## 八、迁移优先级建议

不建议一次性重构全部模块。推荐按以下三期推进。

### 8.1 第一阶段：先显式拆出 `admin/site`

本阶段已完成的模块：

- `comment`
- `friendlink`
- `post`
- `category`
- `tag`
- `topic`
- `user`

这一步只做：

- 包路径调整
- 目录归类
- import 更新

尽量不改业务逻辑。

### 8.2 第二阶段：拆分 `site` 大模块

本阶段已完成第一轮拆分，已将原 `module/site` 拆成：

- `siteconfig`
- `homepage`
- `storage`
- `site`

当前仍保留的后续优化空间：

- 进一步评估 `site` 是否需要继续细分为 `publicsite` / `publiccontent`
- 评估 `homepage/service` 是否继续下沉到更明确的 `admin/service` 与共享 service 组合
- 评估 `storage` 是否补充 `internal` 目录承载未来回调或内部任务接口

### 8.3 第三阶段：补充公共领域层

当业务复杂度继续上升后，再补：

- `domain`
- 公共 query service
- 公共 rule service

## 九、落地约束规则

目录重构后，建议补充以下开发约束：

### 9.1 目录引用约束

- `admin` 不应直接引用 `site/vo`
- `site` 不应直接引用 `admin/dto`
- `controller` 不应直接访问 `mapper`

### 9.2 接口边界约束

- `/api/admin/**` 只放在 `admin/controller`
- `/api/site/**` 只放在 `site/controller`
- `/api/internal/**` 只放在 `internal/controller`

### 9.3 文档与 OpenAPI 约束

- Admin API 文档按 `admin` 目录组织
- 前台 API 文档按 `site` 目录组织
- skill 只消费 `admin` 文档，不读取 `site` 公开接口

### 9.4 复用约束

- 共用数据表可以共享 `entity/mapper`
- 共用转换逻辑优先放 `convert`
- 共用规则优先放模块内公共 service 或 `domain`

## 十一、结论

对于 Yunyu 当前阶段，最适合的不是一步到位引入完整 DDD，而是采用：

- 保持单体项目
- 保持 `module/<业务域>` 第一层结构
- 在业务域内部显式拆出 `admin / site / internal`
- 将超大的 `site` 模块按职责拆分
- 后续再按需要逐步补充 `domain`

该方案的优点是：

- 改造成本可控
- 能快速厘清后台与前台边界
- 对 OpenAPI、AI skill、MCP 扩展更友好
- 与当前仓库结构兼容度高，适合渐进演进
