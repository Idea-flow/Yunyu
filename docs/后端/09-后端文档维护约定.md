# 后端文档维护约定

## 文档目标

本文档用于约定 `docs/后端` 目录下各核心文档的职责边界与新增专题文档的拆分规则。

目标是避免后续继续出现以下问题：

- 多篇文档重复讲同一件事。
- 同一个主题既写在基础文档里，又写在重构方案里。
- 新增一篇专题后，不知道应该挂到哪里，也不知道应该和哪篇主文档互相引用。

## 一、主文档职责分工

### 1.1 `01-后端结构与依赖说明.md`

负责内容：

- 仓库层与 `yunyu-server` 应用层基础结构。
- `common`、`config`、`infrastructure`、`security` 的职责。
- 基础依赖、启动骨架、初始化资源的入口说明。
- 新同学第一次进入后端仓库时的基础认知。

不负责内容：

- 解释目录为什么要重构。
- 给出完整业务模块地图。
- 规定新增 controller / service / dto / vo 应该放哪。

维护原则：

- 只保留“基础骨架”。
- 如果某段内容已经在 `08` 里更适合表达，就不要在 `01` 里重复展开。

### 1.2 `06-后端模块目录重构方案.md`

负责内容：

- 旧结构的问题分析。
- 目录重构目标、目标结构与迁移策略。
- 已完成迁移的阶段性总结。

不负责内容：

- 当前真实目录的唯一事实来源。
- 日常新增代码的放置规则手册。

维护原则：

- 允许保留历史问题示例。
- 如果当前真实结构已经发生变化，要同步更新“现状说明”，但不要把它改写成纯地图文档。

### 1.3 `07-后端目录约束规范.md`

负责内容：

- 新增代码的目录落位规则。
- `admin`、`site`、`internal`、`publicapi`、`bootstrap` 的边界。
- 共享 `service`、DTO、VO、Controller 的放置规范。
- 禁止事项与检查清单。

不负责内容：

- 详细迁移历史。
- 当前所有模块的全量清单。

维护原则：

- 这是“规则文档”，不是“目录扫描结果”。
- 规则优先写“应该怎么做”，而不是堆大量当前类名。

### 1.4 `08-当前真实模块地图.md`

负责内容：

- 当前真实业务模块清单。
- 当前各模块已经落地的接口面分布。
- 精简版真实目录树摘要。
- 查代码时的快速定位建议。

不负责内容：

- 解释为什么重构。
- 规定未来新增代码必须如何放置。
- 说明基础依赖与安全骨架。

维护原则：

- 这是“事实地图”。
- 当真实目录发生迁移时，应优先更新这篇文档。
- 当 `01` 与 `08` 存在重叠时，优先让模块明细收敛到 `08`。

## 二、什么时候新开专题文档

出现以下情况时，建议新开专题文档，而不是继续往 `01 / 06 / 07 / 08` 里堆：

- 某个功能链路已经形成独立方案，例如上传、对象存储、AI 集成、评论审核。
- 某个专题需要包含设计目标、接口约束、实现步骤、回滚策略、测试要点。
- 某个主题已经超过“目录结构说明”的范畴，进入业务方案或技术方案层面。
- 某个子专题后续还会单独持续演进，例如 Native Image、部署、S3 直传。

适合新开专题文档的典型例子：

- 附件上传与对象存储方案
- AI 能力接入与提供商配置方案
- 初始化与引导链路说明
- Native Image 支持与部署
- MCP / skill / agent 驱动后台操作方案

## 三、什么时候只更新主文档，不新开文档

以下情况通常只需要更新现有主文档：

- 后端基础骨架发生小范围调整：更新 `01`
- 目录边界规则调整：更新 `07`
- 目录迁移现状或历史结论变化：更新 `06`
- 真实模块新增、删除、迁移：更新 `08`
- 文档导航关系变化：更新 `README`

## 四、新增文档的命名与挂载建议

建议规则：

- 主线结构文档继续使用编号：`01`、`06`、`07`、`08`、`09`
- 独立专题文档使用专题名，不强行塞进主线结构文档
- 如果一个主题已经形成多个子文档，可单独建子目录，例如 `native-image/`

新增专题文档后，至少要同步两处：

1. 更新 [README.md](/Users/wangpenglong/projects/full-stack-project/Yunyu/docs/后端/README.md) 索引
2. 在最相关的一篇主文档中补一个“相关文档建议”链接

## 五、维护检查清单

每次修改 `docs/后端` 后，建议至少自查以下问题：

1. 这段内容是在讲“基础骨架”、“重构背景”、“落位规则”还是“真实地图”？
2. 如果已经有更合适的主文档承载，是否应该移动而不是重复写？
3. 当前文档里的路径、模块名、接口面名是否仍与真实代码一致？
4. 是否需要同步更新 `README` 的导航？
5. 是否应该补一条“相关文档建议”，帮助读者继续跳转？

## 六、推荐维护顺序

如果后端目录再次发生变化，推荐按以下顺序更新文档：

1. 先更新 [08-当前真实模块地图.md](/Users/wangpenglong/projects/full-stack-project/Yunyu/docs/后端/08-当前真实模块地图.md)
2. 再看是否需要更新 [07-后端目录约束规范.md](/Users/wangpenglong/projects/full-stack-project/Yunyu/docs/后端/07-后端目录约束规范.md)
3. 如果变化涉及目录治理结论，再更新 [06-后端模块目录重构方案.md](/Users/wangpenglong/projects/full-stack-project/Yunyu/docs/后端/06-后端模块目录重构方案.md)
4. 如果变化影响基础骨架认知，再更新 [01-后端结构与依赖说明.md](/Users/wangpenglong/projects/full-stack-project/Yunyu/docs/后端/01-后端结构与依赖说明.md)
5. 最后更新 [README.md](/Users/wangpenglong/projects/full-stack-project/Yunyu/docs/后端/README.md) 导航
