# 内容模型与渲染策略

## 文档目标

本文档用于定义 `云屿 / Yunyu` 的平台内容如何存储、如何渲染、如何兼顾 SEO、性能与未来 App 原生渲染能力。

## 一、核心结论

本项目采用“内容源与展示结果分离”的设计：

- 内容源：以 `Markdown` 为主
- 富内容：以结构化内容块为辅
- Web 展示：由 `Nuxt 4 SSR` 输出 HTML
- 内容主存储：服务端数据库
- HTML：作为 Web 端高性能展示用的派生数据

本项目不把 HTML 视为唯一真相，也不把客户端渲染作为主详情页方案。

## 二、为什么不只存 Markdown

如果详情页每次请求都实时把 Markdown 转 HTML，会带来以下问题：

- SSR 渲染链更长
- 内容页首屏性能波动更大
- 目录、摘要、纯文本等派生字段需要额外处理
- 缓存粒度不够理想

因此推荐：

- 存 Markdown 原文
- 同时存 HTML 派生结果

## 三、为什么不只存 HTML

只存 HTML 的问题在于：

- 不利于编辑和再次加工
- 不利于未来多端复用
- 不利于 App 侧做原生视图渲染
- 不利于富内容结构化扩展

因此 HTML 只能是派生结果，不能是唯一主内容。

## 四、最终推荐的内容结构

### 4.1 基础内容

基础内容采用 Markdown 表达，适合：

- 标题
- 段落
- 引用
- 列表
- 代码块
- 普通图片

### 4.2 富内容块

富内容采用结构化内容块表达，适合：

- 九宫格图片
- 商品卡片
- 下载区块
- CTA 区块
- FAQ 区块
- 对比表
- 视频模块

### 4.3 内容数据形态

服务端建议同时维护以下数据：

- `content_markdown`
- `content_html`
- `content_plain_text`
- `content_toc_json`
- `content_blocks_json`
- `summary`
- `reading_time`

其中：

- `content_markdown`：主内容源
- `content_html`：Web 详情页高性能展示用
- `content_plain_text`：搜索、审核、摘要基础
- `content_toc_json`：目录
- `content_blocks_json`：富内容块

## 五、渲染职责划分

### 5.1 Spring Boot 4 负责

- 存储内容源
- 存储富内容块
- 发布时生成 HTML、摘要、纯文本、目录等派生数据
- 提供内容详情 API

### 5.2 Nuxt 4 负责

- 在 SSR 阶段获取内容详情
- 输出完整 HTML 页面
- 控制正文排版、代码高亮、主题样式
- 把结构化内容块渲染为 Vue 组件

### 5.3 客户端不负责

对于主内容详情页，不推荐由客户端首次渲染 Markdown 再生成正文 HTML。

## 六、详情页性能策略

详情页为了做到尽可能快，建议采用以下策略：

- 内容发布时预生成 HTML
- 详情接口直接返回 HTML
- 列表接口不返回正文大字段
- Nuxt SSR 直接拼装详情页
- 边缘缓存与服务端缓存结合使用

## 七、为什么九宫格图片不能只靠纯 Markdown

九宫格图片、本质上属于富内容模块，而不是普通正文段落。

如果只靠纯 Markdown，会存在这些问题：

- 表达能力不足
- 后台编辑体验差
- App 侧难以原生渲染
- 后续展示样式改造不够优雅

因此九宫格图片建议采用结构化内容块表示。

示例结构：

```json
{
  "type": "image_grid",
  "layout": "3x3",
  "items": [
    {
      "url": "https://example.com/1.jpg",
      "alt": "图片1"
    },
    {
      "url": "https://example.com/2.jpg",
      "alt": "图片2"
    }
  ]
}
```

Nuxt Web 端可将其 SSR 为 HTML 网格结构，iOS 端则可渲染为原生视图。

## 八、Web、后台、App 的统一策略

### 8.1 Web

- 使用 `content_html + content_blocks_json`
- Nuxt SSR 输出页面
- 样式由前端控制

### 8.2 后台

- 编辑器维护 Markdown
- 特殊模块维护结构化内容块
- 发布时生成派生数据

### 8.3 App

- 不以 HTML 作为唯一渲染真相
- 可优先读取结构化内容块
- 基础段落可由 Markdown/HTML 转换结果辅助展示

## 九、阶段性实施建议

### 第一阶段

- 存 Markdown 原文
- 存 HTML 派生结果
- 存目录与摘要
- 富内容块先支持少量类型

### 第二阶段

- 扩充块类型
- 支持结构化内容编排
- 为 App 输出更稳定的结构化接口

## 十、当前定稿结论

本项目当前内容渲染方案正式定为：

- 主内容源使用 `Markdown`
- 富内容使用结构化内容块
- 内容主数据存服务端数据库
- HTML 作为详情页展示派生结果
- `Nuxt 4 SSR` 负责 Web 最终页面输出
