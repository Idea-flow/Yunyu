# 总体技术架构

## 文档目标

本文档用于确定 `云屿 / Yunyu` 从个人内容站第一阶段到内容平台阶段的技术主线，覆盖前端框架、UI 方案、Markdown 渲染、后端技术栈、ORM、认证体系与演进边界。

## 项目名称

- 中文名：`云屿`
- 英文名：`Yunyu`

## 一、总体结论

当前项目采用以下核心技术栈：

- 前端主技术：`Nuxt 4`
- 渲染模式：`SSR + 部分静态预渲染`
- UI 技术：`Tailwind CSS + 自定义设计系统 + 局部引入 Nuxt UI`
- 内容渲染：服务端内容存储 + `Markdown` 发布时转换 + Nuxt SSR 输出
- 后端主技术：`Spring Boot 4`
- ORM：`MyBatis-Plus`
- 数据库：优先 `PostgreSQL`，如团队更熟悉也可使用 `MySQL 8`
- 认证体系：后端统一 `Spring Security`，前台与后台共用认证基础设施但采用不同登录策略

本项目定位不是一次性博客站，而是具备平台扩展能力的内容系统，因此技术设计遵循以下原则：

- 前端体验与 SEO 优先
- 后端按平台能力建模，不按页面堆接口
- 早期保持单体架构，避免过早微服务化
- UI 体系可定制，避免被组件库完全锁死
- Markdown 内容能力必须支持后续商品、专题、下载资源、富组件嵌入
- 平台主内容必须存放于服务端，而不是长期绑定在前端仓库文件系统中

## 二、Nuxt 4 技术定位

### 2.1 为什么以 Nuxt 4 作为主技术

选择 `Nuxt 4` 的原因：

- 适合内容站与 SEO 场景，支持 SSR、SSG、Hybrid Rendering。
- 路由、布局、中间件、服务端数据获取能力成熟。
- 与 `@nuxt/content` 配合后，可以高效支持 Markdown 内容渲染与结构化查询。
- 后续无论是官网前台、用户中心还是运营后台，都可以保持在同一前端技术体系内。

Nuxt 官方文档说明 Nuxt 应用可以部署到 Node.js、静态托管以及 serverless / edge 环境。[Nuxt Deployment](https://nuxt.com/docs/4.x/getting-started/deployment)

### 2.2 推荐渲染策略

本项目不建议全站纯 SSR，也不建议全站静态化，推荐采用混合模式：

- 首页：SSR 或 ISR 风格缓存
- 文章详情页：SSR，配合边缘缓存
- 专题页：SSR
- 商品详情页：SSR
- 帮助文档、部分说明页：预渲染
- 管理后台：登录后 SPA/SSR 混合即可，不追求 SEO

### 2.3 是否部署到 Cloudflare

可以，且适合本项目的前台站点。

根据 Nuxt 官方部署文档，Nuxt 可部署到 serverless 或 edge 环境；`@nuxt/content` 官方模块页也明确提到支持 Cloudflare Workers 等 edge 环境。[Nuxt Deployment](https://nuxt.com/docs/4.x/getting-started/deployment) / [Nuxt Content 模块页](https://nuxt.com/modules/content)

#### 适合放在 Cloudflare 的部分

- 博客前台
- 内容详情页
- 专题页
- 商品展示页
- 静态资源分发
- CDN 缓存与边缘访问

#### 不建议强依赖 Cloudflare 的部分

- 后台复杂管理逻辑
- 大量实时写操作
- 重型 Java 后端

#### 成本判断

Cloudflare Pages Functions 的计费按 Workers 计费；静态资源请求免费且不限量，只有触发 SSR/Functions 的请求才计入 Workers 配额。[Cloudflare Pages Pricing](https://developers.cloudflare.com/pages/functions/pricing/) / [Cloudflare Static Assets Billing](https://developers.cloudflare.com/workers/static-assets/billing-and-limitations/)

Workers 官方定价页面显示：

- 免费计划：每天 `100,000` 次请求，共享 Workers 配额
- 付费标准计划：每月包含 `10 million` 请求，超出按量计费
- 静态资源请求免费

[Cloudflare Workers Pricing](https://developers.cloudflare.com/workers/platform/pricing/) / [Cloudflare Workers Limits](https://developers.cloudflare.com/workers/platform/limits/)

#### 架构建议

推荐部署方式：

- `Nuxt 4` 前台部署到 Cloudflare
- `Spring Boot 4` 部署到传统云主机或容器环境
- 数据库部署到独立数据库服务

这样可以兼顾：

- 前台访问性能
- SEO
- 成本控制
- Java 服务的稳定运行环境

## 三、UI 技术方案

### 3.1 推荐结论

推荐采用：

- 基础样式：`Tailwind CSS`
- 设计体系：自定义 `design tokens`
- 组件策略：优先自建业务组件，局部使用 `Nuxt UI`

不建议一开始完全依赖某个整套组件库来决定站点风格。

### 3.2 三种 UI 路线对比

#### 方案 A：原生 Tailwind CSS

优点：

- 自由度最高
- 最适合做品牌感强的内容站
- 不容易做成“默认模板站”

缺点：

- 组件沉淀成本更高
- 表单、弹窗、表格、反馈类组件要自己补更多细节

适合：

- 首页
- 文章页
- 专题页
- 商品页

#### 方案 B：Nuxt UI

优点：

- 和 Nuxt 生态贴合
- 后台管理类界面开发效率高
- 表单、表格、弹层、交互反馈能力比较完整

缺点：

- 容易让前台站点视觉风格偏“控制台化”
- 如果前台页面大量使用，会削弱品牌辨识度

适合：

- 管理后台
- 用户中心
- 作者工作台

#### 方案 C：其他 Tailwind 组件体系

例如 Headless UI、Radix 风格封装、自建组合组件。

适合用来补交互能力，但不建议在本项目里同时混用太多 UI 体系，否则后续维护成本会增加。

### 3.3 最终 UI 策略

本项目最终建议：

- 前台站点：`Tailwind CSS + 自定义组件`
- 中后台：`Nuxt UI + Tailwind CSS`

也就是：

- 前台强调品牌与内容表达
- 后台强调交互效率与可维护性

## 四、Markdown 如何渲染成 HTML

### 4.1 推荐技术

对于本项目，平台主内容不推荐使用 `@nuxt/content` 作为主内容引擎。

原因是本项目目标是“最美个人内容/博客平台”，后续还会扩展为服务端内容平台，因此主内容必须具备：

- 后台管理
- 草稿、审核、发布、下线
- 内容状态流转
- 分类、标签、专题、作者体系
- 商品与虚拟产品挂接
- 可被 API 化输出

这些能力更适合由 `Spring Boot 4 + 数据库` 统一管理。

因此推荐内容主链路为：

- 后台编辑器产出 `Markdown`
- `Spring Boot 4` 存储 Markdown 原文
- 内容发布时由服务端转换为 HTML
- `Nuxt 4` 在 SSR 阶段读取服务端返回的 HTML，并输出完整页面

这里可以使用 `markdown-it` 作为 Markdown 解析器。`markdown-it` 官方文档说明它既可以在 Node.js 中运行，也可以在浏览器中运行，本质上是一个可扩展的 Markdown 解析器。[markdown-it API](https://markdown-it.github.io/markdown-it/)

### 4.2 为什么不是手写 `markdown-it`

这里需要区分两件事：

- `markdown-it`：只是 Markdown 解析器
- `@nuxt/content`：是以文件内容为中心的内容管理与查询方案

如果项目只是简单博客，直接在 Nuxt 项目中用 `markdown-it` 或 `@nuxt/content` 都可以。

但本项目后续目标是服务端内容平台，因此主内容需要放在服务端，而不是主要存放在前端仓库中。  
在这种情况下：

- `@nuxt/content` 更适合静态文档、项目说明、少量固定帮助页
- `markdown-it` 更适合放在服务端内容处理链路中，负责把 Markdown 转为 HTML

因此本项目不是“手写一套 Markdown 系统”，而是：

- 使用数据库保存主内容
- 使用 `markdown-it` 处理 Markdown
- 使用 `Nuxt SSR` 输出最终 SEO 页面

### 4.3 到底由谁渲染 HTML

这个问题对 SEO 很关键。

推荐结论如下：

- **不推荐客户端渲染 Markdown 后再生成正文 HTML**
- **推荐由服务端先把内容准备好，再交给 Nuxt SSR 输出页面**

可选路径有两种：

#### 路径 A：Spring Boot 发布时渲染 Markdown

流程：

1. 后台保存 Markdown 原文
2. 内容发布或更新时，`Spring Boot` 使用 `markdown-it` 兼容方案或等价解析链将 Markdown 转为 HTML
3. 同时生成摘要、目录、阅读时长、纯文本等派生字段
4. Nuxt 页面请求内容详情接口
5. Nuxt 在 SSR 阶段拿到已转换好的 HTML，直接输出到页面

这是当前项目最推荐的方案，因为：

- SEO 最稳定
- 首屏 HTML 完整
- 前端职责更清晰
- 文章页性能更稳定
- 后续更容易做缓存

#### 路径 B：Nuxt SSR 时渲染 Markdown

流程：

1. `Spring Boot` 返回 Markdown 原文
2. Nuxt 在服务端渲染阶段使用 `markdown-it` 转成 HTML
3. 再把 HTML 输出到首屏响应

这个方案同样可以满足 SEO，因为根据 Nuxt 官方文档，Nuxt 默认支持服务端渲染，并会向浏览器返回完整 HTML，这有利于搜索引擎索引。[Nuxt Introduction](https://nuxt.com/docs/4.x) / [Nuxt Rendering Modes](https://dev.nuxt.com/docs/4.x/guide/concepts/rendering)

但这个方案不如路径 A 稳定，原因是：

- Markdown 渲染规则落在前端服务
- 每次请求时可能重复做渲染
- 后续做多终端输出时复用性较弱

#### 最终选择

本项目推荐：

- `Spring Boot 4` 负责主内容存储与发布时转换
- `Nuxt 4` 负责 SSR 页面拼装与 SEO 输出

也就是说：

- **内容 HTML 的生成责任，优先放在 Spring Boot**
- **页面 HTML 的最终输出责任，放在 Nuxt SSR**

### 4.4 为什么不推荐客户端渲染

`markdown-it` 官方文档也支持浏览器端运行，但对本项目主内容页不推荐这样做。[markdown-it API](https://markdown-it.github.io/markdown-it/)

原因：

- 客户端渲染会让首屏正文依赖 JavaScript 执行
- 对 SEO 不如 SSR 稳定
- 首屏体验和性能更差
- 对分享抓取、预览、弱网访问都不够友好

### 4.5 推荐内容渲染方案

- 内容主数据来源：服务端数据库
- 编辑格式：`Markdown`
- 转换位置：`Spring Boot` 发布链路
- 传输格式：HTML + 摘要 + 目录 + SEO 字段
- 页面输出：`Nuxt SSR`
- 富组件扩展：通过后端内容 AST 扩展或受控的自定义块组件实现

后续可以支持：

- 文章内插入商品卡片
- 文章内插入下载区块
- 文章内插入提示框、对比表、FAQ、CTA

### 4.6 `@nuxt/content` 的定位

`@nuxt/content` 并非完全不能使用，而是不应作为本项目“平台主内容”的核心方案。

它更适合放在这些场景：

- 项目帮助文档
- 开发者文档
- 固定说明页
- 少量与业务后台解耦的静态内容

## 五、Spring Boot 4 与 MyBatis-Plus

### 5.1 是否可以使用 MyBatis-Plus

可以。

MyBatis-Plus 官方 Quick Start 已提供 `Spring Boot 4` starter，文档中明确写到自 `3.5.13` 起支持 `Spring Boot 4`，可使用：

- `mybatis-plus-spring-boot4-starter`

[MyBatis-Plus Quick Start](https://baomidou.com/en/getting-started/)

### 5.2 是否推荐使用 MyBatis-Plus

推荐，但要有边界。

适合使用 MyBatis-Plus 的部分：

- 单表 CRUD
- 后台管理列表页
- 标签、分类、配置类数据
- 基础分页、逻辑删除、自动填充

不建议过度依赖其通用封装的部分：

- 复杂统计查询
- 多表强业务查询
- 内容推荐类 SQL
- 商品与订单的复杂聚合查询

建议策略：

- 常规数据访问：MyBatis-Plus
- 复杂查询：手写 Mapper SQL
- 极复杂报表：单独查询层或读模型

## 六、认证体系设计

### 6.1 后端认证基础

后端统一使用 `Spring Security`。

Spring 官方文档说明 Spring Security 是 Spring 体系中事实标准的认证与授权框架，支持用户名密码、OAuth2 Login、JWT 等多种认证方式。[Spring Security](https://spring.io/projects/spring-security) / [Authentication Reference](https://docs.spring.io/spring-security/reference/servlet/authentication/index.html)

### 6.2 后台认证方案

后台建议：

- `用户名/邮箱 + 密码`
- `Spring Security`
- `JWT Access Token + Refresh Token`
- Token 优先放在 `HttpOnly Cookie`
- 后台接口做基于角色与权限码的鉴权

后台角色建议至少包含：

- `SUPER_ADMIN`
- `ADMIN`
- `EDITOR`
- `AUTHOR`

后台鉴权建议：

- 路由级别：页面守卫
- 接口级别：Spring Security
- 方法级别：注解授权

### 6.3 前台认证方案

前台用户体系建议与后台共用同一用户中心，但登录策略分离。

前台建议支持：

- 手机号验证码登录
- 邮箱验证码登录
- 第三方登录作为后续扩展

前台认证技术建议：

- 后端仍然使用 `Spring Security`
- 前端 Nuxt 使用服务端会话感知
- 登录态通过 `HttpOnly Cookie` 维持
- Nuxt 中通过服务端请求当前用户接口恢复登录态

不建议前台把长效 Token 明文放在 `localStorage`。

### 6.4 为什么前后台要分开设计

原因不是技术不同，而是业务模型不同：

- 后台是“管理身份”
- 前台是“消费身份”

两者可以共用一套用户表和认证基础设施，但在以下方面必须区分：

- 登录入口
- 权限模型
- 路由保护
- 风控策略

## 七、推荐的前后端职责边界

### Nuxt 4 负责

- 页面渲染
- SEO 元信息生成
- 内容展示
- 商品展示
- 用户中心页面
- 管理后台页面
- BFF 级聚合逻辑

### Spring Boot 4 负责

- 用户认证与授权
- 内容管理
- 商品管理
- 订单与虚拟资产
- 评论、收藏、订阅
- 文件元数据
- 后台运营接口

## 八、当前阶段的明确决策

当前项目先固定如下技术决策：

1. 前台主框架使用 `Nuxt 4`
2. 前台采用 `SSR + 部分静态预渲染`
3. 前台可部署到 Cloudflare，后端与数据库不强绑定 Cloudflare
4. UI 采用 `Tailwind CSS + 自定义组件` 为主，`Nuxt UI` 用于后台和中台场景
5. Markdown 使用 `@nuxt/content`
6. 后端使用 `Spring Boot 4 + Spring Security + MyBatis-Plus`
7. 前后台共用认证基础设施，但区分登录策略与权限体系
