# 云屿第一版部署规划

## 文档目标

本文档用于确定 `云屿 / Yunyu` 第一版可上线方案，覆盖代码构建、镜像产出、服务编排、前端托管与发布顺序。

## 一、第一版部署目标

- 后端 `yunyu-server` 使用 `Java 25` 构建 Docker 镜像
- 通过 `GitHub Actions` 自动构建并推送后端镜像到 `GHCR`
- 使用 `docker-compose` 在服务器上运行 `Spring Boot + MySQL`
- 前端 `yunyu-web` 部署到 `Cloudflare Pages`
- 前端通过环境变量访问独立部署的后端 API
- 所有部署方案统一沉淀到 `docs/部署/` 目录

## 二、部署架构规划

### 1. 后端部署

- 部署位置：云服务器中的 Docker 环境
- 运行方式：`docker compose` 启动 `yunyu-server` 与 `mysql`
- 暴露端口：后端默认 `20000`，数据库默认 `3306`
- 镜像来源：`GitHub Actions` 构建并推送的 `GHCR` 镜像

### 2. 前端部署

- 部署位置：`Cloudflare Pages`
- 构建方式：连接 GitHub 仓库后自动构建 `Nuxt 4`
- 关键环境变量：`YUNYU_PUBLIC_API_BASE`
- 域名建议：
  - 前端站点：`www.xxx.com`
  - 后端接口：`api.xxx.com`

### 3. 数据层规划

- 第一版直接使用 `docker-compose` 内的 `MySQL 8`
- 数据目录通过 Docker Volume 持久化
- 后端容器启动时自动检查数据库与初始化表结构

## 三、发布链路规划

### 1. 后端链路

1. 开发者推送代码到 GitHub `main`
2. `GitHub Actions` 自动构建 `yunyu-server` 镜像
3. 镜像推送到 `GHCR`
4. 服务器执行 `docker compose pull`
5. 服务器执行 `docker compose up -d`

### 2. 前端链路

1. 开发者推送代码到 GitHub
2. `Cloudflare Pages` 自动拉取前端代码
3. 使用 `YUNYU_PUBLIC_API_BASE` 指向后端域名
4. Cloudflare 发布新的前端版本

## 四、环境拆分规划

### 1. 本地开发环境

- 后端默认 `dev` profile
- 数据库地址默认本机 `127.0.0.1:3306`
- 前端默认请求 `http://127.0.0.1:20000`

### 2. 生产环境

- 后端显式使用 `prod` profile
- 数据源地址通过环境变量注入
- JWT 密钥通过环境变量注入
- 前端通过 Cloudflare Pages 环境变量配置 API 地址

## 五、第一版执行顺序

### 阶段 1：后端镜像化

- 新增后端 `Dockerfile`
- 新增生产环境配置
- 保证容器启动时可通过环境变量连接数据库

### 阶段 2：CI 构建镜像

- 新增 GitHub Actions 工作流
- 自动构建 `linux/amd64` 与 `linux/arm64` 镜像
- 推送到 `GHCR`

### 阶段 3：服务器编排

- 新增根目录 `docker-compose.yml`
- 配置 `mysql` 与 `yunyu-server` 服务
- 配置持久化卷、端口和环境变量

### 阶段 4：前端上线

- 将 `yunyu-web` 接入 `Cloudflare Pages`
- 设置 `YUNYU_PUBLIC_API_BASE`
- 绑定正式域名并验证接口联通

### 阶段 5：运维收口

- 补齐部署文档
- 形成发布步骤
- 后续再补充 HTTPS、反向代理、备份与监控

## 六、当前阶段的取舍说明

- 第一版优先强调“先上线可用”，因此数据库先内置在 `docker-compose`
- 第一版后端先由 `1Panel` 负责反向代理到后端端口，后续再补独立 `Nginx` 或 `Caddy` 一键部署方案
- 第一版镜像仓库优先选择 `GHCR`，因为与 GitHub Actions 集成成本最低
- 第一版前端与后端解耦部署，便于后续分别扩容

## 七、第二阶段建议

- 为后端补齐独立 `Nginx` 反向代理方案与 HTTPS
- 将 `MySQL` 迁移到独立托管实例
- 增加对象存储与上传能力
- 增加日志采集、健康检查与告警
- 增加灰度发布与回滚流程
